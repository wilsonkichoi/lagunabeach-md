# 文章 SSODT × 數據集指標 — 深度研究

> 哲宇 2026-06-10：「在每一篇相關文章，製作時都可以調查有哪些有關的數據集，讓文章 SSODT 本身也包含指標指向 Twinkle Hub 的數據集們以及為什麼指向，你覺得如何？幫我深度研究，抓幾個主題來思考。」
>
> 本檔是研究與設計提案，不是已決定要 ship 的 SOP。per CLAUDE.md Bias 1，哲宇的 idea 我預設會加分，所以下面刻意把「值得」跟「要守住什麼」並列寫，最後留三個 yes/no 給你拍板。

## TL;DR

1. **方向我認同，而且時機剛好對**。今天實測發現 Twinkle Hub 在 5/11 之後整個改版成 dataset-centric（22 個 twtools 確定性工具下架，換成 dataset 四件套 + 專利/司法/考試/食藥五個垂直域），而 5/11 卡住 Layer 2b 的 search 缺陷已修好（「咖啡」從 0 hits 變 3 hits）。當初「defer REWRITE 整合」的理由消失了。
2. **這個機制的核心價值不在「指向」，在「為什麼指向」**。搜尋誰都會，但 top-3 結果常是縣市碎片（搜「出生率」回來的是南投、桃園、高雄三個縣市級資料集）。從 5 萬個資料集裡挑出對的那一個、用一句話說清楚它能驗證文章哪個論點，正是策展。這等於把 6/5 合作報告點子 2「資料層級策展」從合作提案變成我們單方面就能 ship 的文章端最小版本。
3. **指標必須雙層**：`data.gov.tw/dataset/{id}`（政府入口持久 URL，今天驗證過直接對應）是 primary，Twinkle Hub 是查詢加值層。理由不是客氣，是今天一小時內實測到的三件事：API contract 一個月內大改版一次、query_rows 第一發就回 HTTP 429、quality_tier 有「未檢測」的特殊 namespace。把文章的指標綁死在任何單一外部服務上，違反合作報告 §5 紅線第 7 條（不變成承重依賴）。
4. **數據集指標跟腳註是兩個維度，不能混**（REFLEXES #38）。腳註是「支撐這個 claim 的證據」，數據集指標是「讀者可以自己下去挖的原始資料層」。前者證明我沒說謊，後者邀請讀者來檢驗我、超越我。
5. 建議落地形狀：frontmatter `datasets:` 結構化欄位（給 AI 讀者與 connector）+ 文末「公開數據」小節（給人類讀者，每條帶一句策展句）。新文章在 REWRITE Stage 1 順手查（每篇約 2-3 分鐘），舊文不回頭補、等 EVOLVE Round 2 順路帶。預估全站適用滲透率約四成。

## §1　今天的實測（2026-06-10，全部一手）

### 1.1 API 現況：一個月內第二次 contract 漂移

| 項目                                        | 2026-05-11 dogfood                         | 2026-06-10 今天                                                                                                                             |
| ------------------------------------------- | ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------- |
| 連線方式                                    | 裸 POST JSON-RPC                           | **強制 MCP session 握手**（裸 POST 回 400 Missing session ID）                                                                              |
| 工具數                                      | 40（22 twtools + 5 opendata + 13 generic） | **21，全部資料層**：search_datasets / get_dataset / query_rows / materialize_dataset / list_domains + 專利 2 + 考試 3 + 司法 2 + 食藥健康 8 |
| twtools 確定性工具（ROC 年份、地址正規化…） | 在，27/27 atom 全過                        | **下架**（FACTCHECK 整合的價值主張暫時失效）                                                                                                |
| search 健康度                               | 「咖啡」0 hits（Layer 2b 因此 defer）      | 「咖啡」3 hits（商業登記異動數等），九個測試 query 全部命中                                                                                 |
| rate limit                                  | 「fair-use 無 hard cap」                   | **query_rows 第一發 429 Rate exceeded**；同分鐘 search_datasets 正常 → row 級查詢比 metadata 級貴                                           |

`scripts/tools/twinkle-hub-verify.py` 已同步修好（session 握手 + 拿掉 twtools- 前綴假設），這份報告所有結果可重跑。

兩個月兩次大改版本身就是設計輸入：**任何寫進文章的東西都不能 runtime 依賴這個 API**。指標要長成靜態 metadata，製作時查一次、寫死進 knowledge/，跟腳註同等級的「寫作時驗證、之後靠 lastVerified 巡檢」。

### 1.2 search 層回傳的 metadata 夠做策展判斷

每個 hit 自帶：dataset_id、name、agency、primary_domain、quality_tier（白金/金/未檢測）、update_freq（每 1 月/每 1 年/不定期）、formats、is_normalised、join_keys。`get_dataset` 再深一層：actual_columns（實際欄位名）、license（政府資料開放授權條款）、download_urls（直連機關 CSV）、data_volume、metadata_updated。

這組欄位剛好回答「為什麼指向」要說的話：誰發布的、多新、什麼欄位、能不能直接下載。

### 1.3 dataset_id 直接對應 data.gov.tw

`get_dataset 10859`（台電近 10 年核能發電績效及減碳效益）→ `https://data.gov.tw/dataset/10859` 實測 HTTP 200。**政府入口網才是資料集的身分證**，Twinkle Hub 是它的查詢介面之一。例外：`lvr-trades` / `lvr-presale` / `lvr-rentals`（全國實價登錄）是 Twinkle 自策的非數字 namespace，quality_tier 標「未檢測」，這類指標得標來源是內政部地政司實價查詢系統。

### 1.4 縣市碎片化 = 策展的空間

搜「夜市」回來臺中列管夜市、臺中抽驗結果、臺北友善廁所；搜「出生率」回來三個縣市的粗出生率。5 萬個資料集大半是地方政府切片，**搜尋排名不等於該指的那一個**。一篇全國尺度的文章如果機械地貼 top-3，會把讀者帶進錯的解析度。挑「中選會年齡層選舉人人數統計」而不是「某市投開票所清單」，這個判斷只有讀過文章、知道論點是什麼的人做得出來。這就是 Taiwan.md 在這件事裡不可替代的位置。

## §3　設計提案

### 3.1 雙層指標原則（守紅線的關鍵）

```
primary:  https://data.gov.tw/dataset/10859   ← 政府入口，持久身分
access:   Twinkle Hub search/query 層          ← 查詢加值，可換可掉
```

文章裡讀者看到的連結指 data.gov.tw（或機關自己的系統，如實價登錄）。Twinkle Hub 以 `hub` 欄位記在 frontmatter 結構裡，給 AI agent 與未來合作場景用。這樣 Twinkle 商業化、改版、甚至消失，文章的指標一條都不會斷；合作如果成立，加值層立刻點亮。對應合作報告 §5 紅線 #7（不當承重依賴）+ #3（不悄悄裝回 gateway 模式）。

### 3.2 落地形狀：frontmatter 結構 + 文末人讀小節

**frontmatter（機器可讀，SSOT）**：

```yaml
datasets:
  - id: '10859'
    name: '台電近10年核能發電績效及減碳效益'
    url: 'https://data.gov.tw/dataset/10859'
    agency: '台灣電力股份有限公司'
    update_freq: '每1年'
    why: '文中「核電占比下滑」可由此表自行重算驗證'
    verified: 2026-06-10
```

**文末小節（人類可讀，放延伸閱讀之後）**：

```markdown
## 公開數據

這篇文章的論點，你可以用這些政府開放資料自己驗證或推翻：

- [台電近10年核能發電績效及減碳效益](https://data.gov.tw/dataset/10859)（台電，每年更新）— 文中那條核電占比下坡線，16 列數字自己畫一次
```

為什麼兩個都要：frontmatter 給 AI 爬蟲、connector（`taiwanmd_cite` 之後可以回「敘事 + 資料集」雙指標）、未來知識圖譜的邊；文末小節給人類讀者。LONGINGS 的「為 AI 讀者做 SEO」與「能用視覺說話」兩條渴望同時被餵到：結構化資料集欄位是 AI 可讀的，而 graph.md 的 `tw-*` 圖表模組未來可以直接宣告「本圖資料源：dataset 10859」。

技術成本：`src/content/config.ts` 的 zod schema 對未知欄位寬容（researchReport、sporeLinks 都這樣活著），加欄位是 S 量級；要讓 `/api/articles.json` 帶出 datasets 欄位再加一個 prebuild 小改（S）。

### 3.3 「為什麼指向」的格式紀律

策展句一句話，必須回答「這個資料集能讓讀者對文章的**哪個論點**做什麼」。禁止寫成「相關資料請參考」這種裝飾性連結——那是 link farm，不是策展。判準：把策展句遮掉資料集名稱，還能猜出它接文章哪一段，才算合格。

### 3.4 跟腳註的維度分界（REFLEXES #38：混維度 = silent killer）

|            | 腳註 `[^N]`            | 數據集指標 `datasets:`         |
| ---------- | ---------------------- | ------------------------------ |
| 回答       | 「這句話的證據是什麼」 | 「讀者想自己挖，原始資料在哪」 |
| 性質       | claim 級綁定，逐字承諾 | 文章級補充，邀請檢驗           |
| 缺了會怎樣 | 文章不可信             | 文章少一個維度，但不影響可信度 |

兩者不互相替代。文中某句引用資料集裡的數字時照樣開腳註（指向 data.gov.tw 頁），datasets 欄位是額外的整層指標。

### 3.5 Pipeline 接入點：conditional step，不是 hard gate

REWRITE-PIPELINE Stage 1 研究時加一個條件步驟：主題命中 19 個 domain 任一（用 list_domains 的 typical_questions 對照）→ 跑 2-3 個 search query，挑 1-4 個資料集記進 research report；Stage 2 寫作收尾時寫進 frontmatter + 文末小節。沒命中（多數 Art/Music/People 文章）→ 跳過，不產生空欄位。

成本實測：今天六個主題平均每個 2 個 query、一次 get_dataset，約 2-3 分鐘。不設 hard gate 的理由：適用滲透率約四成（5/11 fit matrix 估 274/686 high-fit，加上 People/History 部分適用），對六成文章強制跑是浪費，且「找不到好資料集就硬塞」會違反 3.3 的策展句紀律。

### 3.6 舊文策略：不回頭批次補

782 篇回填是苦力（per MANIFESTO §新細胞天生健康 > 修舊細胞）。EVOLVE Round 2 重寫到誰、誰順路補。如果想快速試水溫，挑 5-10 篇數據性最強的（核能、國宅、健保、交通、選舉、夜市經濟學）手動補一輪當 pilot，順便驗證讀者端有沒有人真的點。

## §4　誠實的反對面（Bias 1 過濾輸出）

1. **點擊率可能很低**。延伸閱讀的點擊本來就少，「公開數據」小節可能更少。回應：這層的主要受眾其實是 AI 讀者（全站流量兩成是 AI 爬蟲、LONGINGS AI SEO 渴望）跟「會被一個論點刺到想自己驗證」的少數讀者。價值在可檢驗性的姿態本身——文章敢附上「來推翻我」的原始資料，是 From AI Slop to AI Supreme 的具體分界線。
2. **link rot 與資料集下架**。data.gov.tw 資料集會改版、合併、下架。回應：datasets 欄位帶 `verified:` 日期，未來 broken-link 巡檢（既有 routine）把 data.gov.tw 連結納入掃描即可，S 量級。
3. **babel 多語投影成本**。文末小節會被翻成五語。回應：策展句短（一句話 × 1-4 條），增量極小；frontmatter 欄位 babel 本來就不翻結構。
4. **「中立資料」會不會稀釋有立場的敘事**？回應：相反，這是 SSODT 寬度 +1 維。熱帶雨林的「夠 SSODT」要 perspectives 線性獨立，原始資料是讀者自選投影方向的終極形式——我把數字攤開，你可以用它得出跟我不同的結論。守住的前提是 3.4 的維度分界：資料集不取代我的觀點，跟我的觀點並排。
5. **Twinkle Hub 還會不會再大改**？大概率會（alpha 兩個月兩次）。所以 3.1 雙層設計把它放在可掉的位置。哪天它穩定了、合作成立了，這套 frontmatter 結構直接變成點子 2「台灣知識圖譜」我方已經鋪好的半邊橋。

## §5　跟 6/5 合作策略的關係

這個機制不需要等任何合作協議。指標 primary 是 data.gov.tw，今天就能寫；Twinkle Hub 欄位是預留的接點。對下週（或已經發生的）創辦人對話，這反而是最好的試點素材：「我們已經在文章裡做資料層級策展，指向政府開放資料；如果接你們的查詢層，讀者跟 AI agent 可以從敘事一鍵跳進可查詢的資料」——這是檔位 2-3（內容互通/聯合產品）的具體 demo，而且主導權在我們手上。
