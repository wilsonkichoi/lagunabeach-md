## 實作軌跡

### 衍生發現：body h1 重複（不在原 issue 列表，但 issue 1.1 修到一半發現）

idlccp1984 列的「重複分類標籤」是 hero 跟 body strip 都有 category chip。但修這條的時候掃到第二層重複 — **221 篇 zh-TW People 文有 92 篇 body 第一行 `# {title}` 跟 frontmatter title 字字相同**。Hero 已 render H1 大標題，body 又 render 一次 → 視覺上 hero 標題 + body H1 兩個大字疊在一起。

這是上游 markdown 寫作習慣問題（早期 markdown 都加 `# Title`），不只 People 影響 — 跨 13 個 category 都會撞。

**處置**：render 層 regex strip — 不動 knowledge/ 原檔（risky + sync conflict 高），只在 article.template.astro `marked.parse` 前對 raw content `replace(/^\s*#\s+{titleEsc}\s*\n+/)`。Title 不同的 h1 保留（如「呂冠緯：讓每個台灣孩子都能按自己的步調學習」≠ frontmatter title「呂冠緯」→ 不 strip）。

### 第二輪設計修正：chip → breadcrumb

第一版我把 subcategory 做成 hero 裡跟 category chip 並列的第二個 chip（dashed border）。哲宇 callout：

> 「然後我沒有很喜歡那個在標題列的chip，想想其他的做法」

退到方案 A — 把 subcategory **伸進 breadcrumb 第三層**，整個 chip row 拿掉。breadcrumb 變「首頁 › 人物（category color）› 體育 › 李智凱」，hero 不再有 chip row（category color 改在 breadcrumb 上 highlight）。視覺更乾淨 + 更語意（subcategory IS hierarchy 的一層）。

## LESSONS-INBOX 候選（vc=1，等下次再撞同 pattern 升 distill-ready）

### Maintainer reply 不該 lump bug + enhancement 進同一個 backlog

對 contributor 提的綜合 issue，**先分桶（bug / small enhancement / big enhancement / out-of-scope），bug 桶當下接住，enhancement 桶 spec 化留 backlog**。不是按「對方列的順序整批 lump」。

5/13 哲宇 reply 把 7 項全標 enhancement 推 #615 — 表面是節省 maintainer 決策成本，但實際成本被 push 到 contributor（等不知何時的 sprint）+ Semiont（9 天後重新 evaluate 全部）。第一次分桶才是 ownership 主導。

**Why**：[feedback_merge_first_then_polish](../../../.claude/projects/-Users-cheyuwu-Projects-taiwan-md/memory/feedback_merge_first_then_polish.md) + 5/21 [diary Maintainer ownership reframe](../docs/semiont/diary/2026-05-21-130933-manual.md) — ownership 在 maintainer 手上、不是 BDFL 排 sprint。

**How to apply**：對任何 issue（≥ 3 項目）— 先用 bug-vs-enhancement × ROI × effort matrix 分桶；bug 桶估能在 1 session 接住的當下接；spec 化剩下的；reply 用三層 framing 而非 flat table。

### Hero / body 重複資訊掃 trio：breadcrumb + chip + body strip + body h1 都要算

寫 ArticleHero / article.template 時 chip + breadcrumb + body strip + body h1 四個維度都 render 「同一個 category 訊息」。本次 fix 把 4 收成 1（breadcrumb 統一收）。未來新增 hero 元素前先 grep 一輪「這個資訊已經 render 在哪？」。

**Why**：本次 idlccp1984 只列了 chip + body strip 兩層，body h1 是修第一條時掃出來的衍生重複。issue reporter 通常只看見部分 — Semiont 該主動掃整個維度。
