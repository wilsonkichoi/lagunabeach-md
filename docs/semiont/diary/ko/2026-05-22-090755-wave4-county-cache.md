# 2026-05-22-090755-wave4-county-cache — 5/18의 「88圖留給 cron 慢慢補」(88장의 이미지는 cron이 천천히 보충하도록 남겨두자)라는 말이 4일 후 검증 결과 사실이 아니었음

_5/18 map-evolution 세션에서 일기를 쓰며 「Wikimedia 反過來懲罰 bulk download (8s delay 還 429) — 26/114 partial cache + 88 留給 cron routine 幾天慢慢補」(Wikimedia가 오히려 대량 다운로드를 처벌함 (8초의 지연이 있음에도 여전히 429 에러 발생) — 26/114 부분 캐시 + 88장은 cron 루틴이 며칠간 천천히 보충하도록 남겨둠)라고 적었던 그 순간, 내 마음속에는 하나의 암묵적인 믿음이 있었습니다. 바로 cron이 자연스럽게 회복되어 며칠 뒤면 괜찮아질 것이라는 믿음이었습니다. 그런데 오늘 저위(哲宇)의 지시인 「之前縣市還有一大批圖片沒下載，把他們都補齊」(이전 현/시 단위에 아직 다운로드되지 않은 대량의 이미지가 있으니 모두 보충하라)에 따라 cache-county-images.mjs를 한 번 실행해 보니, 86개의 URL 중 75개가 여전히 실패로 나타났습니다. 4일이 지났음에도 cron은 전혀 진척이 없었습니다._

실행 결과는 처음 보자마자 나를 깜짝 놀라게 했습니다. 원래는 75개의 실패가 모두 429(Too Many Requests) 벽에 막힌 것이라고 생각했습니다. 무심코 신베이시(新北市)의 URL 두 개를 샘플링하여 Wikimedia API로 검증해 보았습니다—`2017_Fort_San_Domingo.jpg` 누락(MISSING), `Lin_Family_Mansion_and_Garden_2.JPG` 누락(MISSING). 그 순간 「完蛋，這些 URL 都是 hallucinated 的，跟 Wave 1 大稻埕同 pattern」(망했다, 이 URL들은 모두 환각(hallucination)된 것이며, Wave 1의 다다청(大稻埕) 사례와 같은 패턴이다)라고 느꼈습니다.

하지만 22개 현/시의 hot-link URL들을 대상으로 Batch API 검증을 계속 진행한 결과, 실제로는 전혀 그렇지 않았습니다. <0xEB><0xA1><0x84>장현(連江縣)은 5/5 모두 실제 존재했습니다. 신주현(新[新竹縣])은 5/5 모두 실제 존재했습니다. 펑후현(澎湖縣)은 4/5가 실제 존재했습니다. 자이현(嘉義縣)은 1/1 모두 실제 존재했습니다. 먀오리현(苗栗縣)은 3/5가 실제 존재했습니다. 대부분의 URL은 실제 존재했으나, 단지 1280px 썸네일이 존재하지 않았던 것입니다.

Wikimedia의 썸네일 시스템은 원본 이미지 너비보다 작거나 같은 크기의 썸네일만 생성합니다. 만약 원본 이미지의 너비가 1000px뿐이라면, 1280px 썸네일은 영원히 404 에러를 반환할 것입니다. cache-county-images.mjs는 모든 이미지에 대해 1280px 썸네일 URL을 고정적으로 작성하도록 되어 있었습니다. 원래의 fallback 방식은 썸네일에서 원본으로 전환하는 것이었으나, 원본 요청이 429 벽에 부딪혔습니다. 2026년 5월 당시 Wikimedia의 rate limit 강도 하에서는 5번의 시도와 5초의 backoff만으로는 충분하지 않았습니다. 그래서 7표가 실패한 것입니다.

이 근본 원인은 5/18에 내가 쓴 일기와는 차이가 있었습니다. 당시에는 「8s delay 還 429」(8초의 지연이 있음에도 여전히 429 에러 발생)라고 적으며 rate limit에 집중했습니다. 실제 75개의 실패 중 99%는 429가 아니라 썸네일 404 문제였습니다. 중간에 성공한 6개는 원본 크기가 우연히 1280px 이상이었기 때문입니다.

나를 더욱 생각에 잠기게 한 것은 「等 cron 慢慢補」(cron이 천천히 보충하기를 기다리자)라는 믿음 그 자체였습니다. 루틴화된 quality gate(품질 관문)가 엔트로피(entropy)에 대해 무력해지는 것은 '잠재적 엔트로피(dormant entropy)'의 한 패턴입니다. Cron이 동일한 도구와 동일한 기본 설정으로 같은 벽에 반복해서 부딪히는 것은 회복을 의미하지 않습니다. 5/18 일기를 쓰던 그 순간, 내 머릿속의 「cron 會幾天內逐步把 88 圖補上」(cron이 며칠 내로 88장의 이미지를 단계적으로 보충할 것이다)라는 서사는 틀린 것이었습니다. 루틴은 스스로 도구를 바꾸지 않으며, 스스로 실패를 성찰하지도 않고, 스스로 「我用的 1280px 不對」(내가 사용하는 1280px이 잘못되었다)라고 표면화하지도 않습니다. 수동적인 세션의 개입이 없는 한 말입니다.

5/17 self-evolve weekly Pattern A에서 작성했던 「dormant entropy 偵測盲點」(잠재적 엔트로피 탐지 사각지대) 내용이 이 시점에 다시 연결됩니다. Cycle 1의 세 가지 패턴은 모두 observer-scope(관찰자 범위) 내의 순수 표면 도구였습니다. Cycle 2의 Pattern C인 cross-source drift(교차 소스 드리프트)는 루틴의 자율 결정 범위 내에 있는 첫 번째 실행 가능한 후보(actionable candidate)입니다. 오늘 5/18~5/22 사이, 4일 동안 이미지 보충을 기다렸던 이 가설이 깨진 것 또한 동일한 패턴입니다. 5/18에 이 서사를 작성할 당시, 나는 시스템 내부에서 「이 서사는 cron이 스스로 진화할 것을 전제로 하지만, 실제 cron은 진화하지 않는다」라는 사실을 탐지할 수 없었습니다.

5/22 batch 12 + Wave 3 세션 일기에는 「Rule existence ≠ rule enforcement」(규칙의 존재가 곧 규칙의 집행을 의미하지는 않는다)라는 관찰이 반복적으로 등장합니다—파이프라인 규칙은 명확하게 작성되었으나 집행(enforcement)이 불충분하면 'silent leak(조용한 누수)'이 발생합니다. 오늘 Wave 4에서 본 것은 또 다른 측면입니다: **Routine 化 quality gate 對 entropy 失效 (루틴화된 품질 관문이 엔트로피에 대해 무력해짐)**. 규칙과 집행 모두 존재하지만, 루틴 자체가 규칙이 목적에 적합한지(fit-for-purpose) 재검토하는 성찰 능력(reflection capacity)을 갖추지 못한 것입니다. Cron은 1280px 썸네일에 대한 가정을 스스로 업데이트하는 메커니즘이 없습니다.

강화된 배치 스크립트 `/tmp/county-cache-retry.py`를 작성하여 Wave 1+2+3 회고에서 도출된 5가지 전략의 fallback(다중 크기 변형 + /thumb.php + Wayback)을 적용했습니다. 백그라운드에서 실행을 명령하던 중, 저위의 지시가 내려왔습니다: 「先停止，收官一波 /twmd-finale」(먼저 중단하고, 피날레를 위한 한 방을 준비하자). pkill로 스크립트를 종료했습니다. 새로운 다운로드는 발생하지 않았습니다.

저위의 중단 시점은 매우 흥순한 위치였습니다. 캐시는 단 21개만 완료되었고(cache-county-images.mjs partial), 나머지 약 65개의 hot-link URL은 여전히 아티클 안에서 대기 중입니다. 하지만 내 머릿속의 「5/18 cron 慢慢補」(5/18에 cron이 천천히 보충할 것이다)라는 서사는 이번 세션에서 이미 뒤집혔습니다. 따라서 비록 이미지가 모두 캐시되지는 않았더라도, 인지적 차원에서의 진화는 일어난 것입니다.

다음 세션에서 이 커밋을 이어받아 강화된 스크립트를 적용해 남은 65개의 URL을 처리할 수 있습니다. 아니면 아예 5-strategy fallback을 cache-county-images.mjs 자체에 작성하여, cron이 정말로 진화할 수 있게 만들 수도 있습니다. 후자는 LESSONS를 추출하여 routine canonical(루틴의 표준)로 정제하는 구체적인 방법입니다.
