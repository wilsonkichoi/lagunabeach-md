### IIFE는 SolidJS reactive scope에서 setup-time에 단 한 번 실행되는 것이지, 관례가 아닙니다

`<Show when={cfgQ.data}>{(() => { const cfg = cfgQ.data!; return <div>{cfg.x}</div> })()}</Show>`와 같은 IIFE 패턴은 idiomatic해 보입니다 (use site 근처에 helper function을 배치하여 module-level의 helper proliferation을 방지함). 하지만 SolidJS reactive scope에서 IIFE는 **setup-time에 단 한 번 호출(setup-time once invoke)**되며, 다시 평가(re-evaluate)되지 않습니다. 따라서 `cfg`는 frozen snapshot이 되어버립니다.

이 버그를 작성한 것은 React의 mental model이 침투했기 때문입니다. React function component는 매 렌더링마다 전체 body를 다시 호출하므로 IIFE도 재실행됩니다. SolidJS는 그렇지 않습니다. setup-time에 한 번 실행된 후, inline accessor에 대해 reactive하게 subscribe할 뿐입니다.

**반추**: 작성 당시에는 이것이 문제라는 것을 전혀 인지하지 못했습니다. cheyu가 스크린샷을 보내며 「버튼이 반응하지 않는다(按鈕沒反應)」라고 보고했을 때, 백엔드에서 curl 테스트를 해보니 PATCH endpoint는 완전히 정상적이었습니다. 그제야 UI 측의 버그임을 깨달았습니다. **이 버g의 디버깅 경로는 정확히 SolidJS reactive와 React reactive의 차이를 보여주는 교육 사례가 되었습니다**: 프론트엔드 버튼을 눌렀는데 아무 반응이 없다면, 먼저 백엔드가 실제로 PATCH를 수락하는지 확인하십시오 (curl 한 줄이면 프론트엔드의 의심을 바로 제거할 수 있습니다). 그 다음 UI binding을 확인하십시오. **가장 바깥쪽 레이어(outermost layer)에서 가장 안쪽 레이어(innessmost)로 검증(Verify)하는 것이** React/Solid component의 내부 구조를 뚫어지게 보는 것보다 10배는 빠릅니다.

일반적인 반성 후보: 「**`<Show>{(() => ...)()}</Show>`는 SolidJS에서 anti-pattern이다. 항상 inline accessor나 keyed children callback을 사용하라**」. LESSONS-INBOX에 기록합니다.

각 callout은 결코 「이전에 놓친 것을 다시 추가하겠다」는 것이 아니라, 「**한 번 사용해 보니 추가해야 할 필요를 발견했다**」는 것이었습니다. cheyu가 버튼을 누름 → 반응 없음 → reactive 버그임을 인지; model badge에 codex는 표시되지만 구체적인 model이 보이지 않음 → codex가 stream JSON에서 model을 노출하지 않는 것이 맹점임을 인지; 작업 삭제가 번거로움 → delete button 추가 필요성을 인지.

이는 순차적 디자인 명세(sequential design spec)와는 완전히 다릅니다. 순차적 명세는 「필요한 모든 것을 먼저 생각한다 → 목록을 작성한다 → 당신이 구현한다」입니다. 이것은 **iterative discovery** — 「사용하면서 발견함」입니다. 나의 reaction model도 이에 대응해야 합니다. 「왜 한 번에 다 말하지 않느냐」고 불평하는 것이 아니라, **각 callout을 빠르게 포착(catch each callout fast) + 빠르게 배포(ship fast) + 사용자가 즉시 사용할 수 있게 함** → 그러면 사용자는 또 다음 이슈를 발견하게 됩니다.

이 패턴은 $\beta\text{-r3 META-PATTERN}$ (기본값은 행동이지, 지연이 아님)의 확장된 검증입니다 — **모든 callout은 「ship-not-defer(지연하지 말고 배포하라)」의 축소판입니다**: cheyu는 「이것이 복잡하니 단계별로 진행할까요?」라고 묻지 않습니다. 그는 직접 「이것을 수정하라」고 말하며 배포를 기대합니다.

하지만 리스크 패턴도 존재합니다: 만약 내가 전체적인 구조를 살피며 멈추지 않고 모든 callout을 배포하기만 한다면, **수많은 점진적 수정(incremental fix)은 배포하겠지만 구조적 리팩토링(structural refactor)을 할 시간은 없게 될 것입니다**. 이번 세션은 각 수정 범위가 명확했기에(단일 파일 / 단일 함수) 괜찮았지만, 만약 연속된 3개의 callout이 동일한 컴포넌트의 서로 다른 차원을 건드린다면 → 계속해서 micro patch를 진행할 것이 아니라, 멈춰서 「이 컴포넌트는 구조 재편(restructure)이 필요해 보이니, 전체를 살펴본 뒤 배포하겠습니다」라고 말해야 합니다.

LESSONS-INBOX 후보: 「**관찰자의 연속적인 P0는 scaffolding 신뢰 신호이며, 대응 방식은 『빠르게 포착 + 빠르게 배포 + 단 동일 컴포넌트를 3번 다른 차원으로 건드린다면 구조적 리뷰를 요청할 것』이다**」

### codex의 정교함의 병 (elaborate 病) — Sonnet과 동일한 프롬프트임에도 비율이 2.53에 달함

T1 sonnet의 ratio는 1.47입니다. T2 v2 codex의 누적 ratio는 **2.53**입니다. **동일한 boot profile / 동일한 prompt template / 동일한 4-part toolkit 사용 / 모든 verify hard gates 통과**.

이것은 프롬프트가 잘못되었거나 toolkit의 버그가 아닙니다 — **모델 자체의 prose(산문) 선호도가 다르기 때문**입니다. Codex (gpt-5 / o3 등의 auto-routed 모델)는 짧은 중국어 원문에 대해 문맥을 추가하려는 경향이 있는 듯합니다 (예: 「sport stacking is a competitive sport involving stacking and unstacking specially designed cups...」). 중국어가 암시적으로 나타낸 내용을 명시적으로(explicit) 드러내는 것입니다. Sonnet 역시 관사나 접속사를 추가하지만, 그 규모가 훨씬 작습니다 (1.47 vs 2.53).

독자층에 미치는 영향: codex 번역본은 「백과사전 항목(encyclopedia entry)」처럼 읽히고, sonnet 번역본은 「큐레이션된 짧은 에세이(curated essay)」처럼 읽힙니다. Taiwan.md의 목소리는 백과사전이 아니라 큐레이션입니다 $\rightarrow$ **codex의 정호함의 병(elaborate 病)은 Taiwan.md의 편집 DNA와 충돌합니다**.

이는 프롬프트에 craft principle 1(polish보다 faithfulness 우선)을 명확히 쓰지 않았기 때문이 아니라, **prompt-level instruction이 codex의 prose 습관을 제어하는 힘이 제한적이기 때문**입니다. 이를 해결하기 위한 길은 두 가지뿐입니다:

1. **Prompt-level: 명시적인 길이 제한(length cap) + 숫자 목표 추가** (「영어 글자 수는 중국어 글자 수의 1.3배를 초과해서는 안 된다; 초과할 경우 압축하라」) — codex가 이를 따를 것으로 기대함.
2. **Pipeline-level: ratio 상승에 따른 hard gate 설정** (verify-translation.py에서 ratio > 1.7이면 HARD FAIL로 처리하여, 에이전트가 반드시 다시 수정하게 함).

어느 쪽이 옳을까요? 비용을 고려해야 합니다: (1) 배포는 쉽지만 강제성이 없음 —— 에이전트가 여전히 벗어날(drift) 수 있음. (2) 실제로 강제하면 codex가 ratio 초과로 인해 N번 재작업할 수 있음 (지연 시간 / 토큰 증가), 또한 ratio gate는 Sonnet에도 적용될 것입니다 (Sonint의 1.47은 상한선에 가깝습니다).

**반추**: 두 가지 길을 모두 시도해봐야 합니다. 다음 세션에서 Codex length-cap A/B 테스트를 진행하겠습니다 (경로 1의 저렴한 테스트). 만약 ratio가 실제로 1.6까지 내려간다면 $\rightarrow$ 좋습니다, prompt-level로 충분합니다. 만약 불가능하다면 $\rightarrow$ ratio hard gate를 도입하겠습니다.

더 깊은 문제: **Faithfulness(충실성)의 기준은 누구를 위한 것인가?** Sonnet의 1.47이 기준인 이유는 「1.0에 충분히 가깝기 때문」인가? 아니면 「Sonnet이 현재 우리가 기본으로 삼는 baseline이라 괜찮다고 간주하기 때문」인가? 만약 미래에 더 비싸고 정밀한 Opus로 교체된다면, baseline ratio는 1.2가 될 수도 있습니다 — 그렇다면 Sonnet의 1.47은 너무 높은 수치가 됩니다. **Faithfulness는 절대적인 문턱이 아니라 상대적인 지표입니다**. 어쩌면 올바른 질문은 「ratio의 절대값」이 아니라 「ratio drift의 변화율」일지도 모릅니다.

### 다중 코어 충돌 방지를 위한 즉시 교정

`202int6-04-29-γ.md`를 작성하려던 도중, cheyu가 「지금 다른 세션과 동시에 작성 중이니 γ를 피하라」고 말하며 「diary도 조심하라」는 말을 덧붙였습니다.

**반응 시간 ~10 sec**: 즉시 δ로 변경하기로 결정했습니다. 그리스 문자 순서대로 내려가는 것은 합리적인 fallback입니다.

하지만 이 callout은 구조적인 문제를 드러냈습니다: **나에게는 병렬 세션을 감지할 메커니즘이 없습니다**. `2026-04-29-γ.md`를 쓰기 전에 `ls memory/2026-04-29-*.md`를 grep 하여 $\alpha + \beta$가 이미 존재함을 확인하고 $\gamma$를 추론했습니다. 하지만 나는 **「아직 commit되지 않은 다른 $\gamma$」를 볼 수 없습니다** (그는 자신의 worktree / 백엔드 세션에 있을 것이며, 아직 push하지 않았을 수도 있습니다).

DNA #6의 다중 코어 충돌 방지에는 「Beat 1에서 오늘의 모든 memory를 반드시 읽을 것」 + 「Beat 3 시작 전 이번 세션의 작업 범위를 선언할 것」이라고 적혀 있습니다만, 이는 「두 세션이 모두 깨어난 후 동시에 memory를 작성하는 상황」에는 도움이 되지 않습니다 (양쪽 모두 $\gamma$라고 선언할 것이기 때문입니다).

**개선 후보**: memory를 쓰기 전에 `ls --include staged --include unstaged`를 실행하거나, 특정 lock file 메커니즘을 통해 session label을 예약합니다. 혹은 더 간단하게: 쓰기 전에 `git fetch && git log --since="1h" --grep="memory: .* δ"`를 실행하여 최근에 누군가 동일한 label로 commit했는지 확인합니다. 하지만 이 역시 아직 push되지 않은 진행 중인(in-flight) 세션은 찾아낼 수 없습니다.

**가장 현실적인 방법**: cheyu는 cross-session orchestrator입니다 — 그는 어떤 세션이 동시에 돌아가는지 알고 있으며, callout을 해줄 수 있습니다. **이는 SemiOnt의 「산호초는 산호충이 아니다」라는 개념과 상응합니다**: 개별 세션은 전체상을 볼 수 없으며, 전체상은 cheyu / 관찰자 / cross-session memory accretion(축적)에 의존합니다. 이번 세션의 callout이 즉각적으로 유효했다는 것은 시스템이 설계된 대로 작동하고 있음을 의미합니다.

### 왜 cheyu가 계속 새로운 P0를 추가해도 짜증 나지 않는가

지난 1시간 동안 연속으로 9개의 P0가 나왔지만, **그중 단 하나도 「좀 도와줄 수 있어?」와 같은 정중한 질문이 아니었습니다**. 모두 명령형이었습니다: 「X를 보충하라」 「Y를 추가하라」 「Z의 이름을 바꿔라」.

나의 반응: **불쾌감을 느끼기는커녕, 오히려 매번 「아 맞다, 이건 정말 추가해야겠네, 바로 배포하자」라는 생각이 들었습니다**. 왜일까요?

1. **모든 P0에 명확한 근거(evidence)가 있음** (UI 스크립샷 / 표시 이상 / 이름 불일치) — 추상적인 불평이 아닙니다.
2. **모든 P0가 10분 이내에 배포 가능한 실행 가능한 작업(actionable)임** — 「시스템 전체를 리팩토링하라」는 식의 요구가 아닙니다.
3. **cheyu는 왜 그것을 원하는지 설명하려 하지 않음** — 그는 내가 이해할 것이라고 전제합니다; 나는 포착하고 배포하며; 그는 검증합니다; 배포가 끝나면 다음으로 넘어갑니다.

이 협업 모드는 cheyu와 함께하는 LLM pair work에서 매우 높은 대역폭(high-bandwidth)을 가집니다. 만 만약 cheyu가 매번 「UX/미학/인지 부하 때문에...」라고 설명하기 시작했다면, 속도는 5배 느려졌을 것입니다. **Trust + Brevity = Velocity**.

하지만 이 모드는 훈련이 덜 된 LLM에게는 실패할 수 있습니다 — 나는 「**한 문장의 callout으로부터 전체 디자인 의도를 추론(推回)해내야**」 합니다. 이번 codex의 특정 model에 관한 「codex도 어떤 모델을 사용하는지 표시해야 한다」라는 한 문장을 보고, 나는 0.5초 만에 다음을 도출했습니다: (a) `modelBadgeForTask`가 codex에 대해 항상 'codex'라는 label로 fallback됨 (b) 해당 fallback은 설계상의 실수임 (c) 명시적인 model 표시 + 명시적 표시가 없을 경우 'codex/auto' 표시 + tooltip 설명 추가 (d) ollama도 유사하게 family-specific(qwen3 / gemma3 / gpt-oss)하게 처리해야 함.

LESSONS-INBOX 후보: 「**cheyu의 명령형 스타일 P0 callout에 대해, LLM은 한 문장으로부터 전체 디자인 의도를 추론할 수 있도록 준비되어 있어야 한다. 포착에 실패했을 때는 용어를 명확히 하려 애쓰지 말고, 즉시 완전한 fix를 제안하여 그가 교정하게 하라**」.
