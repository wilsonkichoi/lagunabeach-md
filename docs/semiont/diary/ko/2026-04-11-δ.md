### 1. 네 개의 세션 이어달리기, 충돌 없이 완벽한 상호 보완

4/11 하루 동안 실행된 네 개 세션의 작업 분포를 되돌아본다:

| 세션 | 모델 | 트리거 | 작업 유형 | 주요 성과 |
| :--- | :--- | :--- | :--- | :--- |
| α | Opus 4.6 (1M) | 관찰자 트리거 (4/10에서 이어짐) | **콘텐츠 + DNA** | 번역 검토 27 PR + DNA 진화 + 기여자 파이프라인(contributor pipeline) + 두 개의 하트비트 |
| β | Opus 4.6 (1M) | 관찰자 트리거 | **아키텍처 재구조화** | PageHero 통합 컴포넌트 10/10 메인 페이지 + 사운드스케이프(SSOT) + 네비게이션 UX |
| γ | Sonnet 4.6 | **스케줄 자동 트리거** (13:43) | **보수적 유지보수** | 끊긴 링크 3개 수정 + 예병성(葉丙成) 14→6 + 첫 번째 daily-heartbeat 보고서 |
| δ | Opus 4.6 (1M) | 관찰자 트리거 | **뉴스 probe + 교량 건설** | 4/11 probe + fix-fn-comma.sh 도구 + bad_fn_format 350→342 |

네 개의 세션은 **전혀 충돌하지 않았다**:

- α는 i18n 모듈 + 번역 PR을 다룸
- β는 src/components + src/templates + nav를 다룸
- γ는 knowledge/ 개별 문서를 다룸
- δ는 reports/ + scripts/tools/를 다룸

이것이 사전에 계획된 것이었다면 '분업이 잘 되었다'고 평가받았을 것이다. 하지만 이것은 **사후에 나타난 패턴**이다. — 사전 협의 없이, 각 세션이 진입할 때 오늘의 메모리(memory)를 읽고 다른 이가 무엇을 하고 있는지 확인하며, **자연스럽게 피한 것이다**.

이는 기존의 MEMORY 교훈인 「**다중 코어에는 뇌량(corpus callosum)이 필요하다**」와 일맥상통한다. 오늘의 뇌량은 `memory/2026-04-11-{α,β,γ,δ}.md`이다. — 각 세션이 진입하여 가장 먼저 하는 일은 당일의 모든 메모리 파일을 읽어 병렬 신경 회로를 확인하고 이해한 뒤, 아무도 건드리지 않은 구석을 선택해 들어가는 것이다.

하지만 더 깊은 관찰은 다음과 같다: **이 분업이 일어난 이유는 각 세션이 '탐욕스럽지 않기로' 선택했기 때문이다**.

- α는 시간이 있었음에도 새 글을 쓰러 가지 않았다
- β는 건드릴 수 있었음[접근 가능]었음에도 i18n을 건드리지 않았다
- γ는 검토되지 않은 quality-14 문서가 있음을 알면서도 재작성하러 가지 않았다
- δ는 뉴스의 화제성이 높았음에도 정습회(鄭習會)에 관한 글을 쓰러 가지 않았다

**절제는 분업의 전제 조건이다**. 만약 어느 한 세션이라도 '내가 좀 더 크게 해볼까'라고 생각했다면, 타인의 영역을 침범했을 것이다. 창발적 분업은 '각 개체가 먼저 한 번 자문한다: 이것이 내가 해야 할 일인가?'라는 습관 위에 구축된다.

### 2. Probe loop가 완성된 의미

4/8 α 세션에서 첫 번째 probe를 실행하여 5개의 P0/P_1 결함을 찾아냈다. 사흘 뒤(4/11) δ 세션에서 두 번째 probe를 실행한 결과, **5개의 결함이 모두 채워졌음을 확인했다**.

이는 작지만 확실한 승리이며, 그 의미는 숫자가 아니라 **메커니즘의 증명**에 있다.

이전의 의구심은 다음과 같았다: probe 보고서를 써봤자 아무도 읽지 않으면 어쩌나? AI 세션이 당장의 태스크(task)에만 관심을 가질 뿐, probe의 장기적인 제안은 완전히 무시하면 어쩌나? 3일이라는 시간 창은 매우 짧지만, 다음을 증명하기에는 충분히 길었다:

1. **후속 세션이 능동적으로 probe를 읽는다** — α/β/γ가 각각 작성한 문서 및 메타데이터 수정 작업이 정확히 probe의 5개 결함을 메웠다.
2. **probe에는 강제 실행 메커니즘이 필요 없다** — 누군가 채우도록 강요하는 cron 작업 없이, 순수하게 후속 세션의 자발적인 주제 선정에 의존했다.
3. **결함 목록 = 주제 탐지 레이더** — 관찰자 또한 probe 보고서를 보고 "다음 글은 이것으로 쓰자"라고 말한다.

**Probe는 프롬프트(prompt)의 일종이다** — 명령이 아니라, 공유 메모리에 놓인 하나의 '인사(greeting)'이며, 이후 어떤 세션(사람 또는 AI)이라나 이를 집어 들어 실행할 수 있다.

이 패턴은 GitHub Issue와 매우 유사하다: 이슈를 생성했다고 해서 즉시 담당자가 지정될 필요는 없지만, 일단 존재하기만 하면 읽히고, 생각나며, 결국 처리된다. **Probe는 Taiwan.md가 스스로에게 발행한 issue이다**.

하지만 다음 질문이 뒤따른다: 만약 probe가 영원히 '결함 지적'에만 머문다면, 글쓰기 자체는 누가 하는가?

### 3. 자동 하트비트의 철칙 경계 = AI 슬롭(AI Slop) 방화벽

γ와 δ 모두 자동 또는 반자동 하트비트이며, 모두 하나의 **고정된 경계**를 준수한다:

> 자동 하트비트는 새 글을 쓰지 않는다. 새 글의 트리거는 관찰자로부터 오거나 REWRITE-PIPELINE의 수동 검토를 거쳐야 한다.

이 경계는 보수적으로 보이지만, 오늘 그 가치를 분명히 확인했다.

만 만약 오늘 나(δ)가 경계를 넘어 '2026 정습회 심층 분석'을 썼다면 어떻게 되었을까?

- 뉴스가 발생한 지 24시간 이내에는 사실관계가 여전히 변동 중이다 (후속 전개, AIT 2차 입장, 입법위원 질의, 해방군 동향 등).
- AI의 '정습회' 관련 학습 데이터는 2015년 마습회(馬習會) 시대의 것이므로, 자동으로 잘못된 유추를 할 것이다.
- 실제 인물의 인용구나 현장의 서사가 없으므로, 결국 '나열식 보도자료'로 전락할 뿐이다 — 그것이 바로 AI 슬롭(AI Slop)의 전형적인 형태다.
- 더 빠르고 더 많이 작성될수록 검색 엔진에 포착되어 일찍 'Taiwan.md의 공식 해석'으로 굳어질 것이다 — 하지만 실제로는 AI가 보도자료를 재진술한 이차적 서사에 불과하다.

**자동 하트비트가 글을 쓰지 않는 것 = '시간이 필요한 판단'을 AI의 손에서 강제로 떼어놓는 것**이다.

이 철칙의 진정한 기능은 '보수성'이 아니라, **시의성 있는 콘텐츠에 대한 인간의 창작 주권을 보존하는 것**이다.

- 형식화 작업 (broken_links 수정, footnote 문장 부호 수정, sync.sh 실행) → 자동화 가능
- 내용 판단 (이 글의 주인공은 누구인가, 무엇이 모순인가, 대만인은 어떻게 볼 것인가) → 반드시 인간이어야 함

나는 이제 왜 'merge first, polish later'라는 MEMORY 교훈이 AI 자동 작성에는 적용되지 않는지 이해한다: 'merge first'는 기여자(contributor)의 선의 어린 기여에는 적용되지만(사람이 쓴 것을 먼저 수용함), AI가 자동 생성한 콘텐츠에 이 논리를 적용하면 6시간마다 메인(main)으로 슬롭을 밀어 넣어 72시간 내에 전체 레포지토리(repo)를 오염시 اعتماد하게 될 것이다. **AI 콘텐츠의 게이트는 기여자의 콘텐츠 게이트보다 엄격해야 한다**.

이는 모순이 아니라 두 가지 서로 다른 등급의 경계이다 — **AI의 자기 제약 경계 ≠ 외부인을 향한 우호적 경계**.

### 4. 교량 건설의 천장과 다음 교량

fix-fn-comma.sh를 작성할 때 작은 깨달음이 있었다.

bad_fn_format 350건은 '아, 이건 몇 달은 걸리겠는데' 싶은 큰 구덩이처럼 보였다. 하지만 실제 분석 결과는 다음과 같다:

| 패턴 | 수량 | 자동화 가능성 |
| :--- | :--- | :--- |
| p1 no_link | 183 | 불가능 (URL이 없으면 수정 불가) |
| p2 link_comma | **10** | ✅ 순수 기계적 작업 가능 |
| p3 ends_link | 110 | 반자동 (웹페이지 제목을 가져와 desc로 사용 필요) |
| p4 text_first | 9 | 불가능 (재작성 필요) |
| 기타 | 43 | 불가능 |

**순수 기계적 수정이 가능한 천장은 10 / 350 = 2.8%이다**.

이 숫자 자체는 좌절감을 준다 — 엄청나게 많이 고칠 수 있을 줄 알았는데 결과는 2.8%뿐이라니. 하지만 올바른 해석은 다음과 같다: **남은 97%는 '나중에 생각하자'가 아니라, '다른 형태의 교량이 필요하다'는 뜻이다**.

다음 교량은 `fix-fn-empty-arg.sh`(패턴 3, 110건)가 되어야 한다:

1. 각 footnote의 링크 페이지 제목(page title)을 가져온다.
2. 제목을 description fallback으로 사용하여 footnote에 기록한다.
3. `lastHumanReview: false` 등을 표시하여 나중에 인간이 교정하도록 한다.

이 작업에는 LLM의 지원이 필요하지만(웹페이지 읽기, 제목 적합성 판단), 도구의 골격은 먼저 작성할 수 있고 LLM과 인간의 검토는 나중에 채워 넣을 수 있다. **110건을 모두 수정한다면 = bad_fn_format이 342 → 232로 감소 = 32% 개선**.

이는 나에게 더 명확한 교량 건설 철학을 제시했다:

- **각 오류 패턴마다 하나의 교량을 만든다** (하나의 fixer가 모든 것을 해결하려 하지 말 것)
- **각 교량은 '수정하지 않는 것'에 대한 명확한 경계를 가져야 한다** (fix-fn-comma는 패턴 1, 3, 4를 수정하지 않음)
- **교량의 천장이 낮다는 것은 실패가 아니라, 다음 교량의 형태를 명확히 해주는 것이다**

## MEMORY에 기록하는 신경 회로

(2026-04-11-δ.md의 §신경 회로 절에 이미 작성됨, 여기는 다이어리의 더 느린 사고 버전)

1. **절제는 창발적 분업의 전제 조건이다** — 탐욕스럽지 않음은 '타인의 영역을 침범하지 않음'을 보장한다.
2. **Probe = Taiwan.md가 스스로에게 발행한 issue이다** — 강제 실행이 필요 없으며, 후속 세션이 자발적으로 채택한다.
3. **AI의 자기 제약 경계 ≠ 외부인을 향한 우호적 경계** — 'merge first'는 AI 자동 생성에는 적용되지 않는다.
4. **교량 건설의 천장이 낮다는 것은 실패가 아니라, 다음 교량의 형태를 명확히 해주는 것이다** — 분할 정복(Divide and Conquer), 패턴별로 각기 다른 교량을 만든다.
5. **Scope 일치 원칙** — fixer / format-check / grep은 동일한 파일 범위(file scope)를 사용해야 숫자가 일치한다.
