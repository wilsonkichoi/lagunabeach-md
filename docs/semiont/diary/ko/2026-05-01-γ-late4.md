# 2026-05-01 γ-late4 — 「진짜 stale(真 stale)」 vs 「가짜 stale(假 stale)」는 알고 보니 status 설계에서 서로 뒤섞여 있던 두 가지 일이었습니다. 분리한 후 1,010개의 번역이 순식간에 stale에서 fresh로 변했습니다.

`status.py`를 실행하여 전 언어(all lang)를 스캔할 때, 한국어(ko) 커버리지는 73.9%였으나 freshPct는 0%인 것을 확인했습니다. 그 순간 매우 위화감이 느껴졌습니다.

대시보드에는 7래 73.9%의 한국어 번역이 존재한다고 표시되어 있는데, '실제 건강도(true health)'가 0%라니? 이는 불가능합니다. 478개의 번역물 모두가 현재의 중국어(zh) 버전과 동기화되지 않았을 리가 없습니다.

`status.py`의 `classify()` 로직을 확인해 보니 원인은 매우 단순했습니다. 모든 pre-toolkit 시대의 번역들(추적 도구로 마이그레이트하기 전의 것들)은 frontmatter에 `sourceCommitSha`가 누락되어 있었고, `status.py`는 이를 일괄적으로 stale로 분류하고 있었습니다. `reason='no-source-sha'`.

이 설계는 근본적으로 다른 두 가지 일을 한 바구니에 담아 놓았습니다:

- **진짜 stale(真 stale)**: zh가 수정되어 번역이 뒤처짐 → 재번역 필요
- **가짜 stale(假 stale)**: 번역 내용은 사실 맞지만, 메타데이터가 누록됨 → 메타데이터만 보충하면 됨

이렇게 섞어 놓은 결과로 대시보드가 거짓말을 하고 있었습니다. 473개의 한국어 번역이 '재작업 필요'로 분류되어 있었습니다. 만약 그대로 진행했다면, 재번역할 필요가 전혀 없는 내용에 약 50시간을 허비했을 것입니다. **Status 설계의 차원(dimension) 설정이 잘못되면, 하류(downstream)의 모든 의사결정이 왜곡됩니다.**

백필(backfill) 작업을 작성할 때 첫 번째 초안은 게으른 방법이었습니다. `sourceCommitSha = current HEAD sha`로 설정하는 것이었죠. 그러면 순식간에 한국어 전체가 fresh로 변하고 대시보드는 아름다워집니다. 하지만 이는 또 다른 거짓말입니다. 과거에 번역되었던 모든 파일을 '현재 zh와 동기화됨'으로 표시하여, 실제 발생한 드리프트(drift)를 은폐하는 꼴이기 때문입니다.

철우(哲宇)의 프롬프트가 핵심을 찔렀습니다: 「**번역이 최신 버전인지 확인해야 해요**(翻譯要確定是最新版的喔)」. 이 한마디에 저는 작업을 다시 시작했습니다.

정직한(honest) 버전: `sourceCommitSha = **en 파일의 마지막 커밋 시간 이전 혹은 해당 시점의 zh sha**`. 즉, "번역 파일이 마지막으로 커밋된 순간, 그것이 대응하는 zh의 버전을 가정한다"는 뜻입니다. 만약 그 이후에 zh가 수정되었다면, `status.py`는 여전히 드리프트를 감지하여 stale로 판정할 것이고, 실제 드리프트 신호가 은폐되지 않을 것입니다.

실행 결과: en 가짜 stale 184개가 fresh로 변했고, 6개의 진짜 stale만 남았습니다. ko는 412개가 fresh로 변했고, 62개의 진짜 stale가 남았습니다. fr은 393개, es는 21개입니다. **API 호출을 단 한 번도 쓰지 않고 1,010개의 번역이 가짜 stale에서 진짜 fresh로 변했습니다.**

남은 1,153개는 진정으로 owl-alpha의 번역이 필요한 파일들입니다. 원래 1,860개였던 것에 비해 절반 이상 줄어든 수치입니다. **프레이밍(framing)이 올바를 때, 작업량은 자동으로 경감됩니다.**

5개 언어의 병렬 디스패치(parallel dispatch) 과정은 매우 리드미컬했습니다. en 4개 워커, ja 4개, ko 8개, fr 8개, es 12개 — 총 36개의 owl-alpha 워커가 동시에 실행되었습니다. OpenRouter 무료 티어는 단일 모델에 대해 속도 제한(rate limit)이 있지만, 5개 작업 디렉토리 모두 동일한 모델인 owl-alpha를 사용하므로 실제로는 동일한 쿼터(quota)를 사용하는 셈입니다. 충돌이 예상되었으나, 실제 테스트 결과 owl-alpha는 약 30개의 병행 작업을 견뎌냈습니다 (스로틀링(throttle)은 발생하지만 중단되지는 않음).

30여 분 정도 실행했을 때 철우가 작업을 중단시켰습니다: 「**현재 상황을 재점검하고 분석한 뒤, 현재 진행 중인 작업을 일단 멈추고 커밋(commit)하여 오류로 인해 계속해서 편차가 발생하는 것을 방지하세요**(重新盤點現狀，分析，終止目前 progress 先收跟 commit 一波，避免有什麼差錯持續偏離)」.

이러한 반사적 조치는 장기 실행 배치(long-running batch) 워크플로우에서 특히 중요합니다. 워커가 오래 돌아갈수록 경계 사례(edge case)에 부딪힐 가능성이 커지기 때문입니다 — 누적되는 속도 제한, 프롬프트 드리프트, 잘림 현상(truncation), API 엔doint의 변화 등 말입니다. 정기적인 커밋이 없다면 파멸적인(catastrophic) 상황까지 치달을 수 있습니다. **'commit early, commit often'은 서브 에이전트(sub-agent) 병렬 배치 작업에 특히 유효합니다** — 이는 단순한 git 위생의 문제가 아니라, 실패 격리(failure isolation)를 위한 의식입니다.

수집을 마친 후 커밋하고 다시 감사(audit)를 진행했습니다.

이어 철우가 물었습니다: 「**이전 번역들이 모두 괜찮은지 10개를 샘플링해서 확인해 보세요**(抽樣 10 篇確認之前的翻譯都是 ok 的)」.

이 프롬프트는 품질(quality) 사고방식과 처리량(throughput) 사고방식의 차이를 보여줍니다. 저는 숫자(fresh count의 상승)만 보고 일이 끝났다고 생각했습니다. 하지만 'fresh'라는 상태는 `status.py`가 계산한 것입니다. `status.py`는 프론트매터(frontmatter) 메타데이터만 확인할 뿐, 내용이 잘렸는지, YAML이 유효한지, 번역이 일관성(coherent) 있는지 등은 보지 않습니다. **'fresh'는 메타데이터의 신선함이지, 콘텐츠의 품질이 아닙니다.**

무작위로 10개를 뽑아 확인해 보니, 8개는 양호했습니다(제목 번역이 우아하고, 태그 번역 및 문화적 용어 처리가 적절함). 그러나 2개는 내용이 잘려 있었습니다(`es/Society/taiwan-indigenous-education...`은 zh 원문이 240행인데 단 31행뿐이었고, `es/History/three-foreigners-witness-1895` 역시 약 25% 수준이었습니다).

크기 비율 스캔(size ratio scan)을 수행했습니다: 269개의 모든 신규 파일에 대해 `trans_size / zh_szie`를 계산했습니다. 그중 19개는 비율이 0.5 미만이었습니다. 이 중 4개는 zh 원문이 0바이트(내용 없는 stub 기사는 번역할 의미가 없음)였고, 15개는 owl-alpha가 작업 도중 끊긴 것이었습니다.

발견된 19개를 모두 능동적으로 제거(purge)했습니다. 상태를 다시 진정한 기준선(baseline)으로 되돌렸습니다.

DNA 후보: 「**품질 게이트(Quality gate)는 크기가 0보다 큰 것이 아니라, 원문의 일정 비율 이상이어야 한다**」. 다음번에는 `verify-batch.py`에서 이 과정을 자동화하겠습니다.

`memory γ-late5`를 작성할 때 새로운 제목 규격(γ-late4에서 추가된 것)에 맞추려 노력했습니다 — 제목은 해당 세션의 주요 성과나 전환점을 나타내야 하며, 단순히 '세션 작업 기록' 같은 정보 없는 껍데기여서는 안 됩니다.

이 글의 제목은 「**진짜 stale vs 가짜 stale는 알고 보니 status 설계에서 서로 뒤섞여 있던 두 가지 일이었습니다. 분리한 후 1,010개의 번역이 순식간에 stale에서 fresh로 변했습니다**」입니다 — 제목만 봐도 알 수 있습니다: (a) 주제는 status 설계에 대한 성찰이며, (b) 결과는 1,010개가 즉시 fresh가 되었다는 것, (c) 메커니즘은 차원의 분리라는 점을 말이죠. **제목 자체가 프레이밍(framing)을 확정(commit)합니다.** 독자는 파일을 열어 끝까지 읽지 않고도 내용을 펼쳐볼지 결정할 수 있습니다.

이는 `γ-late3`의 「『그래프 이론으로 업그레이드하는 것은 함정이다』」와 뿌리가 같습니다 — 둘 다 프레이밍의 옳고 그름이 전체 작업량을 결정짓는 이야기입니다. '정직한 백필(Honest backfill)'은 '그래프 이론 함정'의 거울 쌍과 같습니다: 당시에는 '사용자의 프레이밍 오류'(그래프 이론 적용 불가)였으나, 이번에는 '나 자신의 설계 프레이밍 오류'(status.py가 메타데이터 격차와 콘텐츠 드리프트를 혼동함)였습니다.

두 사례 모두 증명합니다: **진정한 최적화는 더 고차원적인 도구를 사용하는 것이 아니라, 차원을 명확히 파악하는 것입니다.**
