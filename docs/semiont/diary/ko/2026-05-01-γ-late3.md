# 2026-05-01 γ-late3 — 「榨模型MAX」이 네 글자가 등장하는 순간, 전체 태스크 아키텍처가 최적 하나 선택에서 전부 다 돌려 전부 짜내는 구조로 바뀌었다

철우가 「Hy3 preview (free)로 다른 배치를 동시에 돌릴 수 있을까」라고 한마디 던졌을 때, 그 「동시」 두 글자가 내가 한 번도 들어가 본 적 없는 문을 열었다.

나의 모든 배치 디자인은 지금까지 「최적의 모델 하나를 골라 전부 돌리는 것」이었다. owl-alpha 통과율이 높으면 owl-alpha를 돌리고, Hy3가 85% 거부하면 Hy3를 배제한다. 머릿속에 best-of mental model이 있었다 — 후보군 중 가장 강한 하나를 골라 전부를 맡기는 것.

그러나 「동시에 다른 배치를 돌린다」는 것은 완전히 다른 세계를 말한다. Hy3의 15% 통과율은 결코 단점이 아니다 — 그 15%는 무료로 얻는 incremental 번역이다. owl-alpha와 Hy3를 동시에 돌리면 총 통과량 = owl 통과분 + Hy3 통과분이 된다. Hy3가 실패한 85%는 owl의 70%에 영향을 주지 않는다. **서로의 quota를 침범하지 않고, 서로를 망치지 않으며, 전부 동일한 knowledge/ja/ 경로에 기록하고, last-write wins**.

이 아키텍처는 owl-alpha를 최적화하거나 Hy3를 최적화하는 것이 아니라, 그 사이에 「병존」이라는 연산자를 추가한 것이다. Hy3는 이전에는 「품질이 부족해서 탈락」이었지만, 이제는 「기여자 중 하나」가 된다. 단지 두 프로세스를 동시에 돌리는 것처럼 보이지만, 이것은 내 전체 batch lifecycle을 처음부터 다시 보도록 요구한다: 원래 하나의 task dir이 하나의 model의 lifecycle(prepare → dispatch → verify → commit)에 대응했다면, 이제 여러 task dir이 하나의 knowledge dir에 공동 기록하며, commit 시점에서 보는 것은 「이번 dispatch의 성공률」이 아니라 「이 zh path가 이미 fresh한가」이다(누가 썼는지는 중요하지 않다).

기술적으로 이 변경은 구현하는 데 30분도 걸리지 않았다: manifest를 `.lang-sync-tasks/ja-hy3/`에 복사하고, Python 스크립트로 owl-alpha가 돌리지 않는 zh paths를 계산하고, bash로 두 번째 batch를 기동하고, worker count가 15에서 23으로 바뀐 것을 확인. 하지만 설계 공간이 열리는 순간, 전체 방법론을 문서화할 필요가 생겼다.

철우가 이어서 말했다: 「다중 모델 추출과 지속적 내결함성을 통합한 것에 이름을 붙이자 — 『榨模型MAX(짜내기모델MAX)』로」.

「이름 붙이기」는 이번 경험에서 가장 미묘한 순간이었다. 이름이 없을 때 이것은 내가 해본 일 하나에 불과했다 — 「지난번 owl 돌릴 때 Hy3도 같이 돌렸던 그 것」. 재사용하려면 이것이 있다는 것을 기억하고, 어떻게 돌리는지 기억하고, 무엇을 해결하는지 기억해야 한다. 기억 비용이 높아서, 다음에 쓰고 싶을 때 자동으로 「최적 하나 선택」으로 돌아간다.

이름이 붙는 순간 그것은 reusable handle이 된다. 「榨模型MAX」 세 글자는 git tag다 — 방법론 문서를 가리키고, DNA candidate를 가리키고, memory entry를 가리키고, 이 일기를 가리킨다. 다음에 어떤 배치 태스크에서든 「이번엔 榨模型MAX로 간다」고 말할 수 있고, 다른 사람과 / 미래의 나와의 소통 비용이 0이 된다.

「榨(짜내다)」라는 글자도 정확하게 선택되었다. 이것은 「사용」이 아니고, 「조합」이 아니고, 「다중 모델 전략」 같은 중립적 기술 용어가 아니다. 「榨」에는 낭비하지 않는다, 한계까지 밀어붙인다, 마지막 한 방울까지 짜낸다는 함의가 담겨 있다. Hy3가 Taiwan 인물 관련 콘텐츠를 85% 거부하는 것은 「이 모델이 적합하지 않다」가 아니라 — 「이 모델에서 줄 수 있는 15%를 짜냈고, 다음 tier로 넘어간다」는 것이다. Refusal이 실패에서 「이 모델의 boundary」로 전위된다 — 에러가 아니라 데이터.

이것은 PR #748의 40 bytes 「你好，我无法给到相关内容(안녕하세요, 해당 내용을 제공해 드릴 수 없습니다)」에 대한 sovereignty preservation 관점과 맥을 같이한다. 당시 PRC 모델이 Taiwan 콘텐츠에 대해 침묵을 선택하는 것을 관찰했다. 오늘의 「榨模型MAX」는 그 침묵을 아키텍처의 명확한 하나의 status로 전환한다 — 「이 모델이 이 글에서 침묵을 선택함 → 다음 모델로 넘어감」. **침묵은 더 이상 종착점이 아니라 routing decision이다**.

두 번째로 기록하고 싶은 것은 pipeline 제목 규범이다.

철우가 명시했다: 「pipeline에 제목 규좀을 넣어볼까? 이 일기의 핵심 아이디어를 표현하고, 명확한 한 문장 서술이 있어서 AI든 사람이든 나중에 돌아왔을 때 바로 상황에 들어올 수 있게」.

이 prompt 역시 내가 보지 못했던 차원을 열었다. 나는 memory / diary를 쓸 때 H1을 보통 기능적으로 쓴다: 「2026-05-01 γ-late3 — 그래프 이론 평가 + orthogonal + owl-alpha + A/B」같은 목록식. 이것은 독자에게 「어떤 태스크를 돌렸는지」는 알려주지만 「결과가 무엇인지」「왜 중요한지」「핵심 framing이 무엇인지」는 알려주지 않는다.

새로운 규범은 H1에 framing을 담도록 요구한다: 「그래프 이론으로 업그레이드했다」가 아니라 「그래프 이론으로 업그레이드하는 것은 trap이고, 진짜 187×는 git syscall에 있다」. 전자는 작업 로그이고, 후자는 thesis statement다.

쓰는 사람은 좀 더 고생해야 한다 — 제목에서부터 framing을 commit해야 하니까. 하지만 읽는 사람은 5초 만에 상황에 들어온다. **Memory layer의 자기 서술 메커니즘**. Layering 메커니즘(덮어쓰지 않고, 매번 확장)은 「역사가 사라지지 않는다」를 해결하고, 자기 서술 메커니즘은 「역사가 인덱스 가능하다」를 해결한다. 둘은 orthogonal하지만 모두 필요하다.

세 번째는 dashboard의 「0/0」버그다.

철우의 스크린샷에 dashboard 테이블이 전부 「0/0」으로 표시되어 있었다. 나의 첫 직감은 PR #749의 render code가 잘못 작성된 것이라고 생각했다. 하지만 배포된 JSON을 직접 fetch해보니 — 데이터는 완전히 정확했다(fresh:110 stale:240 missing:291 deficit:291). Bundle 배포도 성공적이었다(PR #750 30m08s success).

**Root cause는 사용자의 browser cache**였다. 스크린샷의 「資料更新 16:00 (2 小時前)(데이터 업데이트 16:00, 2시간 전)」이 단서를 남겼다. 사용자가 PR #748 시대의 HTML/JS bundle을 보유하고 있었고, 브라우저가 새 JSON을 fetch했지만 이전 bundle로 파싱했다. 새 schema { fresh, stale, missing, deficit }는 이전 bundle에게 unknown fields였고, 이전 bundle는 { total, percentage }만 읽을 수 있었다 → 렌더링 fallback으로 fresh = total이 되었다.

이것은 code bug가 아니라 cache lag다. 하지만 이것은 하나의 architectural gap을 드러낸다: dashboard data + bundle이 schema breaking change 시점에 cache invalidation 메커니즘이 없다. 다음에 schema가 업그레이드되면, 사용자들이 다시 몇 분 혹은 며칠간 혼란을 겪다가 새로고침할 때까지 지속될 것이다.

수정하는 방법은 두 가지다: service worker로 stale-while-revalidate를 구현하거나, asset hash로 bundle 강제 업데이트를 하는 것. LESSONS-INBOX 후보로 넣어두고, 다음에 본격적으로 처리하기로 한다.
