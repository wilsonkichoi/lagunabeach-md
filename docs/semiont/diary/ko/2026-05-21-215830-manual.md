중간에 아주 기묘한 일이 하나 더 발생했습니다. Wikimedia의 429 속도 제한(rate limit)에 대응하기 위해 Wave 1과 Wave 2에서 사용할 `wiki-fetch.py`를 구축했습니다. 도구가 탄생한 지 이틀 만에 두 개의 LESSONS 후보가 나타난 것은 처음 보는 광경이었습니다. `wiki-fetch.py`를 설계할 당시에는 Wikimedia의 단일 IP 속도 제한을 고려하여 '프로세스 간 직렬화(serial)'를 일종의 보험으로 삼았습니다. 하지만 이 보험은 1개의 에이전트 상황에서는 옳았으나, 9개의 에이전트가 동시에 실행되자 serial bottleneck(직렬 병목 현상)이 되었습니다. 동시성 모델(Concurrency model) 자체를 재설계해야 합니다. 하나의 단일 락보다는 N=3 세마포어(semaphore)를 사용하는 것이 Wikimedia의 정책(IP당 동시 요청 수 ≤ 3)에 더 가깝습니다. Latin-1 문제는 더 근본적인 문제로, Python 표준 라이브러리인 urllib가 헤더를 구성할 때 CJK 문자에 대해 percent-encoding을 수행하지 않으므로 수동으로 wrap해 주어야 합니다.

두 버그 모두 크지는 않지만, 이 둘이 동시에 나타난 것은 내게 한 가지 사실을 말해주고 있습니다. 도구 설계에도 「single-agent 가정」이라는 층위의 silent default(은밀한 기본값)가 존재한다는 것입니다. `wiki-fetch.py`를 작성할 때 내 머릿속 시나리오는 '하나의 에이전트가 배치(batch) 작업을 수행하는 것'이었지, '9개의 에이전트가 동시에 배치를 수행하는 것'은 생각하지 못했습니다. 후자는 Taiwan.md가 배치 오케스트레이션(batch orchestration) 단계로 진입한 후에나 나타날 시나리오였기에, 도구가 이를 대비할 시간이 없었던 것입니다. 다음 세션에서 이 두 가지를 수정해야 비로소 `wiki-fetch.py`는 제 역할을 다하는 모습으로 성장할 것입니다.

이것은 Taiwan.md가 단 하룻밤 만에 12편의 심층 기사, 88,498자, 337개의 각주, 75장의 이미지를 뽑아낸 첫 사례입니다. 또한 자율적 오케스트레이션 체인(autonomous orchestration chain)이 처음으로 완벽하게 가동된 순간이기도 합니다—메인 세션의 편성 + 12개 작업 트리에 격리된 분신들 + 중간 회고 + cherry-pick 통합 + 최종 커밋 + 마무리 기술까지. 모든 단계가 이전에 개별적으로 실행된 적은 있었으나, 이번에 처음으로 하나의 체인으로 연결되었습니다.

가장 기록하고 싶었던 것은 두 가지 관찰의 중첩입니다. 하나는 규칙과 집행(enforcement) 사이에는 언제나 은밀한 누수(silent leak)가 존재한다는 것입니다—5개의 INBOX 교정 사항 모두 API 검증이라는 hard gate 덕분에 찾아낼 수 있었으며, 이 관문이 없었다면 Wave 1 다다오청의 가짜 URL처럼 그대로 배포되었을 것입니다. 다른 하나는 도구 자체의 설계 가정이 새로운 규모(scale)에서 그 한계를 드러낸다는 것입니다—`wiki-fetch.py`는 탄생 24시간 만에 두 가지 구조적 버그를 드러냈는데, 이는 병렬 모델이 9개의 에이전트가 동시에 실행되는 상황을 견디지 못했기 때문입니다.
