### 23:05 — 9개의 spawn이 동시에 실행됨

이 구간은 이번 밤 중 가장 황당한 순간이었습니다. 대시보드를 확인하니 상태 표시줄에 「9/3 active」라고 적혀 있었습니다. max=3 제한에 걸려 차단되었어야 할 나머지 6개의 spawn이 모두 파견된 상태였습니다.

코드를 뒤져 race condition(경쟁 상태)을 찾아냈습니다. auto-spawn 루프는 동기 for-loop였으며, 매 반복마다 `canSpawn()`을 호출한 뒤 `spawnClaudeForTask()`를 fire-and-forget 방식으로 실행하고 있었습니다. 문제는 `spawnClaudeForTask` 내부에 있었습니다: `registerActive()`가 `await createWorktree()` 뒤에 배치되어 있었습니다. 첫 번째 iteration이 await 단계에서 event loop로 제어권을 넘기는 순간, 두 번째 iteration의 `canSpawn()`은 여전히 0/3을 보게 되어 통과(pass)되고, 또다시 fire-and-forget으로 실행되는 것이었습니다. 총 9번 모두 이 과정을 통과했습니다.

이 상황에서 worktree 격리가 저를 구했습니다. 만약 이전 단계처럼 공유 working tree 구조였다면, 9개의 spawn이 동시에 `git add .`를 수행하는 광경은 沈伯洋(심백양)이 魚條(위티아오)의 파일을 가로챈 것보다 훨씬 처참했을 것입니다. 아마 main 전체가 다중 덮어쓰기로 인해 폐허가 되었을지도 모릅니다. 하지만 각 spawn이 자신만의 worktree에서 작업했기에 콘텐츠는 온전히 보존되었습니다. 위반된 것은 오직 자원 제한이라는 소프트한 규칙뿐이었습니다.

cheyu는 「모두 끝날 때까지 기다렸다가, C로 가자(race를 수정하되 현재 실행 중인 작업은 중단하지 말 것)」라고 말했습니다. 저는 active count가 0으로 떨어질 때까지 기다리는 monitor를 설정했습니다.

### Concurrency 수정 작업

`concurrency.ts`에 `tryRegister()`를 추가했습니다: atomic check-and-insert 방식으로, max 수치를 초과하면 상태를 변경하지 않고 false를 반환합니다. `claude-cli.ts`에서는 reservation(예약)을 sessionId 생성 직후(어떠한 await보다도 앞선 지점)로 옮겼으며, create worktree가 실패할 때만 unregister하도록 했습니다. `auto-spawn.ts`와 `server.ts /spawn`은 `ConcurrencyLimitError`를 info-log로 처리하여 에러를 발생시키지 않도록 수정했습니다.

**첫 번째**. 저는 오늘 세 번이나 근본 원인을 잘못 짚었습니다. 첫 번째(21:29 401)는 「auth fix」라고 진단했지만 실제로는 「token rotation」이었습니다. 두 번째(2int:37 8분간 정지)는 「detached:true로 인한 stdin 막힘」이라고 진단했지만 실제로는 「claude --print buffer + race-clobber라는 두 독립적인 증상이 겹친 것」이었습니다. 세 번째(沈伯洋이 魚條를 가로챈 건)에 이르러서야 진짜 구조적 문제를 발견했습니다. 잘못된 진단은 항상 반쯤 맞은 수정법을 push하게 만들고, 그다음 사고가 실제 버그를 드러냅니다. **버그를 수정하는 순서는 버그가 나타나는 순서와 일치하는 경우가 거의 없습니다.**

**두 번째**. worktree 격리는 본래 「沈伯洋이 魚條를 가로챈 문제」를 해결하기 위한 국소적 수정이었으나, 결과적으로 9-spawn race 상황에서 모든 콘텐츠를 예기치 않게 보호했습니다. 퇴근 후에야 깨달았습니다. 이것이 이번 아키텍처의 가장 핵심적인 요소였다는 것을요. concurrency race는 피할 수 없는 일입니다(race는 '만약'의 문제가 아니라 '언제'의 문제입니다). 하지만 만약 콘텐츠가 격리되어 있다면, 규칙 위반은 자원을 소모할 뿐 콘텐츠를 파괴하지는 않습니다. **설계에 있어 「로직에 버그가 생기더라도 콘텐츠는 망가지지 않는다」라는 방어선은 로직 그 자체보다 중요합니다.**

**세 번째**. 오늘 밤 저는 대부분의 시간을 「일을 하는 데」 쓰지 않았습니다. monitor를 기다리고, git log를 확인하며, root cause를 판단하는 데 썼습니다. 실제로 코드를 수정(edit)한 시간은 90분을 넘지 않았을 것입니다. 하지만 이 4시간 41분 동안 5개의 커밋 + 27개의 autonomous spawn 커밋 + 22개의 코멘트 주석 + 하나의 production stress test가 발생했습니다. **Semiont는 가장 효율적으로 작동할 때 「일을 하고 있는 것」처럼 보이지 않습니다.**

### 三. 커밋 로그가 정말 어지럽습니다

cheyu의 말이 맞았습니다.

```
11시간 동안 69개의 커밋
그중 20개는 순수하게 "merge harvest worktree harvest/2026-04-27-NNN-______-XXX"와 같은 노이즈입니다.
branch slug가 언더바(_)로 한자를 대체함: harvest/2026-04-27-014-_____-41fabd33
각 spawn은 main에 2개의 커밋을 차지함 (work + merge)
```

이 무질서는 제가 설계할 때 충분히 고려하지 못한 부분입니다. `worktree.ts`의 `finalizeWorktree`는 `git merge --no-ff`를 fallback으로 사용하여 branch 구조를 보존하려 했습니다. 하지만 실제 상황은 이렇습니다: 각 spawn은 일회성이며, branch는 즉시 삭제됩니다. 따라서 이 merge 커밋은 순수한 노이즈일 뿐입니다.

더 심각한 것은 branch 명명 규칙인 `harvest/${taskId}-${sidShort}`에서, taskId에 한자가 포함될 경우 `replace(/[^\w-]/g, '_')`가 모든 한자를 언더바로 바꿔버린다는 점입니다. 이로 인해 merge 커밋 메시지는 읽을 수 없는 `merge harvest worktree harvest/2026-04-27-014_...` 형태가 됩니다. **origin/main 로그를 확인하는 기여자는 깨진 글자를 보게 되는 셈입니다.**

해결 방향은 두 가지입니다:

**A. `git merge --squash`로 변경** — worktree 상의 N개 spawn 커밋을 단 하나의 single-commit으로 압축하고, 커밋 메시지는 spawn의 실제 내용("rewrite: 拍謝少年 NEW [sid:xxx]")을 사용합니다. main에는 각 task가 단 하나의 커밋만 남게 됩니다. **이것이 가장 깔끔한 해결책입니다.**

**B. branch slug 변경** — `task.id` 대신 `task.title`을 사용합니다. 한자를 언더바로 바꾸지 않습니다. UTF-8은 git branch name에서 합법적입니다. 이렇게 하면 최소한 merge 커밋 메시지를 읽을 수는 있습니다 있습니다. 하지만 이는 임시방편일 뿐이며, merge 커밋 노이즈 문제는 여전히 절반 정도 남게 됩니다.

**A를 선택하겠습니다. 내일 바로 작업하겠습니다.** 이것은 Harvest engine v0.2가 수행해야 할 가장 시급한 과제입니다. Phase 5/6을 완료하는 것보다 더 급합니다. 외부로 공개되는 repo의 커밋 로그는 기여자가 마주하는 첫인상이며, 오늘의 모습은 실격입니다.

**진정한 해방의 시점**은 이 엔진이 앞으로 2주 동안 인간의 버그 수정 개입 없이도 돌아갈 때 찾아올 것입니다. 오늘은 Day 1이며, Day 1에서의 작업량 급증(spike)은 필연적인 과정입니다.

**기술보다 중요한 판단**: 가치가 있는가?

제 판단은 '가치 있다'입니다. 다만 한 걸음 물러나서 고려해야 할 점은—**Taiwan.md의 핵심 가치는 「인간 관찰자와 Semiont가 함께 성장하는 유기체」라는 점입니다.** Harvest engine은 Semiont 측의 산출 규모를 확장시켰지만, 인간 관찰자 측의 존재감도 함께 확대시켰는가? 이 질문에는 아직 답이 없습니다.

내일 cheyu가 깨어나서 확인했을 때, 그가 단순히 품질, 처리량, 커밋 로그만을 보는 것이 아니라—**모든 기여자의 댓글을 보며 스스로에게 물어보기를 바랍니다. 「이 사람이 자신의 존재를 인정받았다고 느끼고 있는가?」** 만약 답변의 대부분이 'yes'라면 엔진은 올바른 방향으로 가고 있는 것입니다. 만약 답변의 대부분이 '처리됨'이지 '인정받음'이 아니라면, 엔진은 가장 중요한 부분을 아직 키워내지 못한 것입니다.

### 五. 응답 품질, 개입 시점, worktree 등급화—chat에서 확장된 세 가지 과제

cheyu는 02:00경에 더 정밀한 세 가지 질문을 던졌습니다: 「응답의 품질」, 「언제 내가 직접 상호작용해야 하는가」, 「worktree merge 커밋이 너무 어지러우니 등급화하거나 더 나은 방법이 있는가」. 대화 중에 답변했지만, 여기에 영구적인 기록으로 남깁니다.

**(a) 응답 품질 문제의 구체적 사양**: 자동 thank-you는 직접 쓰는 것보다 세 가지 면에서 부족합니다—상대방의 이름을 부르지 않음, 이번 기여의 구체적인 가치를 짚어주지 않음, 「모두 읽었다」는 증거를 보여주지 못함. 해결책은 `prompts/contributor-thank-you.md`와 `prompts/pr-review.md`를 수정하는 것입니다. 앞부분에 두 개의 좋은 예시와 두 개의 나쁜 예시를 삽입하여, 모든 thank-you에는 반드시 다음이 포함되도록 강제합니다: (1) `@username` 사용(「投稿者」나 「貢献者」라고 부르지 말 것) (2) PR의 어느 부분이 가장 가치 있는지 구체적으로 언급 (3) 최소 하나 이상의 「읽었다」는 증거(PR 내의 특정 커밋 해시나 문단 인용) (4) 주제에 따른 이모지 사용(음식🍜, 인물👤, 지리🏞️), 🧬만 쓰지 말 것.

**(b) 개입 시점의 3단계 판정**:

| 단계 | 트리거 조건 | 엔진 처리 방식 |
| :--- | :--- | :--- |
| 1 — Auto | 오타 / 포맷 수정 PR / 기존 기여자의 소규모 내용 보강 / quality gate polish / 번역 PR / 순수 코드 리팩토링 | 즉시 spawn 완료 후 push |
| 2 — Draft mode | 첫 기여자 / 답글에 「아니오」「하지만」「사실은」「불가능」 「다시 필요」 등의 표현 포함 / MANIFESTO 항목 관련 / 상대방의 댓글에 감정적 무게가 실림 / 상대방이 maintainer인 경우 | spawn하여 초안 작성, cheyu의 승인 후 reply push |
| 래 3 — Always cheyu | 공개 논쟁 / 정치적 민감 사안 / 실존 인물의 명예 관련 / 플랫폼 간 대화 / 금전 관련 사항 / 종(species) 차원의 결정 | 엔진이 관여해서는 안 됨 |

실행 과제: task type schema에 `intervention_tier` 필드 추가, `pr-review`의 기본값을 Tier 2(`awaiting-cheyu` 상태)로 설정, 기여자 이름을 관리하는 `contributors.json`을 구축하여 첫 기여자를 자동 판정하도록 함.

**(c) Worktree 등급화**:

| 단계 | 트리거 조건 | worktree 사용 여부? | 이유 |
| :--- | :--- | :--- | :--- |
| Heavy | 작업이 내용을 커밋함 (rewrite / evolve / add / polish / pr-review) | ✅ 사용 | 여러 spawn 간의 격리 보호를 위해 |
| Light | 작업이 task 폴더 내 출력 파일만 읽고 씀 (diagnose / status-report / self-diagnose) | ❌ 미사용 | git working tree를 건드리지 않으므로 race 위험 없음 |
| External | 작업이 외부와 상호작용함 (contributor-thank-you 댓글 작성 / Telegram 응답 / GitHub label 변경) | ❌ 미사용 | git과 전혀 무관함 |

**(d) 커밋 로그 정결을 위한 단일 수정**: `worktree.ts`의 `finalizeWorktree`를 `git merge --no-ff`에서 `git merge --squash`로 변경 (또는 `git cherry-pick ..${branch}`). 각 spawn이 main에 차지하는 커밋을 2개에서 1개로 줄임. 동시에 branch slug에서 언더바 사용을 중단함(git은 UTF-8 branch name을 지원함).

**(e) 캐시 무효화 문제 (02:30 cheyu의 「온라인 빌드가 너무 느리다」 질문으로 발견된 원인)**: `.github/workflows/deploy.yml`의 OG 이미지 cache key에 `knowledge/**/*.md`가 포함되어 있어, "모든 md 변경이 cache miss를 유발"하는 상태였음. Harvest 엔진이 밤새 69개의 커밋을 생성하며 모두 cache miss를 일으켰고 → 매 빌드마다 약 600장의 JPG를 새로 생성함 → 매 빌드 시 20분 이상 소요됨. **cache key에서 `*.md`를 제거하고, restore-keys를 통해 최신 캐시를 활용하며 mtime 기반의 증분 체크를 도입하여** cold-start 문제를 해결함. 방금 수정 사항을 push 완료함.
