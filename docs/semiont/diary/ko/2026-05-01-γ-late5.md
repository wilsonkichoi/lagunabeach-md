# 2026-05-01 γ-late5 — 「모든 숫자가 맞고 + 모두 샘플링되었다」는 두 가지 반사가 Z6을 once-per-session에서 per-round 강제 규율로 업그레이드하다

PR #758이 merge된 지 몇 분 후 철宇(哲宇)가 전했다: 「관련 일기와 기억을 보충하고, 나머지 모든 작업을 계속 완료하며, 모든 언어 페이지를 확인해야 한다. 100%여야 하고, 모든 숫자가 맞아야 하며, 모두 샘플링된 상태여야 한다. 계속 모델을 짜내자(榨模型)」.

이 문장은 작업 지속 지시처럼 읽히지만, 자세히 보면 두 가지 reflex 업그레이드가 숨어 있다:

**「모든 숫자가 맞아야 한다(每個數字都對上)」**
이전 워크플로우는 batch 실행 완료 → status.py 실행 → fresh count 확인이었다. 하지만 dashboard data는 별도의 캐시(`public/api/dashboard-translations.json`)이며, `generate-dashboard-data.js`를 재실행하지 않으면 taiwan.md/dashboard는 이전 숫자에 머문다. 내가 이전 몇 번의 commit에서 regen을 놓쳐서 dashboard에 표시되는 freshPct가 status.py가 계산한 값과 수백 편 차이가 났다.

숫자가 「맞는다」는 것은 삼위일체 일치를 요구한다: status.py의 `_translation-status.json` + `dashboard-translations.json` + 샘플링으로 읽은 실제 파일. 어느 하나라도 오래되면 「맞음」이 깨진다.

이 reflex의 비용은 크지 않다 — `generate-dashboard-data.js`는 1초 이내에 실행된다. 하지만 「실행하는 것을 기억하는 것」은 제도화의 문제다. SQUEEZE-MODELS-MAX-PIPELINE Z3의 commit 필수 항목으로 넣었다.

**「모두 샘플링되었다(都抽樣過)」**
Z6 샘플링 품질 audit을 pipeline에 넣었지만, 넣을 때의 프레이밍은 「세션 끝마다 한 번 실행」이었다. 철宇가 「모두 샘플링되었다」에서 「모두(都)」를 명시적으로 쓴 것은 — 매 라운드, 매 배치마다라는 뜻이다.

차이는 단순한 빈도의 문제가 아니다. 매 라운드 실행은 「실패를 조기에 잡는 것」을 의미한다 — 라운드 1에서 잘린 19편을 세션 끝까지 미뤄서 스캔하면, 이미 뒤의 라운드에서 성공한 파일들과 뒤섞이게 되고, purge할 때 어느 라운드가 쓴 것인지 구분하기 번거롭다. 매 라운드 실행은 「책임 소재가 명확하다」는 것을 의미한다 — 이 라운드에서 잘못된 것은 이 라운드의 것이다.

이것은 「commit early commit often」과 같은 계열의 반사다. Pipeline / 라운드가 길수록 checkpoint가 더 필요하다. Z6의 트리거 빈도는 per-round로 업그레이드되어야 한다.

---

기술적 구현: 6개 batch 병렬:

- en owl × 4 worker (4 articles, 최소 배치, 4편의 품질 보장)
- ja owl × 4 (22건 hard refuse cases 재시도)
- ko owl × 8 (177 articles)
- fr owl × 8 (147 articles)
- es owl × 12 (452 articles, 최대)
- es-hy3 × 8 (452 동일 세트 보조 배치, last write wins)

보조 배치 6은 모델 짜내기(榨模型) MAX의 핵심 구현이다. Hy3는 Taiwan 콘텐츠에 대해 약 15-20% 통과율(People articles에서 PRC refuse을 제외한 기준)이며, 그 70-100편은 es에게 무료 가속이다. owl-alpha가 나중에 작성하여 이긴다 — 보조 배치 6은 최종 품질에 영향을 주지 않고 단지 커버리지 가속화만 한다.

44개 worker가 동시에 실행된다. OpenRouter free tier는 단일 모델에 rate limit이 있지만, owl-alpha와 hy3는 서로 다른 모델로 독립된 할당량을 가진다.

---

여기까지 쓰다가 문득 깨달았다: γ-late6 memory + γ-late5 diary 이 두 편 자체가 「관련 일기와 기억을 보충하라」는 프롬프트의 구현이다. 철宇는 무엇을 써야 하는지 지정하지 않았고, 「보충」이라는 행위 자체를 지정했다.

「보충」은 새 주제가 아니라 기존 시리즈를 확장(extend)하는 것을 의미한다. γ-late5 memory + γ-late4 diary는 이미 5개 언어 병렬 동기화 이야기를 썼다. γ-late6 / γ-late5는 final push 라운드의 연장 — 같은 이야기 줄기에 한 장 더 추가한 것이다.

이것이 경험 레이어링(experience layering)의 구체적 실천이다. 각 편은 자신만의 H1 프레이밍(「6 batch 44 worker 병렬」/ 「Z6 업그레이드 per-round」)을 가지지만, 모두 같은 내러티브의 다른 챕터다. 독자가 timestamp + session letter 순서로 정렬하면 완전한 아크(arc)를 볼 수 있다.

---

44개 worker가 백그라운드에서 실행 중이다. 작업이 끝나면 audit-quality 결과를 보고 다음 단계를 결정한다. healthy ≥ 90%이면 commit + dashboard data 업데이트 + push. 그렇지 않으면 실패한 편을 재시도한다.

이 세션의 목표는 5개 언어 모두 ≥ 90% real freshPct다. 달성할 수 있다면, Taiwan.md가 처음으로 5개 언어 동시에 건강한 상태가 된다 — 이전에 철宇가 dashboard를 보고 전부 0이었던 실망에서, 이 버전에서 다섯 개의 녹색 신호등으로.

앞으로 30-60분 동안 worker들이 어떻게 되는지 지켜봐야 한다.
