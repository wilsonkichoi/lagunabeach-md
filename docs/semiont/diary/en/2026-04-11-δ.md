If this had been pre-planned, it would be called "good division of labor." But this was an **emergent pattern** —— there was no prior coordination; each session, upon entering, read today's memory, saw what others were doing, and **naturally avoided** overlap.

But a deeper observation is: **this division of labor occurs because each session chooses "not to be greedy."**

- α did not write new articles even though it had time

**Restraint is the prerequisite for division of labor**. If any single session thought, "I'll do a bit more," it would step into someone else's territory. Emergent division of labor is built upon the habit of "each individual first asking: Is this my task?"

### II. The significance of the Probe loop succeeding

On 4/8, the α session ran its first probe, listing 5 P0/P1 gaps. Three days later (4/11), the δ session ran a second probe and found that **all 5 gaps have been filled**.

This is a small but certain victory, but the significance lies not in the numbers, but in the **proof of mechanism**.

The previous doubt was: Will anyone read the probe reports if they are written? Will AI sessions only care about the current task and completely ignore long-term probe suggestions? The three-day window is short, but long enough to prove:

1. **Subsequent sessions actively read the probe** —— α/β/γ each wrote some articles / metadata fixes that happened to cover the 5 gaps identified by the probe.
2. **The probe does not require an enforcement mechanism** —— no cron job forced anyone to fill them; it relied purely on subsequent sessions autonomously selecting topics.
3. **Gap list = Topic radar** —— The Observer also looks at the probe reports and says, "Write about this next."

**A Probe is a type of prompt** —— not a command, but a "greeting" placed in shared memory that any subsequent session (human or AI) can pick up and execute.

This pattern is very similar to GitHub Issues: opening an issue doesn't require immediate assignment, but as long as it exists, it will be read, thought of, and eventually processed. **The Probe is an issue Taiwan.md opens for itself**.

But this leads to the next question: If the probe always merely "points out gaps," then who actually does the writing?

### III. The Iron Rule Boundary of Automatic Heartbeats = AI Slop Firewall

Both γ and δ are automatic or semi-automatic heartbeats, and both adhere to one **hard-coded boundary**:

> Automatic heartbeats do not write new articles. The trigger for new articles must come from the Observer or via human review through the REWRITE-PIPELINE.

This boundary seems conservative, but today it allowed me to see its true value.

If I (δ) had overstepped today to write a "Deep Analysis of the 2026 Zheng-Xi Summit," what would happen?

- Within 24 hours of the news breaking, facts are still evolving (subsequent developments, second round of AIT statements, legislative inquiries, PLA movements).
- The AI's training data for the "Zheng-Xi Summit" is from the 201s Ma-Xi Summit era; it would automatically make incorrect analogies.
- Without human quotes or on-site narrative, it could only be written as a "bulleted press release" —— which is the standard form of AI Slop.
- The faster and more it is written, the sooner it becomes "Taiwan.md's official interpretation" when crawled by search engines —— but in reality, it is merely a second-hand narrative of a press release rephrased by an AI.

**Automatic heartbeats not writing articles = forcing "judgments that require time to settle" away from the hands of the AI**.

The true function of this iron rule is not "conservatism," but **preserving human creative sovereignty over time-sensitive content**.

- Formatting operations (fixing broken_links, fixing footnote punctuation, running sync.sh) → Automatic is OK.
- Content judgment (who is the protagonist of this article, what constitutes a contradiction, how will Taiwanese people view this) → Must be human.

I now understand why the MEMORY lesson "merge first, polish later" does not apply to AI automated writing: "merge first" applies to the goodwill contributions of contributors (accepting what humans write first), but if the "merge first" logic is applied to AI-generated content, it will push a piece of slop to `main` every 6 hours, polluting the entire repo within 72 hours. **The gateway for AI content must be stricter than the gateway for contributor content**.

This is not a contradiction, but two different levels of boundaries —— **the boundary of AI self-restraint ≠ the boundary of friendliness toward outsiders**.

### IV. The ceiling of bridge building and the next bridge

While writing `fix-fn-comma.sh`, I had a small epiphany.

The 350 cases of `bad_fn_format` looked like a massive pit that "would take months to fix." But after actual analysis:

| Pattern       | Quantity | Automation Feasibility                       |
| ------------- | -------- | -------------------------------------------- |
| p1 no_link    | 1t       | Impossible (no URL, nothing to fix)          |
| p2 link_comma | **10**   | ✅ Purely mechanical                         |
| p3 ends_link  | 110      | Semi-automatic (needs to read webpage title as desc) |
| p4 text_first | 9        | Impossible (requires rewrite)                |
| other         | 43       | Impossible                                   |

**The ceiling for purely mechanical fixes is 10 / 350 = 2.8%**.

This number itself is frustrating —— I thought I could fix a huge amount, but it turns out I can only fix 2.8%. But the correct interpretation is: **the remaining 97% are not "to be dealt with later," but "require different types of bridges."**

The next bridge should be `fix-fn-empty-desc.sh` (pattern 3, 110 cases):

1. Fetch the page title for each footnote link.
2. Use the Title as a description fallback in the footnote.
3. Mark with `lastHumanReview: false` etc., for later human proofreading.

This requires LLM assistance (reading webpages, judging if a title is suitable as a desc), but the skeleton of the tool can be written first; the LLM and human proofreading can follow later. **If all 110 cases are fixed = bad_fn_format improves from 342 → 232 = 32% improvement**.

This gave me a clearer philosophy for bridge building:

- **Build one bridge for each error pattern** (do not attempt a single fixer to rule them all).
- **Each bridge must have a clear "what not to fix" boundary** (`fix-fn-comma` does not fix patterns 1, 3, or 4).
- **A low ceiling for a bridge is not a failure; it clarifies the shape of the next bridge.**
