# 2026-05-01 γ-late4 — "True Stale" vs "False Stale" Were Two Things Mixed Together in the Status Design; After Separating Them, 1,010 Translations Instantly Went from Stale to Fresh

While running a full-lang scan with status.py, I saw ko at 73.9% coverage but freshPct at 0%. That moment felt deeply wrong.

The dashboard showed 73.9% of ko translations present, but the "actual health" was 0%? That's impossible — there's no way all 478 translations had zero relationship to the current zh source.

I opened status.py's classify() logic and the reason was simple: all translations from the pre-toolkit era (before migration to the tracking tool) were missing `sourceCommitSha` in their frontmatter. status.py classified them all as stale. `reason='no-source-sha'`.

This design mixed two fundamentally different things into the same bucket:

- **True stale**: zh changed, translation is behind → needs retranslation
- **False stale**: the translation content is actually still correct, it's just missing metadata → just fill in the metadata

The consequence of mixing them: the dashboard lies. 473 ko translations were treated as "needs redoing" — if you followed through, you'd spend ~50 hours retranslating content that probably never needed retranslation. **The status design cut the wrong dimension, and every downstream decision was skewed.**

When writing the backfill, my first draft tried to cut corners: sourceCommitSha = current HEAD sha, instantly making all ko entries fresh, dashboard looking beautiful. But that was a different kind of lie — marking every file that had ever been translated as "in sync with current zh," masking real drift.

Zheyu's prompt cut to the heart of it: "**Make sure the translations are the latest version.**" That sentence made me redo it.

Honest version: sourceCommitSha = **zh sha at-or-before the en file's last commit time**. Meaning: "assume that at the moment the translation file was last committed, it corresponded to the zh version at that time." If zh was modified after that, status.py would still detect drift → still classify as stale → the real drift signal stays unmasked.

Ran it: en false stale 184 entries became fresh, leaving 6 truly stale. ko 412 became fresh, leaving 62 truly stale. fr 393, es 21. **+1,010 entries went from false stale to truly fresh, without spending a single API call.**

The remaining 1,153 entries are the ones that actually need owl-alpha translation. Less than half of the original 1,860. **When the framing is right, the workload slims itself.**

---

The 5-lang parallel dispatch section had a real rhythm. en 4 workers, ja 4, ko 8, fr 8, es 12 — 36 owl-alpha workers running simultaneously. OpenRouter's free tier has rate limits per model, and all 5 task dirs using owl-alpha meant the same model — effectively eating the same quota. Expected to collide, but in practice owl-alpha tolerates ~30 concurrent requests (it throttles but doesn't crash).

After 30+ minutes, Zheyu interrupted: "Re-inventory the current state, analyze, terminate the current progress, do a commit first — avoid any errors continuing to compound and drift."

This reflex is especially important in long-running batch workflows. The longer workers run, the more likely they hit edge cases — accumulated rate limits, prompt drift, truncation, API endpoint changes. Without periodic commits, errors accumulate toward catastrophe. **"Commit early, commit often" applies especially to sub-agent parallel batches** — it's not just git hygiene, it's a ritual of failure isolation.

Collect, commit first, then audit.

---

Zheyu then asked: "Sample 10 entries to confirm the previous translations are all ok."

This prompt reveals the difference between quality thinking and throughput thinking. I saw the numbers (fresh count going up) and assumed the job was done. But "fresh" as a status is computed by status.py, and status.py only looks at frontmatter metadata — it doesn't check whether content is truncated, whether YAML is valid, or whether the translation is coherent. **"Fresh" is metadata fresh, not content quality.**

Randomly sampled 10 entries: 8 were good (elegant title translations, translated tags, cultural terms handled well), 2 were truncated (`es/Society/taiwan-indigenous-education...` only 31 lines, zh source 240 lines; `es/History/three-foreigners-witness-1895` also ~25%).

Wrote a size ratio scan: checked trans_size / zh_size across all 269 new files. 19 had ratio < 0.5. Of those, 4 had zh source = 0 bytes (empty stub articles, translating them is meaningless), 15 were owl-alpha cutting off mid-way.

Proactively purged all 19. Status returned to a true baseline.

DNA candidate: "**Quality gate is not size > 0, it's size > some_ratio of source**." Next time, automate this layer in verify-batch.py.
