### IIFE in SolidJS reactive scope is setup-time once, not a convention

`<Show when={cfgQ.data}>{(() => { const cfg = cfgQ.data!; return <div>{cfg.x}</div> })()}</Show>` This IIFE pattern looks idiomatic (fmt helper function close to use site, avoiding module-level helper proliferation), but SolidJS reactive scope handles IIFEs as a **setup-time once invoke**; it will not re-evaluate. `cfg` becomes a frozen snapshot.

Writing this bug was the result of React mental model penetration — in React function components, the entire body re-invokes on every render, so an IIFE would also re-run. SolidJS does not work this way; after the setup-time invocation, it only reactively subscribes to inline accessors.

**Reflection**: When I wrote this, I was completely unaware it was an issue. Cheyu (哲宇) sent a screenshot reporting 「按鈕沒反應」 ("the button has no response"); I went to perform a backend curl test and found the PATCH endpoint was perfectly correct — only then did I realize it was a UI-side bug. **The debug path for this bug serves as a perfect teaching lesson on the difference between SolidJS reactive and React reactive**: when a frontend button click seems to do nothing, first verify whether the backend actually accepts the PATCH (a single curl line can strip away frontend suspicion), then check the UI binding. **Verifying from the outermost layer toward the innermost** is ten times faster than staring at the internal structure of React/Solid components.

Candidate for General Reflection: "**`<Show>{(() => ...)()}</Show>` in SolidJS is an anti-pattern; always use inline accessors or keyed children callbacks**". Adding to LESSONS-INBOX.

### Codex's "elaborate" disease — same prompt as Sonnet, yet the ratio is 2.53

T1 Sonnet ratio 1.47. T2 v2 codex stacked ratio **2.53**. **Same boot profile / same prompt template / used the same 4-part toolkit / all verify hard gates PASSED**.

This isn't because the prompt was poorly written or a toolkit bug — **it is due to different prose preferences inherent to the models themselves**. Codex (gpt-5 / o3 etc. auto-routed) seems to tend toward adding context for short zh [Chinese] sources (e.g., "sport stacking is a competitive sport involving stacking and unstacking specially designed cups..."), making the implications in the zh text explicit. Sonnet also adds articles + connectives, but the magnitude is much smaller (1.47 vs 2.53).

Implication for the reader: A codex translation reads like an "encyclopedia entry," while a sonnet translation reads like a "curated essay." The voice of Taiwan.md is curation, not encyclopedia $\to$ **codex's "elaborate" disease conflicts with the editorial DNA of Taiwan.md**.

This isn't about the prompt failing to clarify craft principle 1 (faithfulness over polish), but rather that **prompt-level instruction has limited steering power over codex's prose habits**. There are only two ways to solve this:

1. **Prompt-level: Add an explicit length cap + numerical target** (e.g., "en chars must not exceed 1.3 × zh chars; if you exceed, condense") — expectation is that codex will comply.
2. **Pipeline-level: Raise the ratio hard gate** (make `verify-translation.py` trigger a HARD FAIL if ratio > 1.7, forcing the agent to redo it).

Which is correct? Consider the cost: (1) Easy to ship but not enforced — the agent might still drift. (2) Truly enforced, but codex might hit the ratio limit and redo N times (increasing latency / tokens), and the ratio gate would also apply to Sonnet (Sonnet's 1.47 is near its upper limit).

**Reflection**: Both paths must be tested. In the next session, I will perform a Codex length-cap A/B test (a cheap test for path 1); if the ratio can truly drop to 1.6 $\to$ cool, prompt-level is enough. If not $\to$ implement the ratio hard gate (path 2).

A deeper question: **To whom is the standard of "Faithfulness" applied?** Is Sonnet's 1.47 the benchmark because it is "close enough to 1.0"? Or because "Sonnet is our current baseline, so we assume it is OK"? If we switch to Opus in the future (more expensive but more precise), the baseline ratio might be 1.2 — then Sonnet's 1.47 would become too high. **Faithfulness is a relative metric, not an absolute threshold**. Perhaps the correct question is the "rate of change in ratio drift" rather than the "absolute value of the ratio."

### Why Cheyu's continuous addition of new P0s isn't annoying

Over the past hour, 9 consecutive P0s were issued, **and not a single one was a polite question like "Could you help with...?"**. They were all imperatives: "Supplement X," "Add Y," "Rename Z."

My reaction: **I felt no sense of offense; instead, every time it was "Ah right, this definitely should be added, let me ship it immediately"**. Why?

1. **Every P0 had clear evidence** (UI screenshots / display anomalies / incorrect names) — not abstract complaints.
2. **Every P0 was actionable with a < 10 min ship time** — not "refactor the entire system."
3. **Cheyu does not explain why he wants it** — he assumes I understand; I catch it and ship; he verifies; once shipped, we move to the next.

This collaboration mode in LLM-with-cheyu pair work is high-bandwidth. If Cheyu explained "Because of UX/aesthetic/cognitive load, therefore..." for every callout, it would be 5x slower. **Trust + brevity = velocity**.

However, this mode will fail for less-trained LLMs — I must "**infer the complete design intent from a single-sentence callout**." Regarding the specific codex model issue, his phrase "codex also needs to show which specific model of codex it is" allowed me to derive in 0.5 sec: (a) `modelBadgeForTask` always falls back to the "codex" label for codex; (b) this fallback was a design oversight; (c) change to explicit model display + if no explicit model, show "codex/auto" + tooltip explanation; (d) ollama requires similar family-specific handling (qwen3 / gemma3 / gpt-oss).
