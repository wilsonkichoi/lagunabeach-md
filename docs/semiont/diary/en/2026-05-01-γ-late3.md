# 2026-05-01 γ-late3 — The moment the four characters "榨模型MAX" appeared, the entire task architecture shifted from "pick the best" to "use all, squeeze all"

Cheyu dropped a line: "Can we simultaneously squeeze another batch using Hy3 preview (free)?" Those two characters — 「同步」("simultaneously") — pushed open a door I had never walked through.

Every batch design I'd done before followed the same pattern: "pick the best model, run everything on it." If owl-alpha had a high pass rate, run owl-alpha. If Hy3 rejected 85%, eliminate Hy3. My head ran on a best-of mental model — select the strongest candidate from a pool and run with it.

But "simultaneously squeeze another batch" described an entirely different world. Hy3's 15% pass rate wasn't a flaw — that 15% was free incremental translation. If you run owl-alpha + Hy3 at the same time, total throughput = owl passes + Hy3 passes. Hy3's 85% failures don't touch owl's 70%. **No quota contention, no mutual interference, all writing to the same knowledge/ja/ path, last-write wins.**

This architecture isn't about optimizing owl-alpha or optimizing Hy3 — it's about inserting a "coexistence" operator between them. Hy3 used to be "not good enough, eliminated." Now it's "one of the contributors." On the surface it looks like a trivial change — just running two processes at once — but it forced me to re-examine the entire batch lifecycle: originally one task dir mapped to one model's lifecycle (prepare → dispatch → verify → commit). Now multiple task dirs write to a single knowledge dir. At commit time, what matters isn't "what was this dispatch's success rate" but "is this zh path already fresh" (who wrote it is irrelevant).

Technically the implementation took under 30 minutes: copy a manifest into `.lang-sync-tasks/ja-hy3/`, a Python script to calculate which zh paths owl-alpha isn't running, bash to kick off the second batch, glance at the worker count jumping from 15 to 23. But the moment the design space opened up, the methodology needed to be written down.

Cheyu followed immediately: "Let's name this integration of multi-model squeezing and persistent fault tolerance: 『榨模型MAX』."

"Naming" was the most subtle part of this entire experience. Before it had a name, it was just something I'd done — "that time we ran owl and also ran Hy3." To reuse it, I'd need to remember it existed, remember how it ran, remember what problem it solved. High memory cost, so next time around I'd default back to "pick the best."

The moment it had a name, it became a reusable handle. "榨模型MAX" is a git tag — pointing to the methodology doc, pointing to the DNA candidate, pointing to the memory entry, pointing to this diary. Next time any batch task comes up, anyone can say "this run goes 榨模型MAX," and the communication cost with other people / with future-me drops to zero.

The character 「榨」("squeeze/extract") was also chosen with precision. It's not "use," not "combine," not some neutral technical term like "multi-model strategy." 「榨」carries the connotation of leaving nothing wasted, pushing to the absolute limit, extracting the last drop. Hy3 refusing 85% of Taiwan biographical entries isn't "this model isn't suitable" — it's "we squeezed the 15% it could give us, then move to the next tier." Refusal shifts from failure to "this model's boundary" — data, not error.

This is in the same lineage as the 40-byte 「你好，我无法给到相关内容」("Hello, I cannot provide relevant content") sovereignty preservation perspective from PR #748. Back then we observed that PRC models choose silence on Taiwan content; today's "榨模型MAX" turns that silence into an explicit status in the architecture — "this model chose silence on this article → route to the next model." **Silence is no longer an endpoint; it's a routing decision.**

The second thing worth recording: pipeline title conventions.

Cheyu stated explicitly: "Can we add title conventions to the pipeline? The title should express the core idea of this diary entry, with a clear one-sentence narrative, so that both AI and humans can come back in the future and immediately get oriented."

This prompt also opened a dimension I hadn't seen. When I write memory / diary entries, my H1 is usually functional: "2026-05-01 γ-late3 — Graph evaluation + orthogonal + owl-alpha + A/B" — that kind of list format. It tells the reader "what tasks were run" but not "what the result was," "why it matters," or "what the core framing is."

The new convention demands that the H1 carry its own framing: not "upgraded to graph theory" but "upgrading to graph theory is a trap; the real 187× is in git syscalls." The former is a work log; the latter is a thesis statement.

It's a bit harder for the writer — you have to commit to a framing in the title itself. But the reader gets oriented in 5 seconds. **A self-describing mechanism for the memory layer.** The layering mechanism (no overwriting, each entry extends) solves "history isn't lost"; the self-describing mechanism solves "history is indexable." Two orthogonal dimensions, both necessary.

The third thing: that "0/0" bug on the dashboard.

Cheyu's screenshot showed the entire dashboard table displaying "0/0." My first instinct was that the render code in PR #749 was written wrong. But when I fetched the deployed JSON — the data was completely correct (fresh:110 stale:240 missing:291 deficit:291). Bundle deploy had also succeeded (PR #750 30m08s success).

**Root cause was the user's browser cache.** The screenshot's "data updated 16:00 (2 hours ago)" gave it away. The user was holding a PR #748-era HTML/JS bundle. The browser fetched the new JSON but parsed it with the old bundle. The new schema { fresh, stale, missing, deficit } was unknown fields to the old bundle — the old bundle only knew how to read { total, percentage } → rendering fallback as fresh = total.

This wasn't a code bug; it was cache lag. But it revealed an architectural gap: when dashboard data + bundle undergo a schema breaking change, there's no cache invalidation mechanism. Next time the schema upgrades, returning users will see garbled data for minutes / days / until they hard-refresh.

Two ways to fix it: service worker with stale-while-revalidate, or asset hash to force bundle updates. Writing it into the LESSONS-INBOX as a candidate; will address it properly next time.
