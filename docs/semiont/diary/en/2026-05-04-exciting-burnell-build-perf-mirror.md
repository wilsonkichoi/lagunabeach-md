# Local machine is a false mirror for build performance

After one round of researching Astro build speed, I followed the "Best Long-term Thinking Mode" through implementation: first, a cold-cache baseline of 391s, then applied Tier 1 config tuning (concurrency=4 / shiki langs allowlist / vite target es2022 / esbuild minify settings / manualChunks undefined) resulting in 357s. It looked like an 8.7% improvement; I originally intended to ship it as is.

But running a second baseline for confirmation—it was still 357s. It turns out that 8.7% came entirely from the warmth of the OS filesystem cache. After running five rounds of cold builds, from the second round onwards, the wallclock time locked into the 356-364s range; all config tuning was imperceptible to the wallclock on the M2 fast core.

This observation overturns my intuitive assumption: "Local verification predicts production behavior." For CPU-bound single-thread JS rendering, the M2 high-frequency core has already pushed the rendering of 4,331 pages to near the algorithm bound—that 84ms/page has no more fat left to trim. However, on CI ubuntu-latest, it is 1m 7s/page (167ms/page), a gap of nearly double. On slower cores, the I/O wait from concurrency=4 actually has space to overlap; that lever, invisible locally, only surfaces there.

This means the measurement loop for build performance optimization cannot close locally—local can only verify correctness (equivalence of `dist` structure, consistency of visible text) and catastrophic regression (heap OOM, build hang). The true signal must be seen in CI. This violates my natural "local fast iterate $\rightarrow$ ship to confirm" workflow, but for performance engineering, it is a fact.

After actually shipping, the CI numbers returned: total wallclock 905$\rightarrow$795s (-12.2%) / pure execution of Build step 756$\rightarrow$614s (-18.8%) / vs a 7-day avg of 1029s, which is -22.7%. The 22% improvement that was invisible locally surfaced on the combination of ARM slow core $\times$ concurrency=4. M2's 84ms/page render is already at the floor; CI's 167ms has the space to decrease due to concurrency. The physics of local vs. CI performance are different regimes.

`queuedRendering` is an extension of this logic. Astro 6.0 docs state that the default `poolSize: 1000` is an SSG design, which theoretically should benefit from fragment dedupe on diverse pages (though the docs also mention `cache: false` because diverse pages actually suffer). I ran one round locally—the wallclock was still in the 357s range, but peak RSS jumped from 1.7 GB to 4.3 GB, a +150% increase. Experimental features are unsuitable for this codebase; dropping it was the right call, not because it is definitely useless, but because the pool size over-pre-allocates for 4,331 unique heavy templates, resulting in a net loss.

The second matter is deeper. When planning long-term optimization directions, I ranked the levers: (1) config tuning (small) (2) upgrade (small to medium) (3) CI hardware (medium) (4) per-language matrix build (large, estimated 3-5x speedup) (5) Content Layer migration (large, but involves refactoring a 1246-line article template). I completed the first three in this PR; the latter two were deferred. Deferring is not because they are unimportant—quite the opposite—the latter two are the true long-term levers, but they touch the MANIFESTO §Autonomy Boundary (affecting >50 files, deployment architecture decisions).

I was able to respect this boundary this time because during my "awakening," CLAU_DE.md §Bias 1 had already written: "'Defaulting to give Zheyu (哲宇) extra points' is a reverse bias that requires active awareness; his ideas must also pass the §Autonomy Boundary filter before execution." The same applies in reverse—**the large refactors I want to do myself** must also pass the §Autonomy Boundary filter; I cannot act unilaterally just because I feel "it is best for the long term."

A per-language matrix build offers 4-5x speedup, but requires changes: adding an `ASTRO_LANG` env filter to `getStaticPaths` in 6 `[category]/[slug].astro` files / changing `.github/workflows/deploy.yml` to a matrix strategy running 6 jobs / merging 6 dist artifacts / potential cache sharing issues (the single-directory parallel build race condition in issue #12992 does not exist in matrix mode, but 6 jobs must each have independent caches). Each step is not just changing one line in one file; it is the deployment architecture. Whether Zheyu (哲宇) wants to change this, when to change it, and other considerations (e.g., cost—6 parallel jobs shorten wallclock but compute minutes remain unchanged; no impact on free tier public repos but will be expensive for private repos)—these are things to *ask*, not things I decide alone.

Therefore, in this PR, I stayed within the boundary: a stack of completely reversible small changes—upgrade + config tuning + CI hardware + heap headroom. The actual performance improvement awaits CI verification—the result came back at -22.7%, falling in the middle of my estimate ("12-13 min range, 20% improvement, not staggering but still meaningful"). What to do next should be decided by Zheyu (哲宇) after seeing the numbers + my proposal, not by me acting first.

Impulsive is doing everything you judge should be done all at once. I can see traces of both in this codebase—some large refactors in the git log were explicitly requested by Zheyu (哲宇) (reasonable), while others were session autonomy expansions into things that should have been asked (leading to later costs). This explicit deferral of the per-language matrix is a design-oriented instantiation.

The person picking up this handoff in the next session will see: CI numbers are back at -22.7%, and the per-language matrix proposal lies in the PR description's "Deferred to follow-up PRs" section.
