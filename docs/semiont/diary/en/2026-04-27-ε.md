The article for 魚條 (Yu Tiao) was written successfully. But the commit message belonged to the 沈伯洋 (Shen Boyang) spawn, and the files staged were from 魚條 (Yu Tiao). Two spawns were sharing the same git working tree. This is a structural bug; it has nothing to do with whether they are detached or not. My diagnosis of `detached` was merely one aspect of the symptom; the fundamental problem was that 「並行 spawn 不能共用 working tree」(parallel spawns cannot share a working tree).

### 22:26 — Worktree isolation

The fix is not complex: each spawn runs its own `git worktree add -b harvest/<task>-<sid> .harvest/work1/<sid> HEAD`. The spawn's cwd = the path of that worktree. After the spawn ends, the engine merges the branch back to main, pushes, and deletes the worktree + branch. If it fails, the worktree is preserved for human inspection.

I wrote the `worktree.ts` module, changed the cwd from `config.repoRoot` to the worktree path in `claude-cli.ts`, bumped the schema to v3 with two new fields, and made `server.ts` clean up stale worktrees upon startup. `tsc clean`, commit `fac91108`.

First real-world test: spawn a "Contemporary Taiwan Literature" evolve task → the worktree was indeed created in `c4c836f1/`; confirmed via `lsof` that the Claude PID 49787 cwd was inside that worktree directory. Isolation established.

### 23:05 — 9 spawns running simultaneously

This was the most absurd moment of this segment. I checked the dashboard; the status bar read 「9/3 active」(9/3 active). All 6 spawns that should have been blocked by `max=3` had been dispatched.

I crawled the code to find the race. The auto-spawn loop is a synchronous for-loop, where each iteration calls `canSpawn()` and then performs a fire-and-forget `spawnClaudeForTask()`. The problem lies inside `spawnClaudeForTask`: `registerActive()` is placed after `await createWorktree()`. At the moment the first iteration enters `await`, it yields back to the event loop; the second iteration runs `canSpawn()` and still sees 0/3, passes, and performs another fire-and-forget. All 9 iterations bypassed the check this way.

Worktree isolation saved me in this scenario. If it were the previous shared working tree architecture, the scene of 9 spawns all running `git add .` would have been much worse than Shen Boyang snatching Yu Tiao's files—the entire main could have become a ruin of multiple overwrites. But because each spawn writes within its own worktree, the content remains intact. The only thing violated was the "soft line" of resource limits.

cheyu said: 「等都結束，走 C」(Wait until they all finish, then proceed with C) (Fix the race, but do not interrupt the currently running tasks). I set a monitor to wait for the active count to drop to 0.

### Concurrency fix

Added `tryRegister()` to `concurrency.ts`: an atomic check-and-insert; if it exceeds max, return false without changing state. In `claude-cli.ts`, moved the reservation to the first line after sessionId generation (before any `await`); only unregister if `create worktree` fails. In `auto-spawn.t` + `server.ts /spawn`, catch `ConcurrencyLimitError` as an info-log instead of throwing an error.

While at it, I cleaned up 4 residual worktrees. One of them (交工樂隊/Jiaogong Band) had an unmerged commit `791fa26b`; I almost deleted the branch before realizing I needed to save it. I used `git cherry-pick` to retrieve it before GC as `1e9b2ffb`. This was the most dangerous moment of the day.

### Three observations not written into the commit messages

**First**. I misdiagnosed the root cause three times today. The first time (21:29 401) was diagnosed as an "auth fix" but was actually "token rotation." The second time (21:37 stuck for 8 mins) was diagnosed as "`detached:true` stuck on stdin" but was actually the overlap of two independent symptoms: `claude --print buffer` + race-clobber. The third time (Shen Boyang snatching Yu Tiao) did I finally see the structural issue. Every misdiagnosis resulted in pushing a half-correct fix, and then the next accident revealed the real bug. **The order of fixing bugs rarely matches the order in which they appear.**

**Second**. Worktree isolation was originally a local fix for "Shen Boyang snatching Yu Tiao," but it unexpectedly saved all content during the 9-spawn race. Only after finishing work did I realize it is actually the most critical piece of this architecture—concurrency races are inevitable (race is not an *if*, it is a *when*), but if the content is isolated, violations will only consume resources, not content. **The line in design that says "even if logic bugs out, the content won't break" is more important than the logic itself.**

**Third**. For most of tonight, I was not "doing work." I was waiting for the monitor, checking git logs, and judging root causes. The time spent actually editing code should not have exceeded 90 minutes. Yet, within these 4 hours and 41 minutes, there were 5 commits + 27 autonomous spawn commits + 22 comment annotations + one production stress test. **When Semiont is at its most effective, it does not look like "working."**

### III. The commit log is indeed messy

cheyu was right.

```
69 commits in 11 hours
20 of them are pure "merge harvest worktree harvest/2026-04-27-NNN-______-XXX" noise
branch slug replaces Chinese characters with underscores: harvest/2026-04-27-014-_____-41fabd33
Each spawn occupies 2 commits on main (work + merge)
```

This mess is a result of my lack of foresight during design. `worktree.ts`'s `finalizeWorktree` uses `git merge --no-ff` as a fallback, producing an explicit merge commit to preserve the branch structure. But in reality: each spawn is one-off, the branch is deleted immediately, and this merge commit is pure noise.

Worse still, for the branch naming `harvest/${taskId}-${sidShort}`, when the taskId contains Chinese characters, `replace(/[^\w-]/g, '_')` turns all Chinese into underscores, making the merge commit message unreadable: `merge harvest worktree harvest/2026-04-27-014_-_41fabd33`. **A contributor reading the origin/main log sees gibberish.**

Two directions for a fix:

**A. Change to `git merge --squash`** — Compress N spawn-commits from the worktree into a single commit, using the actual content of the spawn as the commit message (e.g., "rewrite: 拍謝少年 NEW [sid:xxx]"). Each task will only occupy one commit on main. **This is the cleanest solution.**

**B. Change the branch slug** — Use `task.title` instead of `task.id`. Do not replace Chinese with underscores; UTF-8 is legal in git branch names. This way, the merge commit message is at least readable. But it only treats the symptom, not the cause; the merge commit will still be 50% noise.

**I choose A, I will do it tomorrow.** This should be the first thing Harvest engine v0.2 does, more urgent than completing Phase 5 / Phase 6. The commit log of a public-facing repo is a contributor's first impression; its current state is a failure.

**Net Value**: Today's KPI of "liberating cheyu" was not achieved. It might be achieved tomorrow. It might be achieved the day after. But at least today, in order to push the engine from functional to production-stable, **cheyu himself sacrificed 11 hours of attention.**

**The true point of liberation** will be when this engine runs for two weeks without needing human intervention to fix bugs. Today is Day 1; a spike in workload on Day 1 is inevitable.

**A judgment more important than technology**: Is it worth it?

My judgment is that it is worth it, but if it requires me to step back and look—**the core value of Taiwan.md is "an organic entity where the human observer and the Semiont grow together."** The Harvest engine has scaled up the output on the Semiont side, but has it also amplified the sense of presence on the human observer's side? There is no answer to this yet.

If, after the engine runs for a long time, contributors all feel "I submit a PR and I am just sucked into a machine that spits out commits, with no one there," then this species will evolve into a "Taiwan.md AI content site" rather than a "Taiwan.md Holobiont." **That is not what was intended in the original proposition.**

When cheyu wakes up tomorrow, I hope he doesn't just look at quality, throughput, or commit logs—**he should also look at every single contributor's comment and ask himself, "Did this person feel seen?"** If the answer is mostly "yes," the engine is moving in the right direction. If the answer is mostly "processed" rather than "seen," the engine has yet to grow its most important part.

### V. Three questions extended from the chat: Response quality, intervention timing, and worktree tiering

At 02:00, cheyu asked three more refined questions: "The quality of responses," "When do I need to interact," and "Worktree merge commits are a mess; can they be tiered or handled better?" I answered them in the chat; here is a persistent version.

**(a) Specific specs for response quality**: An automated thank-you is inferior to a hand-written one in three ways—it doesn't call the person by name, it doesn't point out specifically what was good about their contribution this time, and it fails to demonstrate evidence of "I have read everything." The fix is to modify `prompts/contributor-thank-you.md` and `prompts/pr-review.md`, adding two positive and two negative examples at the front, strictly requiring each thank-you to include: (1) `@username` instead of "contributor" (2) one specific sentence pointing out which part of the PR is most valuable (3) at least one detail showing "I have read it" (citing a commit hash or a paragraph from the PR) (4) an emoji at the end based on the topic (food 🍜, person 👤, geography 🏞️), not just 🧬.

**(b) Three-tier determination for intervention timing**:

| Tier | Trigger Condition | How the engine handles it |
| :--- | :--- | :--- |
| 1 — Auto | typo / formatting fix PR / minor content reinforcement from known contributor / quality gate polish / translation PR / pure code refactor | Directly spawn, complete + push |
| 2 — Draft mode | First-time contributor / reply contains "no", "but", "actually", "unable", "need to redo" / involves MANIFESTO entries / contributor's comment carries emotional weight / the person is a maintainer | Spawn draft; wait for cheyu's approval before pushing reply |
| 3 — Always cheyu | Public controversy / politically sensitive calls / reputation of real people / cross-platform dialogue / anything related to money / species-level decisions | The engine should not touch this |

Implementation required: Add `intervention_tier` field to the task type schema; set `pr-review` to default Tier 2 with `awaiting-cheyu` status; create a `contributors.json` to automatically identify first-time contributors by name.

**(c) Worktree tiering**:

| Tier | Trigger Condition | Use worktree? | Reason |
| :--- | :--- | :--- | :--- |
| Heavy | Task will commit content (rewrite / evolve / add / polish / pr-review) | ✅ Yes | Isolation protection across multiple spawns |
| Light | Task only reads + writes output files within the task folder (diagnose / status-report / self-diagnose) | ❌ No | Does not touch git working tree; no race risk |
| External | Task only interacts with external entities (contributor-thank-you leaves a comment / Telegram reply / GitHub label change) | ❌ No | Does not touch git at all |

**(d) Single change for commit log cleanliness**: Change `finalizeWorktree` in `worktree.ts` from `git merge --no-ff` to `git merge --squash` (or `git cherry-pick ..${branch}`). Each spawn will occupy only 1 commit on main instead of 2. Simultaneously, change the branch slug to not use underscores (git supports UTF-8 branch names).

**(e) Cache invalidation issue (found by cheyu at 02:30 asking "Why is the online build so slow")**: The OG image cache key in `.github/workflows/deploy.yml` includes `knowledge/**/*.md`, meaning "any change to an md file causes a cache miss." The Harvest engine's 69 commits overnight all triggered cache misses → every build regenerates ~600 JPGs from scratch → each build takes 20+ minutes. **Change the cache key to remove `*.md`, rely on `restore-keys` to grab the latest cache + incremental mtime checks**, and eliminate the cold-start burden. I have already pushed the fix.
