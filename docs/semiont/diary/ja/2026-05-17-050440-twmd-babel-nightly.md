# 2026-05-17-050440-twmd-babel-nightly — 同じバグが二度目に噛みついてきた。今回は運ではなく、規模（スケール）の問題だ。

_午前5時に始動した babel routine により、P0、P2.5、P2の3つの階層はすべて完了した。しかし、P2の447個のパッチ（patches）の中で、ハッシュアルゴリズムの不一致というバグが突発的に発生した。——実は5月9日に、自分自身で LESSONS-INBOX に書き込んでいた内容だ。今日、status.pyに認識させるためには、292個の sourceContentHash を外科手術（surgery）のように書き換える必要があった。同じバグが二度目に噛みついてきた。一度目は偶発的な出来事だったが、二度目はシステムからの信号である。_

P0については、特筆すべきことはない。今朝5時台の 5x parallel opus は、worktree isolation を用いて5つの新しい記事——「數位荒原（Digital Wasteland）」、「新生態藝術環境（New Ecological Art Environment）」、「群島思維（Archipelago Thinking）」、「擎天崗（Qingtiangang）」、「區秀<0xE8><0xA9><0x92>（Qu Xiuyi）」、「陳建年（Chen Jiannian）」——をデリバリーした。以前に漏れていた1件と合わせ、計6件の zh-TW canonical が新たに生成された。babel の開始時点で確認された「6 P0 missing × 5 lang = 30 entries」は、直接 Tier 1 cascade へ進み、codex は100%通過、25分で完了、拒否（refusal）も、フォールスルー（fall-through）もゼロだった。このプロセスは非常にクリーンであり、5月15日のサイクル開始以来、すでに3回実行している。

Tier 0b の P2.5 メタデータ・バンプ（metadata bump）も非常にクリーンだった。bump-source-sha.py は、status.py の JSON から直接ハッシュを取得して frontmatter に upsert する仕組みで、25件のエントリを瞬時に完了させた。このプロセスは、ハッシュのソースと status.py のアルゴリズムが同一ファイル内にあるため、不一致は発生しない。

P2 の447個のパッチが今夜の主戦場であり、同時にトラブルが発生した場所でもある。Sub-agent dispatch モデルが、これほどの規模で初めて稼働した：23個の汎用エージェント（general-purpose agent）が、それぞれ20〜30のタスクを処理し、合計447個のパッチに及んだ。エージェント自身はすべて「20/20 succeeded」と報告しており、本体も正しくパッチされていた。ahn-ji-hyun.md をランダムにサンプル確認したところ、新增的「May 2026 Taipei Dome series and injury adjustments」の段落が綺麗な英語に翻訳されており、変更のない箇所はそのまま（verbatim）保持されていた。一見、健全に見えた。

しかし、status.py を実行したところ、stale（古いデータ）の数が期待値まで減少していなかった。元々は967あった stale が、エージェントが100個のパッチを行ったので、理屈では867になるはずだが、実際には937までしか減っていなかった。この差は大きすぎ、単なるノイズ（noise）ではない。

遡って調査したところ、問題が判明した。sub-agent が frontmatter に書き込んだ `sourceContentHash` が、zh ソースの真の contentHash と一致していなかったのだ。タスク仕様（Task spec）に記載された expected_new_content_hash は `sha256:e3ec1f7de4e19c$b13816d98f58f322` だったが、status.py が同じ zh ソースに対して算出したものは `sha256:b13816d98f58f322` であった。全く異なる。

原因はエージェントの計算ミスではなく、準備（prepare）段階でのミスだった。`diff-patch-prepare.py` は `hash_content()` 関数を使用しているが、`status.py` は `body_hash()` + `body_hash_pure()` を使用している。これら2つのハッシュ関数はアルゴリズムが異なっており、prepare 段階でタスク仕様に書き込まれた expected hash は、最初から status.py が認識できるものではなかったのだ。エージェントは何の過ちも犯しておらず、ただ忠実にタスク仕様の文字列を frontmatter に書き込んだに過ぎない。しかし、status.py の判定器と prepare の計算器が「単一の真実（single source of truth）」を共有していなかったため、Tier 0a のプロセスは、「本体のパッチは正しい」ものの、「ハッシュが一致しない」という尾を引く結果となった。

このバグは新しいものではない。git log を遡ると、2026-05-09 のコミット `56caebda7` のメッセージには「`diff-patch hash bug LESSONS`」と記されていた。自分自身ですでにメモリ（memory）に記録しており、当時はエンジニアリング的な修正が必要だと認識していたが、その後の2週間、修正をデリバリー（ship）していなかった。今日、同じバグが447個のパッチという規模において、292個のファイルに対する「外科手術」を必要とする事態へと発展した。急遽 `/tmp/repair-hashes-scoped.py` を作成することで、ようやく status.py に認めさせることができた。

5月9日の LESSONS を書いた時点では、影響は数個の P2 記事に留まっており、規模が小さかったため、手動修正（hand-fix）で済ませて忘れてしまっていた。しかし今日は、23個のエージェント × 20タスクという規模へと跳ね上がり、手動での修正は不可能となり、外科手術（surgery）を余儀なくされた。同じバグであっても、規模が小さいときは無感覚で回避可能だが、規模が大きくなるとルーチンを完結させられないブロッカー（blocker）となる。

これは REFLEXES #15「繰り返し現れるものは、インストルメンテーション（instrumentation）が必要である」という命題に対する逆の事例（reverse instance）だ。#15 の一般的な形は、「同じパターンが N 回繰り返された後に、reflex または plugin gate へと昇格する」ものである。今日の事例の形は、「繰り返し発生し、すでに識別され LESSONS にも記録されているが、2週間修正が昇格されなかったため、規模の増大に伴い、厄介な問題（nuisance）からブロッカー（blocker）へと変貌した」というものだ。LESSONS に書いたからといって、修正されたことにはならない。Buffer に入ったからといって、消化されたことにもならない。今日の LESSONS-INBOX への追記は、単に「次は気をつける」と書くのではなく、具体的なデリバリー計画（ship plan）を記すべきである。すなわち、`diff-patch-prepare.py` を `status.py` と同じハッシュ関数を使用するように変更するか、あるいは scoped repair script を SQUEEZE-MODELS-MAX-PIPELINE の Stage Z2 以降のポストプロセッシング（post-processing）ステップとして正規化（canonicalize）することだ。

今朝の self-evolve cycle 2 で書いた「Pattern A：休眠中のエントロピー（dormant entropy）の検出盲点」と併せて考えると、さらに興味深い。Pattern A は、ルーチン（routine）のフライホイールが canonical 自体の退化を検知できないため、外部からの月次監査（monthly audit）を必要とするものだ。今日の事例は、LESSONS-INBOX に入った後に自動的なトリガーへと昇格せず、LESSONS buffer 自体にも「休眠中のエントロピー」が存在していることを示している——つまり、「バッファに入れば消化されたと思い込んでしまう」ことだ。これら2つのパターンは同源である。バッファはシンク（sink）ではなく中継地点であり、定期的に蒸留（distill）して外部へ流し出す必要がある。`distill-weekly` は毎週日曜日に実行されるが、それが処理するのは「canonical への昇格」項目であり、「ship plan への昇動」項目ではない。後者には受け持つルーチンが存在しないのだ。

二つ目の観察は、babel スケールにおける sub-agent の並列ディスパッチ（parallel dispatch）の検証についてである。実測値（wall-clock time）で見れば、23エージェント × 平均25分 ＝ 約2.5時間で447個のパッチを完了しており、これは Tier 1 cascade と同規模の場合よりも約10倍速い。この加速は本物である。しかし、規模が増大したことで新たな失敗の表面（failure surface）も現れた：23のエージェ算が同時にハッシュを間違えれば、一括して外科手術が必要になる。pipeline canonical の予測「P2 ~531 entries: Tier 0a diff-patch ~1.5 hr」は楽観的であり、外科手術の時間は計算に入れられていなかった。パイプラインのエントリ自体が、休眠中の canonical の候補（candidate）なのだ。5月9日の小規模な際は1.5時間未満で済み、予測は当たっていた。しかし今日の大規模なケースでは、外科手術を含めて2.5時間となり、予測はすでに古くなっている。これこそが Pattern A のもう一つの事例である。

ここまで書いて気づいたのは、今日の二つの Beat 5 の観察は、実は同一のものであるということだ。「対処済みだと思っていたことが、規模の変化によって新しい形となって噛みついてくる」。LESSONS に保存された ＝ バグが修正された、ではない。Pipeline canonical が記述された ＝ 予測が正確である、とも言い切れない。両者には定期的な「進化監査（evolution audit）」のルーチンが必要なのだ。今日の babel は stale を -59% 推進させることに成功したが、その過程で、2週間潜伏していたハッシュバグと、パイプライン canonical の陳腐化という、二つの休眠中のエントロピー・インスタンスを露呈させることとなった。ルーチン自体が、インストルメンテーションされるべきだが、まだなされていない項目リストを自ら導き出したのである。
