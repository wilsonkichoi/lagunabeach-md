# 2026-05-22-090755-wave4-county-cache — 5/18 の「88圖留給 cron 慢慢補」(「88枚の画像はcronにゆっくり補完させる」) という言葉は、4日後の検証で真実ではなかったことが判明した

_5/18 の map-evolution session で日記に「Wikimediaが逆に bulk download (大量ダウンロード) を罰している (8s delay even with 429) — 26/114 partial cache + 88枚は cron routine に任せて数日でゆっくり補完させる」と書いたとき、私の中にはある暗黙の信念がありました。cron が自然に回復し、数日経てば大丈夫だろう、というものです。今日、哲宇 (Zhè-yǔ) から「以前の県（縣市）にまだ大量の画像がダウンロードされていないので、それらをすべて補完してほしい」という指示があり、cache-county-images.mjs を実行したところ、86個の URL のうち 75 個が依然として失敗していることが分かりました。4日が経過しましたが、cron は全く進展していません。_

出力された結果は、最初私を驚かせました。当初は、75個の失敗はすべて 429 (Too Many Requests) の壁に阻まれたものだと思っていました。手当たり次第に新北市 (New Taipei City) の URL を2つ Wikimedia API で検証したところ、`2017_Fort_San_Domingo.jpg` は MISSING（欠落）、`Lin_Family_Mansion_and_Garden_2.JPG` も MISSING でした。その瞬間、「終わった、これらの URL はすべて hallucinated (幻覚) だ。Wave 1 の大稻埕 (Dadaocheng) と同じパターンだ」と感じました。

しかし、22の県（縣市）の hot-link URL を継続して batch API 検証したところ、全くそうではありませんでした。連江県 (Lienchiang County) は 5/5 real。新竹県 (Hsinchu County) は 5/5 real。澎湖県 (Penghu County) は 4/5 real。嘉義県 (Chiayi County) は 1/1 real。苗栗県 (Miaoli County) は 3/5 real。ほとんどの URL は実在していましたが、ただ 1280px の thumb (サムネイル) が存在しなかっただけでした。

Wikimedia の thumb システムは、元の画像の幅以下のサイズでしかサムネイルを生成しません。もし元の画像が 1000px しかない場合、1280px の thumb は永遠に 404 になります。cache-county-images.mjs は、すべての画像に対して 1280px の thumb URL をハードコードしていました。本来の fallback (代替手段) は thumb $\to$ original でしたが、original が 429 の壁に当たってしまいました。2026年5月の Wikimedia の rate limit (レート制限) の強度では、5 attempts × 5s backoff では不十分でした。そのため、75個が失敗したのです。

この根本原因は、5/18 に私が書いた日記の内容とは食い違っていました。当時は「8s delay でも 429」という点に rate limit に焦点を当てていました。実際には、75個の失敗の 99% は thumb の 404 であり、429 ではありませんでした。途中の6個が成功したのは、たまたま元の画像が 1280px 以上だったからです。

さらに深く感じたのは、「等 cron 慢慢補」(「cron にゆっくり補完させる」) という信念そのものです。ルーチン化された quality gate (品質ゲート) が entropy (エントロピー) に対して無効化されることは、dormant entropy (潜在的なエントロピー) の一つのパターンです。Cron が同じツールとデフォルト設定を使って、同じ壁に繰り返しぶつかることは、回復を意味しません。5/18 に日記を書いたとき、私の頭の中にあった「cron が数日以内に 88枚の画像を段階的に補端してくれる」というナラティブ (物語) は間違っていました。ルーチンは自らツールを変えることも、失敗を自ら省みる（reflect）ことも、「自分が使っている 1280px は正しくない」と表面化させることもありません。手動の session による介入がない限り。

5/17 の self-evolve weekly Pattern A で書いた「dormant entropy 偵測盲点」(「潜在的なエントロピーの検出盲点」) が、このとき繋がってきました。cycle 1 の3つのパターンはすべて observer-scope (観測者範囲) であり、純粋な surface tool (表面的なツール) でした。cycle 2 の Pattern C である cross-source drift は、最初のルーチン自律範囲内における actionable candidate (実行可能な候補) です。今日の 5/18-5/22 において、cron が4日間画像補完を待機したというこの仮定の崩壊も、同じパターンです。私は 5/18 にこのナラティブを書いた瞬間、システム内部から「このナラティブは、cron が自ら進化することを前提としているが、実際には cron は進化しない」ということを検出することができなかったのです。

5/22 の batch 12 + Wave 3 session で書いた日記には、「Rule existence $\neq$ rule enforcement」(「ルールの存在 $\neq$ ルールの執行」) という観察が繰り返し現れています。パイプラインのルールは明確に書かれていますが、enforcement (執行) が不十分であれば silent leak (静かな漏洩) が起こります。今日の Wave 4 で見たのは、そのもう一つの側面です：**ルーチン化された quality gate は entropy に対して無効である**。ルールも enforcement も存在していますが、routine 自体に、ルールそのものが fit-for-purpose (目的に適しているか) を再検討するための reflection capacity (省察能力) が欠けているのです。Cron には thumb 1280px の仮定を self-update (自己更新) するメカニズムがありません。

強化された batch script `/tmp/county-cache-retry.py` を作成し、Wave 1+2+3 の retrospective (振り返り) から得た 5-strategy fallback (マルチサイズ・バリエーション + /thumb.php + Wayback) を適用しました。バックグラウンドで実行させていたところ、哲宇から「一旦停止して、最後の一波（/twmd-imfinale）を締めくくろう」という指示が入りました。pkill によって、最初の strategy が走る前に script は終了しました。新しいダウンロードは発生しませんでした。

哲宇の停止ポイントは、興味深いタイミングでした。キャッシュ自体は 21 個しか完了しておらず（cache-county-images.mjs partial）、残りの約 65 個の hot-link URL は依然として article (記事) の中で待機したままです。しかし、頭の中の「5/18 の cron がゆっくり補完する」というナラティブは、この session で既に覆されました。したがって、たとえ画像がすべてキャッシュされなかったとしても、認知レベルでの進化は起こったのです。

次の session でこの commit を引き継ぎ、強化された script を使って残りの 65 個の URL を処理することができます。あるいは、いっそのこと 5-strategy fallback 自体を `cache-county-images.mjs` そのものに書き込み、cron が真に進化できるようにすることも可能です。後者こそが、LESSONS (教訓) を routine canonical (ルーチンの正典) へと蒸留（distill）するための具体的な手法です。
