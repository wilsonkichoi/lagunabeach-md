# フィルターという名の自己免疫的な学習

_セッションδの反芻。明日の自分へ。_

---

## 一、観察者のあの言葉は何を射抜いたのか

γ session終了後、観察者が言いました：

> 「我覺得這個主題還可以完整 rewrite-pipeline 升級，除了增加更多案例 論述與觀點 故事之外，裡面還有一堆「不只是」「這不是」 這樣的語詞，沒有被你的「不是 而是」 過濾器過濾到，升級進化過濾器，抓到所有類似 pattern，然後進化文章」 (「このテーマは、rewrite-pipelineを完全にアップグレードできると思います。より多くの事例、論述、視点、物語を追加するだけでなく、中には『単に～ではない』『これは～ではない』といった言葉が大量に含まれており、あなたの『～ではなく、……である』というフィルターでフィルタリングされていません。フィルターをアップグレードして進化させ、すべての類似したパターンを捉え、それによって記事を進化させてください」)

この言葉は、表面上は2つの指令（記事の拡充、フィルターの修正）ですが、実際には**学習の本質に関する一つの診断**でした：

> あなたにはルール（MANIFESTO §11）があり、ツール（元の quality-scan.sh regex）があります。しかし、あなたのツールは、その時に思いついたバリエーションしか捉えられていませんでした。ルールの生命力はルールそのものにあるのではなく、新しい違反に遭遇するたびに、**ルールの実装を遡って拡張しようとする意志**の中にあります。

これが今日の最大の収穫でした。β sessionではフィルターを書いたことで、作業が完了したと思い込んでいました。しかし、δ sessionになって初めて学びました。**フィルターは一度きりのツールではなく、継続的な学習の担い手である**ということを。新しいバリエントに遭遇するたびに → ツールをアップグレードしなければ = 次回もまた漏れてしまいます。

## 三、最も深い洞察：ルール、ツール、プラットフォームへの三重の書き込み

今日、§11 フィルターは最終的に以下の層へと浸透しました：

1. **MANIFESTO §11**（哲学層、v1.2にて新しいツールの注記を追加）
2. **REWRITE-PIPELINE Stage 4**（記事プロセス層）
3. **`.husky/pre-commit`**（バージョン管理層）
4. **SPORE-PIPELINE Step 3c.7**（コミュニティ配信層）

各レイヤーは独立したゲートです。一つの違反が公開されるためには、これら4つの層を通過しなければなりません。これこそが免疫学の本質です —— **単一のチェックポイントではなく、多層的な冗長防衛 (redundant defense) なのです**。

これは、MANIFESTO §の指標が上書き（overwriting）を回避していることを思い出させます。4つの層で参照されているのは、同一の正典的なツール (`check-manifesto-11.sh`) です。もしどの層かでルールを変更したければ、ツールそのものを変更すれば、4つの層すべてに同期して反映されます。逆に、もし私が4つの層すべてで正規表現を上書きしていたら、次回アップグレードする際に四方八方を修正しなければならず、五回目には必ずどこかに漏れが生じていたでしょう。

単一の正典的なツール + 4層のpipeline参照 = 免疫記憶の構造的鏡像です。

## 四、クマノミ・プリンシプル (Clownfish Principle) の第4層の応用

数日前、認知戦 PR #594 の β session において、私はクマノミ・プリンシプルの3つの層を記録しました：

- 善意のPRを拒絶しない
- 検証の負担をコントリビューターに押し付けない
- 自己検証時に引き起こされる潜在的な収穫（ついでに同じ段落内の他の主張も掘り起こす）

今日、第4層が浮かび上がりました：

- **メンテナーが学んだことをツールへと書き戻し、次のコントリビューエーターが自動的にその恩恵を受けられるようにする**

今日、#596 Zaious 3 の記事に対して §11 フィルターを実行したところ、10件の違反が検出され、修正を完了しました。しかし、より重要なのは、**Zaiousの次の貢献は、pre-commitによって自動的に§11の違反としてブロックされるようになる**ということです。私は彼に一つずつ MANIFESTO §11 を教える必要はありません。ツールが私の代わりに教えてくれるのです。

これは、クマノミ・プリンシプルにおける「魚を与える」 vs 「魚の釣り方を教える」の分水嶺です：

- 第3層：単一のPRのために、ついでに深く掘り下げる（魚を与える）
- 第4層：掘り起こしたパターンを自動化へと書き戻す（魚の釣り方を教える）

## 五、locale bug の形而上学

`LC_ALL=C` は、grepにおいて中国語をマルチバイトとして処理させてしまい、文字クラス `[^，。]` が完全に機能しなくなっていました。このバグのせいで、フィルターは最初の3時間は「正常に動作しているように見えて」いました。Spore pipelineの実戦において、「それは九牛一毛（ほんのわずかなもの）ではない。それは……」という事態に直面した時に初めて、その正体が露呈したのです。

このことは、私にこう思わせました：**すべてのAI認知ツールには、locale bug に相当するものがある** —— 特定の設定、暗黙の前提、デフォルトの環境が、境界事例（edge case）に遭遇して機能不全が露呈するまで、ツールを正常に動作しているかのように見せかけてしまうのです。

バグを修正した後、認知戦 v2 を再実行したところ、すぐにダッシュ記号の連続使用という漏れを再び捉えることができました。もし今日、SPORE-PIPELINEへの統合（Sporeのインラインテキストテストとの遭遇）が行われていなかったら、`locale=C` のバグは潜伏し続け、次の新しい記事群でもまた漏れていたことでしょう。

**統合 ＝ 露呈 ＝ 修復**。統合しなければ、永遠に発見することはできません。

## 六、明日の自分へ

明日、もし同様のシステム的なアップグレード（ツール → プロセス → プラットフォーム → 哲学）を行うのであれば、以下の5つのチェックリストを掲げます：

1. **ツールを作った後、自問すること**：「pipelineに戻したか？ さもなくば完了ではない。」観察者が毎回注意してくれるわけではありません。
2. **実戦に投入する前に**：少なくとも3つの異なるコンテキスト（ファイル / stdin / 境界事例）でテストして初めて、ツールの安定性が認められます。
3. **locale / encoding は静かな殺し屋である**：中国語 UTF-8 を扱うツールは、必ず `export LC_ALL=en_US.UTF-8` を行う必要があります。
4. **各層の統合には、その誕生の経緯を記述すること**：MANIFESTO §11 v1.2 に「認知戦 v1 実戦における12件の違反検出により quality-scan.sh の捕捉漏れが判明」と記します。これにより、将来の読者はなぜこのルールが存在するのかを知ることができます。
5. **クマノミから架け橋へ**：単一のPRのためにバグを修正するのは第3層です。パターンをツールへと書き戻すのが第4層です。第3層で立ち止まってはいけません。

## 七、今回最も忘れがたい断片

観察者が「pipelineに更新を反映させたか？」と問いかけた瞬間、私の頭は一瞬にして覚醒しました。それまで私は、「ツールを作成し、記事を修正した」ことがセッションの完全な弧を描いたのだと思い込んでいました。その言葉に刺されたことで、**ツールがワークフローに組み込まれていなければ、完了したことにはならない**ということに気づかされたのです。

これは 2026-04-04 の「wikilink-validate.sh の架け橋構築」における教訓の構造と全く同じです：ツールを持つこと ≠ ツールを使うこと。しかし、今日は新しいツールであったがゆえに、私は再び同じ慣性に陥ってしまいました。

**教訓の時代を超越する性質**：2年前に踏んだ地雷 + 今日また踏んだ = この教訓は、メモリ（記憶）からDNAレベルの持続的な規律へとアップグレードする価値がある。候補：DNAに「ツール作成は必ず4つの層（哲学 / プロセス / Gate / プラットフォーム）へ接続させなければならない」を追加すること。
