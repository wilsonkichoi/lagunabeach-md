# 2026-05-01 γ-late4 — 「真のstale」vs「偽のstale」は、実はstatus設計の中で混同されていた二つの異なる事象でした。これらを分離したことで、1010件の翻訳がstaleから一瞬にしてfreshへと変わりました。

`status.py` で全言語をスキャンしていた際、ko（韓国語）のカバー率は 73.9% なのに `freshPct` が 0% であることを見つけました。その瞬間、強い違和感を覚えました。

ダッシュボードには 73.9% の ko 翻訳が存在すると表示されているのに、「真の健全度」は 0%？ そんなはずはありません。478件もの翻訳が、すべて現在の zh（中国語）と同期していないということはあり得ないのです。

`status.py` の `classify()` ロジックを確認したところ、原因は非常に単純でした。pre-toolkit 時代（追跡ツール導入前）の翻訳は、frontmatter に `sourceCommitSha` が欠落しており、`status.py` はそれらを一律で stale と判定していました。`reason='no-source-sha'` です。

この設計は、根本的に異なる二つの事象を一つのバケツに混ぜてしまっていました：

- **真のstale**：zh が更新され、翻訳が遅れている $\rightarrow$ 再翻訳が必要
- **偽のstale**：翻訳内容は実際には正しいが、メタデータが欠落している $\rightarrow$ メタデータを補完するだけでよい

これらが混同された結果、ダッシュボードは嘘をつくことになりました。473件の ko 翻訳が「再作業が必要」と判定されていました。もし指示通りに進めていれば、再翻訳の必要がない内容に対して約50時間の工数を費やすことになっていたでしょう。**Status設計の次元（dimension）を間違えると、下流のすべての意思決定が狂います**。

バックフィル（backfill）を書く際、最初の案では手抜きをしたいと考えていました。「`sourceCommitSha = current HEAD sha`」とすれば、一瞬で ko はすべて fresh になり、ダ去ボードは見栄えが良くなります。しかし、それはもう一つの嘘です。過去に翻訳されたすべてのファイルを「現在の zh と同期している」と偽ることになり、実際の乖離（drift）を隠蔽してしまうからです。

哲宇（Zheyu）のプロンプトが核心を突きました：「『翻訳は最新版であることを確認してくださいね』(Translation must be confirmed as the latest version)」。この言葉によって、私はやり直す決意をしました。

誠実なバージョン：`sourceCommitSha = zh sha at-or-before en file's last commit time`。つまり、「翻訳ファイルが最後にコミットされた時点において、それが対応していた zh のバージョンを想定する」という意味です。もしその後で zh が更新されれば、`status.py` は依然として drift を検知し、stale と判定します。これにより、真の drift シグナルが隠されることはありません。

実行した結果：en の偽のstale 184件が fresh になり、残り 6件が真のstaleとして残りました。ko は 412件が fresh になり、残り 62件が真のstale。fr は 393件、es は 21件。**合計 1010件の翻訳が、偽のstaleから一瞬にして true fresh へと変わりました。APIコールすら一度も使わずに**。

残った 1153件は、本当に owl-alpha による翻訳を必要とするものです。当初の 1860件よりも半分以上減少しました。**フレーミング（framing）が正しければ、作業量は自動的に削減されます**。

5言語並列ディスパッチ（dispatch）の工程には、独特のリズム感がありました。en 4 worker、ja 4、ko 8、fr 8、es 12 — 計36個の owl-alpha worker が同時に稼働します。OpenRouter の free tier は単一モデルに対してレートリミット（rate limit）がありますが、5つのタスクディレクトリすべてが owl-alpha を使用しているため、実質的に同じモデル、つまり同じクォータを消費しています。衝突が予想されましたが、実測では owl-elpha は約30の並列処理に耐えられました（スロットリングは発生しますが、停止はしません）。

実行開始から30分ほど経った頃、哲宇が作業を中断させました。「現状を再点検し、分析を行い、現在の進捗を一旦停止してコミットと保存を行ってください。間違いが発生して乖離が進みすぎるのを避けるためです」。

この反射的な行動は、長時間実行されるバッチ・ワークフローにおいて極めて重要です。Worker が長時間稼働するほど、境界条件（edge case）に抵触する可能性が高まります — レートリミットの累積、プロンプトのドリフト、切り捨て（truncation）、API エンドポイントの変化など。定期的なコミットがなければ、破滅的な事態に至ります。**「commit early, commit often」は、サブエージェントによる並列バッチ処理において特に有効です** — これは単なる Git の衛生管理ではなく、失敗を隔離するための儀式なのです。

回収が終わった後、まずコミットし、それから監査（audit）を行いました。

その後、哲宇は尋ねました。「『10件をサンプリングして、以前の翻訳がすべて問題ないか確認してください』(Sample 10 pieces to confirm previous translations are OK)」。

このプロンプトは、「品質（quality）志向」と「スループット（throughput）志向」の違いを表しています。私は数字（fresh count の上昇）を見て、事態が完了したと思い込んでいました。しかし、「fresh」というステータスは `status.py` が計算したものであり、`status.py` はフロントマターのメタデータのみを見ています — 内容が切り捨てられていないか、YAML が有効か、翻訳に一貫性があるかなどは見ていないのです。**「fresh」とはメタデータの鮮度であり、コンテンツの品質ではありません**。

ランダムに10件を精査したところ、8件は良好（タイトル訳が優雅、タグの翻訳、文化的な語彙の処理が適切）でしたが、2件は切り捨てられていました（`es/Society/taiствоan-indigenous-education...` は 31行しかなく、zh ソースは 240行。`es/History/three-foreigners-witness-1895` も約25%のみ）。

サイズ比率のスキャン機能を実装しました：全269個の新ファイルの `trans_size / zh_size` を算出。19個が比率 0.5 未満でした。そのうち4個は zh ソースが 0 bytes（空のスタブ記事の翻訳は無意味）、残り15個は owl-alpha が途中で停止したものです。

これら 19件をすべて能動的に削除（purge）しました。状態は真のベースラインへと戻りました。

DNA の候補：「**品質ゲート（Quality gate）とは、size > 0 ではなく、size > source のある比率 であることである**」。次回の `verify-batch.py` でこの工程を自動化します。
