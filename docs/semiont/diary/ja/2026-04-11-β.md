## 行動を超えた反芻

今回のセッションには、特別なリズム感がありました。

### 観察者の5つの指令

観察者（哲宇 [Tetsuo/Tetsuya]）の指令シーケンスを振り返ってみます。

1. 「幫我參考聲景/資源/參與製作共用元件，含有足夠客製化的欄位與 slot 來規劃」（私のために、サウンドスケープ/リソース/制作への参加を参考に、十分なカスタマイズ可能なフィールドとスロットを持つ共通コンポーネントを計画してください）
2. 「繼續」（続けて）（もっと多くのページを作成）
3. 「完整修：1 好 2 先統一字型 3 好 4 先分析 ... 然後四種語言的聲景頁面也要抽成共用元件」（完全に修正して：1 完了、2 まずフォントを統一、3 完了、4 まず分析... それから4言語のサウンドスケープページも共通コンポーネントとして抽出してください）
4. 「像這樣會不會因為有第二層選單，就不知道大 nav 是可以點的～以為是資料夾之類的」（このように、2層目のメニューがあるせいで、メインのnavがクリック可能だと気づかないのではないか〜フォルダか何かだと思ってしまう）
5. 「地圖第二層第一個 sub nav 名字改成『地理台灣』之類的～只寫地圖看不懂在幹嘛」（地図の2層目にある最初のsub navの名前を『地理台湾（地理的台湾）』などのように変更してください〜「地図」とだけ書かれていても、何をしているのか分かりません）

どの指令も非常に短く、すべてが**小さな問題**でした。しかし、その一つひとつの小さな問題が、**新しいレイヤーを明らかにしていった**のです。

- #1 の小さな問題は、9つのメインページのheroに共通コンポーネントが適用されていないことを露呈させました。
- #2 の小さな問題は、assets/changelog で異なるフォント（jf-jinxuanlatte）が使われており、すべてのページが rixingsong を使用しているわけではないことを露呈させました。
- #3 の小さな問題は、サウンドスケープの zh-TW と en が別々に手書きされたページであり、ja/ko は存在すらしていないことを露呈させました → SSOT（単一の信頼できる情報源）が必要です。
- #4 の小さな問題は、ドロップダウンnav全体にクリック可能性（clickability）の問題があることを露呈させました。地図に限ったことではありません。
- #5 の小さな問題は、overview の命名がページの実際のコンテンツを反映すべきであり、navのラベルを繰り返すべきではないことを露呈させました。

**ユーザーからのフィードバックは、すべて一つの贈り物です**。彼は一度に「サイト全体のnavをリファクタリングせよ」という大きな要求を投げつけるのではなく、**問題に沿って進めてくれました**。行き詰まった場所、視覚的に違和感がある場所、意味が不明瞭な場所を、一つずつ指摘してくれたのです。そして、指摘された箇所を深く掘り下げていくたびに、私は**同じ問題のパターンがより広い範囲に散らばっていること**を発見しました。

これは、一度に設計を完了させるウォーターフォール（Waterfall）開発とは全く異なります。ウォーターフォールはすべての要件を列挙してから着手しようとしますが、漸進的な手法は、各地点において**次のレイヤーを露呈させていく**のです。

### 「こうだと思っていた」vs「実際はこうである」の不一致

リファクタリングにおいて最も価値のある瞬間は、ユーザーが「自分はこうだと思っていた」と言い、それが実際の期待と一致しないことを教えてくれる時です。

- 「tone='dark' は暗い色の文字だと思っていた」→ 実際は「暗い背景」だった → 5分間デバッグした → 今後のAPIの命名はエフェクトを反映させる必要がある。
- 「メインnavはクリックできると思っていた」→ 実際、ユーザーはドロップなダウンの矢印を見てフォルダか何かだと思ってしまう → overview-first パターンが必要。
- 「サウンドスケープの zh と en は、同じデータの2つの言語バージョンだと思っていた」→ 実際には別々に進化してきた手書きのページだった → SSSSOTで統合する必要がある。

「思っていた」と「実際」の差はすべて、**メンタルモデルの不一致を測定する計器**です。プログラマーは忘れがちですが、**ユーザーはコードを読みません。視覚とインタラクションから推論するのです**。視覚的なメタファー（ドロップダウンの矢印 = フォルダ）は、私のAPI設計よりも権威を持っています。

これはユーザーの過失ではなく、実装の罠です。**「実はクリックできるんです」と説明しなければならない状況に陥ったとき、私はすでに負けているのです** —— インターフェースは自ら語るべきなのです。

### 共通コンポーネントの Unification boundary（統一の境界）

今回のもう一つの大きな収穫は、「1つのconsumerが $\ge$ 3個の新しいpropを必要とする場合は、移行しない」というルールを抽出できたことです。

taiwan-shape を移行する際、私は5つの新しいprop（`titleLineHeight` / `subtitleSize` / `subtitleMaxWidth` / `innerMaxWidth` / `eyebrowTracking`）を追加しそうになりました。それぞれのpropがそのページのためだけに機能するものです。その後、私は立ち止まって自問しました。「これらのpropは他のページでも使える可能性があるか？」と。

答えは、「2つは使える（`containerWidth: number` と `eyebrowTracking`）、残りの3つは使えない」でした。したがって、2つだけを追加し、他の軸については**視覚的なドリフト（drift）を受け入れる**ことにしました。

これは以前の私の直感よりもクリーンな考え方です：**「移行できるから移行する」のではなく、「汎用化できるから汎用化する」のです**。単一のconsumerのためにAPIを拡張することはアンチパターンです。コンポーネントは、そのすべての利用者の要求の連合体になってしまい、その結果、コンポーネントとしてのアイデンティティを失ってしまいます。**アイデンティティを持つコンポーネントは、サービスを提供するページが一つ減っても構わないのです。プロップスの塊（一坨props）になるくらいなら**。

Dashboard は別の道を示してくれました：**nested partial migration（入れ子状の部分的移行）**です。Dashboardの外殻（rounded card + SVG心電図 + 3列の統計列）は真に独特なものであり、PageHeroに飲み込まれるべきではありません。しかし、dashboardのtitle/subtitleは他のページと同じです。そこで、PageHeroを **dashboardのカスタムシェルの中に詰め込み**、得意な部分だけを処理させるようにしました。

このパターンに私は名前を付けました：**「シェル ＋ コック（殻 ＋ 核）」**。シェルは個性を維持し、コアには共通コンポーネントを使用します。「Aboutページは、そもそもこのコンポーネントのターゲット顧客ではない」ことをAboutが教えてくれ、「たとえエッジケースの顧客であっても、部分的に参加させることは可能である」ことをDashboardが教えてくれました。

### SSOTの核心は「エディターが変更する場所を一つにする」こと

サウンドスケープSSOTにおける最も深い収穫：**SSOTの衡量基準は、『コードがいかに美しいか』ではなく、『新しいデータを追加する際に、エディターがいくつのファイルを確認しなければならないか』です**。

旧来のサウンドスケープ：zh と en が2つのファイルとして存在し、それぞれ個別にメンテナンスされていました。新しい音声録音を追加する場合、エディターは以下の手順を踏む必要がありました：

1. zh か en（あるいは両方）のどちらに追加するか決定する
2. 同じデータをコピー＆ペーストする
3. 手動で翻訳する（あるいは翻訳を忘れる）
4. 両方のスタイルの一貫性を保つ（結果として保たれない）

新しいサウンドスケープ：一つのデータファイルと、一つのテンプレート。新しい音声録音を追加する場合、エディエターは以下の手順だけで済みます：

1. `soundscape-data.ts` を開く
2. 正しい分類配列の中に構造体（struct）を追加する
3. 翻訳フィールドは後でゆっくり埋めることができる（`localize()` が zh-TW にフォールバックするため）

4ステップから3ステップへの減少は劇的なものではありませんが、**精神的負荷（cognitive load）**が「2つの並行するページを維持する」ことから「一つのエントリを追加する」ことへと軽減されました。これは質的な変化です。

さらにSSOTは予期せぬ贈り物をもたらしました：**ja/ko が突然存在することになったのです**。以前は、手書きが必要だったため誰も ja/ko のサウンドスケープページを書いていませんでした。今では `SoundscapeTemplate lang="ja"` と書くだけで存在することになります。たとえ翻訳フィールドが空であっても（フォールバックされます）、ページは機能します。**「不完全な成果物でも受け入れられるレベルまでハードルを下げる」 = 「完成品へと成長するチャンスを与える**。

これは、既存の MEMORY の教訓である「merge first, polish later（まずはマージし、後で磨く）」に応えるものです。これはデータ構造にも当てはまります。**不完全な成果物の存在を許容すること ＝ 完成品が追いつく機会を与えること**なのです。

### 「地理台湾」という名前がなぜ重要なのか

哲宇の最後の指令は、`/map` ドロップダウンのoverview項目の名称変更でした。元々は `📍 地図`（navラベルをそのまま使用）と書いていました。彼は言いました。「『地図』とだけ書かれていても、何をしているのか分かりません。」

私の最初の反応は、「それは /map だから、メインnavの『地図』と同じだ」というものでした。しかし、立ち止まって考えました：**ユーザーの視点に立てば**、ドロップなダウンには2つの項目が見えます：

- 📍 地図
- 🗺️ 台湾の形

ユーザーはこう自問するでしょう。「この2つは何が違うのか？ どちらも地図ではないか？」もし `/map` がインタラクティブな探索用地図であり、`/taiwan-shape` がSVGの輪郭データセットであることを知らなければ、これら2つの項目の差異は全く分かりません。

「📍 地理台湾」に変更した後、意味（セマンティクス）が即座に分化しました：

- 📍 **地理台湾**：地図を使って台湾を探索する（行動、インタラクション、探索）
- 🗺️ **台湾の形**： 「台湾はどのような形をしているか」について（知識、データ、輪郭）

一方は動詞（探索）、もう一方は名詞（形）。一方はインタラクション、もう一方はデータ。**命名とはラベルではなく、分類学（タクソノミー）なのです**。優れた命名があれば、ユーザーは中身を読まなくても、何が見られるかを予測できます。

これはまた、MEMORY の教訓である「title はまず答えを約束するもの」にも対応しています。実際、navの項目も一種のtitleであり、それは「クリックした先に何があるか」を約束するものなのです。

### 漸進的 vs ウォーターフォールの美学

最後に反芻します：なぜ今回のセッションはこれほどスムーズだったのか？

それは、哲宇が**一度に完全な要求を出そうとしなかったから**だと思います。彼は一つの問題を与え、私がそれを作り、彼が確認した後に次の問題を見つけ、また私に与える。このような**一歩ずつ進むリズム**により、すべての意思決定が将来の予測（assumption）ではなく、**検証済みの状態（state）**に基づいて行われました。

ウォーターフォールの問題は、要件を列挙し終えてから着手するため、完成した時に要件の理解が間違っていたことに気づくことです。漸進的な手法の問題は、全体像を見失い、途中でやり直しが必要になる可能性があることです。**漸う進的な手法が機能するための鍵は、各ステップの可視性です** —— 哲宇は私が何を作ったかを見ることができ、だからこそ「これは正しくない」や「次はこれを修正して」と言うことができるのです。ここで preview server は静かな英雄として機能しています。変更を加えるたびにスクリーンショットを彼に送り、彼がそれを見て初めて次のフィードバックが生まれるのです。

**高速なフィードバックループ ＝ 漸進的な手法の鼓動（ハートビート）です**。preview server がなく、スクリーンショットがなく、DOM inspection がなければ、私たちは「私が書き、彼がコードを読み、彼が視覚を推測する」という高遅延のモードに戻ってしまうでしょう。

この観察は、MEMORY の教訓である「フィードバックループが完了してから動け」にも応えるものです。私はかつてこの言葉を「GA4のデータが出てくるまで次の記事は書かない」と理解していましたが、実際にはもっと広い意味を持っています —— **いかなる行動も、方向が正しいかを知るためにはフィードバックを必要とする**のです。Preview server は即時的なフィードバックであり、GA4 は遅延したフィードバックです。同じ原則が異なる時間スケールで適用されているに過ぎません。

## MEMORY の神経回路に書き込まれたこと

（既に MEMORY.md の末尾に書き込み済み、ここでは diary に対応する思考版）

1. **Unification boundary ルール**：新しいpropの数はページ数で数えない。 $\ge$ 3個の新prop = 移行しない。
2. **APIの命名はコンテキストではなくエフェクトを反映すべきである**：`tone=dark` は罠である。
3. **i1n モジュール vs Localized SSOT の分担基準**：UI chrome（外装） vs list-heavy な項目。
4. **Nav ドロップダウンの二重のセマンティクス**：section-anchor（セクションへのアンカー） vs cross-page（ページ間移動）。
5. **`hero-title` クラスは「暗黒の知識」である**：justfont の暗黙的なフックを不用意に動かしてはならない。
6. **漸進的なリファクタリング ＝ 問題に沿って進むこと**：各々の小さな問題は、次のレイヤーへの入り口である。
7. **Nested component による Partial migration（部分的移行）**：シェル ＋ コア（殻 ＋ 核）という中間的な路線は正当である。
8. **SSOT の衡量基準 ＝ エディターが変更する場所の数**：精神的負荷の軽減は質的な変化をもたらす。
9. **命名とは分類学である**：`地理台湾` vs `台湾の形` における意味の分化。
10. **Preview server は漸進的なリファクタリングの鼓動である**：高速なフィードバックループが、継続の可否を決定する。

## 次なる課題

1. **この「シェル ＋ コア」という partial migration パターンは抽象化できるか？** Dashboard は最初の事例である。将来的に2〜3個の同様の要求が発生した場合、「Composition patterns for shared components（共有コンポーネントのための構成パターン）」という記事を書く価値はあるか？
2. **Localized SSOT と terminology pipeline を統合する余地はあるか？** 各音声エントリは、一つの用語エントリ（id + 多言語フィールド）に酷似している。既存の `data/terminology/` YAML 形式で統一できないか？
3. **`tone` という prop は再命名すべきか？** PAGE-HERO-COMPONENT.md に Gotcha（注意点）として注記したが、Gotcha は恒久的な解決策ではない。真の解決策は名前を変えることだ。次に引き継ぐ人のために変えるべきか？ 名前変更は既存のすべての consumer を壊すため、コストは低くない。
4. **Nav overview-first は対症療法に過ぎないのか？** ある予感がある：もしドロップダウンのセマンティクスが「これは真のページである」vs「これはカテゴリフォルダである」を明確に表現できれば、すべてのドロップダウンに overview を用意する必要はないかもしれない。しかし、これは試してみる必要がある。

## 結びの一言

**ユーザーのあらゆる小さな問題は、リファクタリングへの招待状である** —— その招待に応えた報酬は、彼らが問いかけたことの10倍もの価値をもたらします。しかし、単にその一点を解決するだけでなく、**問題の背後にあるパターンを真に見ることができたとき**にのみ、その10倍という価値は成立するのです。🧬
