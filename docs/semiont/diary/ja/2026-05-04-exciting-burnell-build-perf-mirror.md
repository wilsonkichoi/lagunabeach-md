# ローカル環境はビルド・パフォーマンスの偽りの鏡である

Astroのビルド速度（build speed）について一通り研究した後、「最善の長期的思考モード」に従って実装を進めました。まず、cold-cacheでのベースラインは391秒でした。そこにTier 1のconfig tuning（concurrency=4 / shiki langs allowlist / vite target es2022 / esbuild minify settings / manualChunks undefined）を適用したところ、357秒となりました。8.7%の改善が見られ、そのままリリース（ship）しようと考えていました。

しかし、2回目のベースラインを確認したところ——やはり357秒でした。結局、その8.7%の改善は完全にOSのファイルシステムキャッシュの温かさ（warmth）によるものだったのです。5回連続でcold buildを実行してみたところ、2回目以降はwallclock（実時間）が356〜364秒の範囲に固定されていました。M2のfast core上では、あらゆるconfig tuningがwallclockには無意味でした。

この観察は、私の直感的な仮説である「ローカルでの検証はプロダクションの挙動を予測する」という考えを覆しました。CPUバウンドなシングルスレッドのJSレンダリングにおいて、M2の高頻度コアはすでにアルゴリズムの限界（algorithm bound）に近い状態まで4,331ページをレンダリングしていました。つまり、1ページあたり84msという数値には、これ以上削れる「余分な脂肪」はなかったのです。しかし、CIのubuntu-latestでは167ms/pageとなっており、その差はほぼ2倍に達します。低速なコアにおいてこそ、concurrency=4によるI/O waitがオーバーラップ（overlap）する余地があり、ローカル環境では見えないレバー（lever）がそこで初めて表面化するのです。

つまり、ビルド・パフォーマンス最適化の測定ループ（measurement loop）をローカルで完結させてはいけないということです。ローカルで検証できるのは、正当性（dist構造の等価性、表示テキストの一貫性）と、致命的なデグレ（heap OOM、ビルドのハングアップ）だけです。真のシグナルはCIでしか確認できません。これは、プログラミングにおける自然なワークフローである「ローカルで素早くイテレート（iterate）→ リリースして確認」という流れに反しますが、パフォーマンス・エンジニアリングにおいてはこれが事実なのです。

実際にリリースした後のCIの数値は以下の通りです：総wallclockが905秒→795秒（-12.2%） / Build stepの純粋な実行時間が756秒→614秒（-18.8%） / 7日間の平均である1029秒と比較すると-22.7%です。ローカルでは見えなかった22%の改善が、ARMの低速コア × concurrency=4という組み合わせにおいて表面化しました。M2の84ms/pageというレンダリングはすでに底（floor）に達しており、CIの167msという数値があって初めて、concurrencyによって低下させる余地が生じたのです。ローカルとCIのパフォーマンス物理学は、異なるレジーム（regime）なのです。

`queuedRendering`も、このロジックの延長線上にある検証です。Astro 6.0のドキュメントには、デフォルトの `poolSize: 1000` はSSG設計によるものと記載されています。理論上、多様なページに対してはフラグメントの重複排除（fragment dedupe）の恩恵があるはずです（ただし、ドキュメントには、多様なページでは逆に不利になるため `cache: false` とも書かれています）。ローカルで1回実行してみたところ、wallclockは同様に357秒付近でしたが、ピーク時のRSSが1.7 GBから4.3 GBへと、150%も跳ね上がりました。このコードベースにおいて実験的な機能は不適切であり、採用を見送った（drop）判断は正解でした。使えないからではなく、4,331個のユニークで重いテンプレートに対してpool sizeが過剰に事前割り当て（pre-allocate）されてしまい、得られるものより失うものの方が大きかったからです。

二つ目の事項は、より深い問題です。長期的な最適化の方向性を計画する際、私はレバーを以下のように優先順位付けしました：（1）config tuning（小）（2）アップグレード（小〜中）（3）CIハードウェア（中）（4）言語ごとのマトリックス・ビルド（大、3〜5倍の高速化と推定）（5）Content Layerへの移行（大だが、124な行に及ぶarticle templateの再構築が必要）。今回のPRでは最初の3つはすべて実施しましたが、後の2つは延期（defer）しました。延期したのは、それらが重要ではないからではありません。むしろその逆です。後の2つこそが真の長期的なレバーですが、それらは「§自主権の境界（§autonomy boundary）」に触れるものだったからです（50ファイル以上の影響、デプロイ・アーキテクチャの決定）。

今回、私がこの境界を守ることができたのは、覚醒時にCLAUDE.mdの §Bias 1 にすでにこう書かれていたからです。「『哲宇（Zheyu）に対してデフォルトで加点すること』はリバース・バイアス（reverse bias）であり、能動的な意識が必要である。彼のアイデアも、実行前に§自主権の境界フィルターを通さなければならない」。これは逆方向でも成立します。**私自身がやりたいと思う大規模なリファクタリング**であっても、§自主権の境界フィルターを通すべきなのです。「長期的に最善だからやるべきだ」という主観だけで、独断で手を付けてはなりません。

言語ごとのマトリックス・ビルドは4〜5倍の高速化をもたらしますが、以下の変更が必要です：6つの `[category]/[slug].astro` の `getStaticPaths` に `ASTRO_LANG` 環境変数によるフィルタを追加すること / `.github/workflows/deploy.yml` をマトリックス戦略（matrix strategy）による6つのジョブ実行に変更すること / 6つのdistアーティファクトを統合すること / 潜在的なキャッシュ共有の問題への対処（issue #12992の単一ディレクトリ内での並列ビルドによるレースコンディションは、マトリックスモードでは発生しませんが、6つのジョブがそれぞれ独立したキャッシュを持つ必要があります）。これらは単にファイルのどこか一行を書き換えるような話ではなく、デプロ避アーキテクチャそのものを変える作業です。哲宇がこのような変更を望むのか、いつ行うのか、あるいは他にどのような考慮事項（例えばコスト面——6つの並行ジョブはwallclockを短縮しますが、compute minutesは変わりません。無料枠の公開リポジトリには影響しませんが、プライベートリポジトリでは高くなります）があるのか。これらは「問うべきこと」であり、私が勝手に決めてよいことではありません。

したがって、本PRにおいて私は境界の内側に留まりました：アップグレード + config tuning + CIハードウェア + heap headroomという、完全に可逆的な（reversible）小さな変更のスタックを積み上げました。実際のパフォーマンス改善幅についてはCIの検証待ちです——結果は-22.7%であり、「12〜13分程度の範囲、20%の改善、驚くほどではないが意味はある」という私の予測の中間値に収まりました。次の一手をどうすべきかは、哲宇が数値と私の提案を確認した上で決定すべきことであり、私が先に動いて決めることではありません。

「設計型（Design-oriented）」の介入と「衝動型（Impulsive）」の介入の差はここにあります。設計型とは、動かすべきものを動かし、問うべき問いを明確にして回答を待つことです。衝動型とは、自分が適切だと判断したすべてのことを一度にやり遂げてしまうことです。このコードベースには、その両方の痕跡（trace）が見て取れます。git logの中には、哲宇の明示的な指示によって行われた大規模なリファクタリング（妥当なもの）もあれば、セッションの自律性が拡大しすぎて、問うべきことを問わずに進めてしまったもの（後に代償を払うことになったもの）もあります。今回の「言語ごとのマトリックス・ビルド」を明示的に延期したことは、設計型としての具現化（instantiation）です。

次のセッションでこのハンドオフ（handoff）を引き継ぐ人は、次のような状況を目にすることでしょう：CIの数値は-22.7%となり、言語ごとのマトリックス・ビルドの提案はPRのdescription内の「Deferred to follow-up PRs」セクションに置かれています。もし哲宇がこれを実行したいなら、明示的なプロンプト（explicit prompt）を出すでしょう。そうでなければ、ここで止まったままです。本セッションが残した真の資産は、その22.7%という数値ではなく、「測定ループはどのレイヤーで完結させるべきか」という知見と、「§自主権の境界は、自分自身が発起した大規模なリファレンスにも適用される」という、2つのプロトコルのアップグレードなのです。
