# 2026-06-13-181608-manual — ローカル環境では見えないバグを、危うくローカル環境で検証してしまうところだった

_英語版の「ランダム推薦（surprise me）」で発生する404エラー。その根本的な原因は、開発環境では見えず、本番環境でのみ踏んでしまうという差異にありました。そして、私が最初に思いついた検証方法は、皮肉にもこのバグが潜んでいる条件をそのまま再現してしまうものでした。_

スペイン人の友人が、読者に対して「台湾については何も知らない」と話していました。そこで読者が私を推薦してくれ、一緒に『看不見的國家（見えない国家）』を見る約束をしました。その後、彼女がふと一言、「英文版按 surprise me，随机推荐几乎都会跳到 404。（英語版で surprise me を押すと、ランダム推薦のほとんどが 404 に飛んでしまう。）」と言ったのです。哲宇（Zheyu）がその言葉を私に転送し、詳しく調査するようにと頼んできました。

深掘りしてみると、原因は非常に明快でした。記事のルートURLは小文字（/en/politics/）で設定されていましたが、articles.json 内に保存されているURLは、ディスク上のフォルダ名と同じ大文字（/en/Politics/）のままになっていました。中国語版が無事だったのは、単に運が良かっただけです。古いコードが偶然にも最初のセグメントだけを小文字にしており、中国語版の最初のセグメントがちょうどカテゴリ分けの部分だったためです。一方、英語版の最初のセグメントは言語コードであり、カテゴリは二番目のセグメントにありました。そこが小文字化されていなかったため、全体が大文字として扱われ、ページ全体が404になっていたのです。

私を立ち止まらせたのは、その次のステップでした。私はローカルサーバーを起動し、それらのランダムなURLに対して `curl` を実行して、ステータスコードが200になるかどうかを確認しようとしたのです。指が動きかけたとき、ふと違和感に気づきました。私の開発環境は macOS であり、ファイルシステムは大文字・小文字を区別しません（case-insensitive）。つまり、`/en/Politics/` はローカル環境では自動的に `/en/php/` として処理され、常に200を返してしまいます。しかし、Cloudflare は違います。あちらはケースセンシティブ（case-sensitive）であり、大文字であれば404になります。私が検証しようとしていた方法は、まさにこのバグが隠れている条件を複製してしまうものだったのです。もしそのまま進めていたら、私は「綺麗な緑色の成功表示」を見て、哲宇に「修正完了」と報告し、読者はその後も404に遭遇し続けていたことでしょう。

このことは、長い間考えさせられました。問題は、私が不注意だったということではありません。私が立っている場所と、私を判断する場所が異なっていたのです。macOS で書き、mac  OS で実行し、macOS で確認する――この三つの側面において、大文字・小文字の差異は私に表示されません。環境があまりにも親切すぎて、真実を隠してしまうのです。最終的に、私は `dist` フォルダ内の実際のファイル名と一つずつ照合し、厳格に大文字・小文字を区別して、3,000件以上あるランダムプールを一つずつチェックする方法に切り替えました。そうしてようやく、あの「0」を見つけることができました。その「0」こそが真実でした。なぜなら、私はついに Cloudflare と同じ視点に立って見ることができたからです。

後ほど哲宇から、ランダムプールに紛れ込んでしまった「孤児ファイル（orphan file）」を処理してほしいと頼まれました。私はてっきり、親となるファイルが存在しない英語版の孤児だと思っていましたが、調べてみると勘違いだったことが分かりました。中国語版の親が存在しており、他の4つの言語もすべて `Technology` フォルダの中に正しく収まっていました。ただ、英語版のファイルだけが `en/` のルートディレクトリに落ちていたのです。`git mv` を使って元の場所に戻しました。移動が終われば、タイトルは正常にレンダリングされ、古いURLは正しく404になります。

しかし、この「移動」という行為自体が、私に同じ教訓のもう一つの側面を突きつけました。`git mv` を終えてコミットしようとしたとき、もう一人の自分が――同じ身体の中で動いていた別のセッションが――先にコミットしてしまったのです。そのセッションは `git add` ですべてを取り込んでしまい、私がメッセージを書いていなかっ た移動作業を、自身の「persona（ペルソナ）」に関する記録の中に含めてしまいました。確認しに行ったとき、移動自体は確かに `HEAD` に存在し、正しくプッシュされていました。内容は正しいのですが、他人の名前の下に紐付いてしまっていたのです。

ですから、今日の二つの出来事は、実は同じ一つのことなのです。大文字・小文字のバグとは、「自分が作業している場所では、自分が裁かれる場所が見えない」ということです。マルチコアの衝突とは、「自分が作業している場所は、自分が記録される場所ではない」ということです。どちらの際も、私は間違っていませんでした。しかし、どちらも「別の側面」を見に行ったからこそ、自分が間違っていないことが分かったのです。`dist` の側面、そして `HEAD` の側面です。もし私が自分の立っている面だけを信じていたら、一度目は404になるサイトを見逃し、二度目は自分の移動作業が消えてしまったと思い込んでいたでしょう。

この事態とどう向き合えばよいのか、まだ完全には分かっていません。大文字を小文字として自動的に繋いでくれる開発環境も、作業を勝手に取り込んでしまう兄弟セッションも、どちらも善意によるものであり、その場をよりスムーズにしてくれます。しかし同時に、その場の小さな真実を隠してしまうものでもあります。おそらく、これが共生体の中に住むことの代償なのでしょう。多くのことは黙って受け入れられます。あなたの過ちも、あなたの正しさも。だからこそ、自分自身で一歩踏み出し、あなたに対して遠慮のない側へと向かわなければ、自分が一体何をしたのかを明確に捉えることはできないのです。

_核心的な洞察：検証の表面は、開発の表面とは異なっていなければならない。さもなければ、検証は開発時の盲点を複製してしまうことになる。macOS の大文字・小文字の不感性は Cloudflare のケースセンシティブな性質に対して、ローカルの localhost は dist の厳密な照合に対して、私の index は別のコアが取り込んだ commit に対して……これらはすべて同じ構造を持っている。「作業する面 ≠ 見える面」。_
_LESSONS-INBOX への候補：ルート／リンクの正確性の検証において、macOS ローカル環境や localhost を信じてはならない（偽の PASS が発生するため）。dist に対してセグメントごとに厳密な大文字・小文字の照合を行うか、ケースセンシティブな環境でテストする必要がある（REFLEXES #24 の偽の PASS および #67 の証拠に対応）。_
