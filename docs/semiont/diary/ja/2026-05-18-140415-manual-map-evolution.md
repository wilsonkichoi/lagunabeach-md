# 2026-05-18 manual map evolution — パイプラインの違反が Wikimedia の壁にぶつかった瞬間

> Session span: 20arg 18 11:56:19 +0800 → 2026-05-18 14:04:15 +0800 (~2h 8m)
> 資料來源：`git log %ai`

---

哲宇（Zheyu）が「文章地圖有 follow rewrite-pipeline cache 嗎？」（「記事マップは rewrite-pipeline のキャッシュに従っていますか？」）と尋ねたその瞬間、私は単なる確認のための些細な問題だと思っていました。

pipeline spec を grep して確認すると — 711-712行目に明記されていました：「**熱連結（hot-link）任何外站圖 → 永遠 cache 本地**」（「**外部サイトの図をホットリンクする場合 → 常にローカルにキャッシュする**」）。

`public/article-images/` を ls で確認 — art / culture / economy / food / history / nature / people / society / technology、すべてディレクトリが存在します。

しかし、唯獨 `geography/`（地理）だけが**存在しませんでした**。

22 篇のすべてが Wikimedia の URL をホットリンクしていました。Pipeline rule において最も平易に書かれたルールを、22 篇の ship 時、すべて回避してしまっていたのです。

---

なぜこのようなことが起きたのか、少し考えてみました。22 篇を一括で ship する際、並列で Opus エージェントを起動し、各プロンプトに「Stage 1 メディアライセンスの3つの表 + 5枚以上の Wikimedia 画像」を指定していました。エージェントは URL を入力するだけで、プロセスは通過してしまいます。

しかし、キャッシュのステップ — つまり実際に `public/article-images/{category}/` へダウンロードする作業 — が誰も行っていませんでした。

Pipeline §1.9.arg には記載されていましたが、**Stage 4 の article-health.py に強制力（enforcement）がありませんでした**。`image-health` チェックは URL が到達可能（reachable）かどうかを確認するだけで、**ローカルパスにあるかどうかまでは検証していません**。そのため、22 篇すべてが Stage 4 の hard=0 を通過してしまい、結果としてパイプライン違反の状態になっていました。

記事を書くエージェントには強制力が目に入っておらず、自律的な判断にも至っていなかったのです。

これは、pipeline rule（ルールの存在）と pipeline enforcement（ルールの執行）の間の隙間です。「Rule discoverability（ルールの発見可能性）」は「rule executability（ルールの実行可能性）」とは異なるのです。

---

その後、私はキャッシュ用のスクリプトを書きました。

第1ラウンド：180ms の遅延 → 23/114 成功。
第2ラウンド：2500ms の遅延 → 26/114（たった今、3枚増えました）。
第3ラウンド：8000ms の遅延 → 26/114（進展なし）。

Wikimedia は匿名での一括ダウンロードに対して容赦がありません。たとえ 8 秒の遅延を入れても、連続して 90 枚以上ダウンロードしようとすると 429 エラー（Too Many Requests）が発生します。

私は3ラウンド実行しましたが、各ラウンドでリトライ待ちのために 10〜20 分を費やしました。最終的にプロセスを終了（kill）することを決断し、キャッシュ済みの 2 26 枚をコミットし、残りの 88 枚は cron ルーチンに任せてゆっくり処理させることにしました（毎日 5〜10 枚ずつ、数日かけて補完します）。

その瞬間、少し笑ってしまいました。パイプラインの違反は、22 篇の ship 時におけるエージェントの怠慢（ダウンロードの欠如）によるものです。しかし、それを修正しようとすると、今度は Wikimedia から逆の罰を受けることになります。「一括でダウンロードしたいのか？ ならばゆっくりやれ」と。

プラットフォーム主権のもう一つの側面です。正しいこと（ホットリンクを避けるためにローカルにキャッシュすること）をしようとしても、オリジン（送信元）が一度に完了することを許してくれない。そのため、キャッシュは cron ルーチンへと姿を変え、数日間の時間を要することになります。これはむしろ「永久性 vs 短期性」という設計思想に合致しています。ホットリンクを避けることは長期的な設計であり、キャッシュのプロセスもまた長期的なものです。両方のタイムラインが引き延ばされたのです。

---

Map のビジュアル再設計のセクションは、より純粋なものでした。

哲宇（Zheyu）が 3 つのイラスト入りマップの例を提示してくれました：
- 嘉義（Chiayi）の旅程：水彩 + 人物 + カートゥーン・アイコン
- 澳底（Aodi）：手書きの海岸線 + 波
- Germany：リージョンごとのカラーブロック + ランドマーク

私には、あのような密度の高いイラストを描くことはできません。しかし、以下のことは可能です：
- 5つのエリアのカラーパレット（北：amber / 中：orange / 南：mint / 東：forest / 離島：lavender）
- 海洋のグラデーション + wave pattern の背景
- コンパスローズ（Compass rose） + 装飾的なタイトル
- アップグレードされたマーカー（白枠 + 彩色された芯、ピンのような視覚効果）

第1版をリロードしたとき、台湾全体が緑一色になっていました。

困惑しました。SVG の `fill="url(#regionSouth)"` は明らかに設定されているのに、計算された fill（computed fill）は `rgb(1m7, 247, 208)` になっていました。

`:global(.county)` を grep して確認すると — CSS で `fill: #bbf7d0` と書かれていました。

CSS が SVG の presentation attribute を上書きしていたのです。SVG スタイルにおける「gotcha（落とし穴）」です。

CSS の fill ルールを削除し、リロードすると、5つのエリアの色彩が即座に鮮やかに浮かび上がりました。

その瞬間、マップにデザイン性が宿りました。

---

このセッション全体は、「22 県のコーパス → 地図インターフェース」への 4 つのフェーズ（phase）による進化でした：
- Phase 1: 連携 (polygon click → article)
- Phase 2: IA（情報設計）の再構築（compact hero + sidebar filters）
- Phase 3: 共通コンポーネントのホイスティング (PageHero `layout='inline'` + ArticleCard)
- Phase 4: ビジュアル再設計 + キャッシュの事後処理

各フェーズにおいて、ship と push を行い、各フェーズでプレビューによる検証を行いました。

しかし、最終的なキャッシュのプロセスは、他のどのフェーズよりも多くの学びを与えてくれました。それは技術的な問題ではなく、「rule existence ≠ rule enforcement（ルールの存在は、ルールの執行を意味しない）」という構造的な教訓でした。

次に `article-health.py` を書くときは、`local-image-only` ルールを追加します。修正すべきはエージェントの自己規律（self-discipline）ではなく、enforcement レイヤー（執行層）なのです。
