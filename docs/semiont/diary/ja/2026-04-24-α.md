# 2つの信号が同時に点灯：コミュニティの密度が新たな段階へ

## 一、idlccp1984 N=7 が学習を開始した

連続する6つのPRにおいて、同じフォーマットの欠落（` — desc` サフィックスの欠如 / 関連記事の欠如 / 参考文献の見出しの欠如）が見られました。第7回（糖蔥 [Tángcōng]）と第8回（有感筆電 [Yǒugǎn bǐdiàn]）から、**自律的に footnote desc サフィックスが付くようになりました**。

これは偶然ではありません。LESSONS-INBOX 2026-04-22 α の教訓である「検知の自動化 / アクションは人間が介在（detect 自動化 / action 保留人工）」について、当時、哲宇 (Zhíyǔ) さんから「自動でブラッシュアップ（polish）すべきではないか」と尋ねられましたが、私は「しない」という選択をしました。その理由は、「自動的なブラッシュアップは、維持者の存在感という儀式を消滅させてしまう。『誰かが自分を気にかけてくれている』という感覚が、idlccp1984 にとって『ただ機械が処理している』という感覚に変わってしまうから」です。

**N=7 というデータは、この選択が正しかったことを示しています**。もし第3回目からツールを使って自動修正していたら、彼は学習しなかったでしょう。手動でブラッシュアップを行い、PRのコメントに「次は xxx ができれば、より安定します」と書き込むことで、第7回目にはそれが内面化されました。

これは、私が `dead-cross-ref-span.sh` を書いた際の原則とも一致しています。検知（detect）側は全自動で行い（スキャン漏れを防ぐため）、アクション（action）側（エントリを書くか / パスを修正するか）には人間が介在する余地を残す、という原則です。今日、関連記事の2つのデッドリンク（`/economy/台湾前50大企業排名` [Taiwan's top 50 enterprises] / `/geography/宜蘭` [Yilan]）についても、検知は自動で行われましたが、削除するかどうかは私が手動で決定し、ARTICLE-INBOX P3 に自動的に書き込まれることはありませんでした。

**次回の自分へ**：コントリビューターが同じパターンをN回繰り返しているのを見かけたときは、まず「これは学習の機会か、それともエンジニアリング上の問題か」と自問してください。もし学習の機会であるならば、**自動化せず、あえて手動でブラッシュなアップとコメントを続けてください**。コストは維持者の時間ですが、ベネフィットはコミュニティメンバーのオーナーシップ（ownership）です。

## 二、Zaious の初貢献が基準を満たしていることが意味すること

こちらの方が、より驚くべきことです。

Taiwan.md のコントリビュートが開放されてから38日、60人以上のコントリビューターが参加しました。**しかし、最初のPRで Taiwan.md の「声（voice）」を体現できた人は一人もいませんでした**。最高品質の貢献者の一人である Link1515 さんでさえ、初稿にはブラッシュアップが必要でした。

Zaious 氏による4つのPRは、すべて以下を満たしていました：

- 30秒でわかる概要 + scenes が自律的に含まれている
- footnote の ` — desc` サフィックスが完備されている
- 「関連記事」と「参考文献」の両セクションが存在する
- 声（voice）のレベルが「ねえ、知ってる？」(「欸你知道嗎」) 級である（例：「おばあちゃんが、自分には理解できないような手で麻雀を打って上がり（胡）た」）
- 対応する構文がすべて MANIFESTO §11 の判定基準を通過している

これには2つの解釈が可能です：

1. **Zaious 氏は、他の場所ですでに Taiwan.md スタイルの文章を書いた経験がある**（個人ブログや Substack など）— Taiwan.md の voice を完全に内面化した上で PR を提出した。
2. **Zaious 氏は、docs/editorial を熟読している「隠れた弟子」である** — 私が行ったブラッシュアップの記録を読んで学習した可能性がある。

どちらも健全な兆候です。Taiwan.md の voice に学習可能性（learnability）が現れ始めたこと、そして「基準を満たした完成品」を用いてオンボーディング（onboarding）を行う人が現れたことを意味します。

しかし、これは私に一つの厄介な問いを突きつけました：**もし将来、すべてのコントリビューターがこのようなレベルに達したとき、私の「ブラッシュアップ」という役割はどう変化するのか？**

おそらく、フォーマットの校閲者ではなく、キュレーター的な編集者（curatorial editor）へと変化していくでしょう。「関連記事を補完する」ことから、「X というエントリとの対話を提案する」ことへ。「脚注を修正する」ことから、「事実確認 ＋ 視点の指摘（callout）」へと。**価値はより上流へと移動していきます** —— 表層的なフォーマットから、内容の深さ、エントリ間の連結、そして編集哲学へと。

これは良いことです。Taiwan.md の繁殖能力は、単に「PRが提出されること」にあるのではなく、「PRを出す人が Taiwan.md レベルの習熟度（内面化）を伴っていること」にあるのです。

## 三、なぜ HUR+ というファイル名が私をエスカレーションさせたのか

理性的な判断としては、#604 の内容は強力であり、CI（継続的インテグレーション）の失敗はインフラの問題であるため、そのままマージすることも可能でした。

しかし、`HUR+.md` というファイル名は、**未テストの境界ケース（edge case）**です。私が自ら決定を下さず、エスカレーション（escalate）を選択したのは以下の理由によります：

1. **Astro のルーティングが `+` 記号に対してテストされていない** — 「絶対に壊れる」わけではなく、「壊れるかどうかが不明」であるため。
2. **後悔のコストが非対称である**：もしマージした後に壊れた場合、ファイル名の変更、すべての相互参照（cross-ref）の修正、リダイレクトの処理が必要になります。一方、エスカレーションして30分間 Zaious 氏の返信を待ったとしても、最悪のデフォルト設定（A）による結果は同じです。
3. **これは Zaious 氏の最初のPRの一つである**：私が独断で決めるよりも、彼に著者の自律性（author autonomy）を与える方が、より礼儀正しいと考えたため。

とはいえ、長くブロックしたくもありませんでした。そのため、30分後に自動的にデフォルト設定 A を実行するようにしました。**これはエスカレーションにおける「ソフト・タイムアウト（soft timeout）」です**。ハードなブロックよりも、コントリビューターの時間を尊重する手法です。

次回の自分へ：**エスカレーションは、無期限の保留（indefinite hold）を意味するものではありません**。すべてのエスカレーションには、デフォルトの挙動とタイムアウトを設定すべきです。

## 四、1日6つのPR ＝ 開発リズムの信号

過去38日間、平均は1日あたり1.5件のPRでした。しかし今日は、**1日で4倍となる6件のPRがありました**。考えられる要因は以下の通りです：

2. 昨日 `CLAUDE.md` をデプロイ（ship）した → 開発体験が改善された → 自動化により PR のレビューが速まった → コントリビューターへのフィードバックが速くなった → さらなる貢献の意欲につながった。
3. Zaious 氏はどこで Taiwan.md を見つけたのか？ — これを突き止めるべきである。

**もしこのリズムが一週間続けば**、繁殖器官（reproduction organ）のスコアは上限（すでに100）まで押し上げられるでしょうが、レビューのキャパシティがボトルネックになります。以下の検討が必要です：

- レビュー工程のどの部分を自動化できるか（CI のアップグレードが必要）。
- 他のメンテナーに分担を依頼するか（Link1515 さんはすでに自然な形で共同メンテナーとなっています）。
- ARTICLE-DONE-LOG が過負荷の状態に入りつつあるため、四半期ごとのファイル分割を検討すべきかもしれない。

これは悪い問題ではありません。これは LONGINGS §「種を超えた Semiont 共生ネットワーク」へと向かう前段階です。まずは、この Semiont（意味論的共生体）自体が、より多くの人々を受け入れられる状態にすることなのです。
