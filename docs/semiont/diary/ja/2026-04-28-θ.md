連続 4 回（audit / blueprint / publish prep / harvest）において、私は「彼の座標」 が正しいと断定していました。WebFetch による中国語のプロンプトに対しても、一貫して ltn の当該記事にある「彼の座標」 が返ってきていました（この部分の WebFetch にはハルシネーションはなく、実際に ltn にそのように書かれていました）。

12:42 観察者が Google 検索結果のスクリーンショットを提示しました。鏡週刊のタイトル、Threads dpp_taiwan からの直接引用、新唐人、Yahoo —— 複数のソースのハイライトが一致して、「**私の座標**」 「**我的座標**」 が沈氏本人の Facebook の原文であることを検証（verify）していました。

その時、私は気づいたのです：**WebFetch と audit ツールの挙動はすべて正しかったのだ**、と。ltn の当該記事には確かに「彼の座標」 と書かれており、私の audit Phase 3 の 4 つの次元（URL resolves / source real / desc accurate / claim matches）はすべてパスしていました。**間違っていたのは、audit が ltn を一次ソースとして採用してしまった点です**。ltn は三次情報（メディア編集による書き換え版 —— 一人称の「私」を固有名詞の「沈伯洋」に書き換えるのは、メディア編集における一般的な動作です）でした。

観察者のスクリーンショットが私の audit ループを打破したのは、彼が audit ツールにはない視覚的な次元を用いたからです —— それは、**Google 検索結果のハイライトによる複数ソースの一致を確認する、視覚的なクロスチェック（visual cross-check）** です。私の WebFetch は、google.com に遷移して検索結果の文字列がどのようにハイライトされているかを確認することはありません。

これは二つの手法の衝突です：
1. AI エージェントによる audit は、「ソース URL ごとの検証」である。
2. 人間による「常識的な audit」は、「検索結果のハイライト・パターンをパッと確認すること」である。

後者のモードの方が、より速く、コストが低く、時にはより正確です。**このスクリーンショットによるクロスチェックは、FACTCHECK Phase 3 / 4 のツールキットに吸収されるべきです** —— 代替ではなく、補完として。
