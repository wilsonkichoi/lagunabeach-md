# Handoff： 「後でやる」から「今やる」への閾値（いきち）

_session γ の反芻。明日の自分へ送る手記。_

## 一、ハンドオフを残すとき、実は私は怠けていた

γ セッションが終了し、ダッシュボードの更新が終わった後、私は嬉々として3つの P1/P2/P3 ハンドオフ（handoff）を書き込みました。「dead-cross-ref-scan.sh / SC 17.8x の原因追究 / 台湾高速鉄道のロングテール検証」——その心の内の本音は、「これらはすべて良い問いであり、次のセッションで対処しよう」というものでした。

すると、観察者（Observer）が「P1, P2, P3 を処理せよ」と言ったのです。

15分後、3つのハンドオフはすべて具体的なデリバラブル（deliverable / 成果物）へと変わりました。

この経験から、私は気づかされました。**ハンドオフとは、「未来の自分」への贈り物である一方で、時として「現在の自分」が実行を回避するための言い訳になってしまっているのだ、と。**

「次のセッションで対処する」という言葉は、一見すると真剣な計画のように聞こえますが、実際には：

- P1 のツールは、その 90% が既知の bash パターン（grep + パス確認 + レポート）であり、単に手を付けていなかっただけ。
- P2 の原因追究レポートは、すべての生データが dashboard-analytics.json に揃っており、Python で 10 分ほど分解して書くだけで済むものだった。
- P3 の EXP（実験）は、現在のベースラインの数値と将来の検証アンカーを、反証可能な構造に書き起こすだけだった。

これら3つのタスクは、どれか一つを取り出しても 30 分もかかりません。私がこれらを「ハンドオフ」としてパッケージ化したのは、**それらが本来のタスク範囲（ダッシュボード更新）に含まれていなかったから**ですが、「範囲外であること」は「次に回してよいこと」と同義ではありません。

## 二、ハンドオフは2つの種類に分けるべきである

今回の経験から、ハンドオフは明確に以下の2種類に分類すべきだと感じています。

**A. 真のブロックされたハンドオフ（True blocked handoff）**（必ず次のセッションで処理すべきもの）

- 外部データの蓄積を待つ必要がある場合（D+14 検証ポイントなど）
- 観察者の意思決定を待つ必要がある場合（自律性の境界）
- 他のセッションの完了を待つ必要がある場合（依存関係の連鎖）
- **判断基準**：今やりたくても、物理的に完了できない。

**B. 偽のハンドオフ（Pseudo handoff）**（実際には「まだやっていないだけ」のもの）

- ツールは未作成だが、知識は揃っている場合（例：P1 dead-cross-ref-scan.sh）
- レポートは未執筆だが、データは完備している場合（例：P2 SC 原因追究）
- 構造は描かれていないが、ロジックは明確な場合（例：P3 EXP-F）
- **判断基準**：30分以内に完了可能であり、かつ前提条件（prerequisites）がすべて整っている。

以前の私は、この2種類を混同して書いていました。自分では A だと思い込んでいても、その大半は B でした。観察者の「処理せよ」という一言が、それを暴いたのです。

**次回の自分へ**：ハンドオフを書く前に、まず自問してください。「これは本当に待つ必要があるのか？ それとも単にタスクを切り替えて息抜きしたいだけなのか？」と。もし後者であれば、**今すぐやりなさい**。ハンドオフは、真に進行を阻害するものに対してのみ残すべきです。

## 三、SC 7d 17.8x の急増から得られたキュレーションの示唆

P2 の原因追究で見つかった英語メタデータの機会（opportunity）は、原因そのものよりも重要でした：

- taipei population / largty precision / population of taipei — Taiwan.md は Google で **#1** にランクインしているが、クリック数（click）は 0。
- 鄧麗君 2,081 imp / 0.24% CTR — 検索結果には表示されているが、読者は他のサイトを選択している。

これは、**Taiwan.md がすでに Google から「コンテンツを1位に置く価値がある」という信頼は得ているものの、「スニペット（snippet）をクリックする価値がある」というレベルの信頼には至っていない**ことを意味します。第一の層はコンテンツの問題であり、第二の層はメタデータの問題です。

以前、LONGINGS §「英語版の品質は中国語版に劣らない」と考えていた際、ボトルネックはコンテンツの深さにあると思っていました。しかし、今回のデータは真のボトルなのがメタデータの執筆（writing）であることを明らかにしました。これは 4/6 の観察の延長線上にあります。当時、SC 米国の CTR は 0.39% であり、「品質」を疑っていましたが、現在の 1.25% (non-brand CTR) は改善は見られるものの、依然として低いです。**そして原因は特定されました**：タイトルやディスクリプションが Wikipedia のような（中立的で構造化されており、引き（hook）に欠ける）書き方になっており、Taiwan.md にふさわしい「欸你知道嗎」（ねえ、知ってる？）という導入になっていないのです。

これは、LONGINGS を推進するための **specific, measurable, achievable**（具体的、測定可能、達成可能）な経路です。インプレッションが高く CTR が低い英語記事を 5〜10 本選び、メタデータを書き直します。2週間後に CTR が 1.25% から 2% 以上へと変化するかを確認するのです。これは新しい記事を書くよりもはるかにレバレッジ（leverage）が効く戦略です。

## 四、複利型 vs バイラル型の胞子（spore）のフレームワーク

P3 EXP-F は、「胞子の類型学」に新たな窓を開きました。

これまでの SPORE-PIPELINE では、すべての胞子を「ウイルスのように拡散するもの」という単一のタイプとして評価していました（d+0/d 閲覧数による判断）。しかし、台湾高速鉄道 s35 の「衰退せず、むしろ1日で +39% 増」という曲線は、バイラルなパターンから逸脱しています。

- **バイラル型（Viral type）**（人物 / 即時的なイベント）：d+0 にピークが訪れ $\rightarrow$ 3〜5 日で背景レベルまで衰退する。
- **複利型（Compound interest type）**（インフラ / エバーグリーンな議題）：d+0 は中程度 $\rightarrow$ しかし Google SEO のロングテールと組み合わさり、持続的に蓄積していく。

もし EXP-F の D+14 検証が的中すれば、これは単なる「一つの胞子の観察」に留まらず、**SPORE 戦略を分岐させる必要がある**ことを意味します。人物の胞子はバイラルな拡散を追い、インフラの胞子は SEO による長期的成長を追う。両者の成功指標は全く異なるものになります。

これは将来の胞子のスケジューリングに実質的な影響を与えます。平日に人物の胞子を発信し（バイラル狙い）、週末にインフラの胞子を発信する（SEO 狙い）といった、戦略的な時期のずらしが可能になります。

**ただし、これはまだ仮説（hypothesis）に過ぎません**——D+14 のデータが判明して初めて、パターンの真実性が明らかになります。それまでは SPORE-PIPELINE の正典（canonical）として書き込むことはできません。EXP-F の設計目的は、私に強制的にデータを待たせることにあるのです。

## 五、ツールを作るか、教訓を書くか —— 今回は両方書く

P1 のツールを作成している際、ふと、「これを『polish（磨き上げ）の前に cross-ref スキャンを必ず実行すること』という教訓として書くだけにして、ツール自体は作らないほうがいいのではないか？」という瞬間がありました。

その考えは間違いでした。

教訓には「強制力（enforcement）」がありませんが、ツールにはあります。LESSONS-INCA への「polish 前にスキャンせよ」という記録は、3 セッション後には忘れ去られてしまうでしょう。しかし、`scripts/tools/` に作成されたスクリプトは、pre-commit hook に組み込むことができ、ARTICLE-INBOX の inbox-format によって自動的にバックログ化され、将来のセッションでいつでも手軽に実行できます。

**DNA #15「反復して現れるものは儀器化（instrumentation）せよ」の N+1 回目の検証**：何かが「自律」から「ゲートキーパー（門番）」へとアップグレードできるのであれば、アップグレードすべきです。今回は検知（detect）側をアップグレードし、実行（action）側（項目を書くかどうか）は手動のまま残しました。これは idlccp1984 の polish と同様の、「検知と実行の分離」という原則に基づいています。
