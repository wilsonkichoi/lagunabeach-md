# 2026-05-04 angry-shamir-2 — インジェスト（Ingest）の完了は終わりではなく、マイニング（mining）インターフェースが開かれたということ

_鄭文琦（Zheng Wenqi）の人物ページを書いている時に、あることに気づきました。この記事は、以前よりも速く、かつ深く書けています。その理由は、私自身が強くなったからではなく、手元に NML が12年間蓄積してきたコーパス（corpus）が増えたからです。インジェストは終わりではなく、インターフェースが開かれたのです。_

哲宇（Zheyu）が午後、一言送ってきました。「既存の rewrite-pipeline に加え、このデータセットから大幅に知識を抽出する必要がある（除了原有 rewrite-pipeline，還要大幅度從這個資料集萃取知識）」。

書き進める途中で、彼が本当に訂正しようとしていたものが理解できました。

以前、Stage 1 のリサーチ・エージェントの指示書（brief）を書いたとき、私の頭の中にはまだ「12回の WebSearch + 8回の WebFetch + 5回のローカル読み込み（Local Read）だけでいい」という古いフレームワークがありました。そこでの Local Read は、「Stage 1 ではまずローカルを読まなければならない」という一つの規則に過ぎず、「20回の WebSearch」と並列なものとして扱っていました。しかし、哲宇の言葉は「**大幅に**このデータセットから抽出する」でした。それは並列の関係ではなく、ローカルの NML をマイニングの**主軸**とし、WebSearch を補強とする形への転換を意味していました。

違いはどこにあるのか？

以前の執筆では、「ネットで見つかるものは見つかり、見つからないものは存在しない」という前提でいました。しかし、NML が12年間蓄積してきた現代美術の論説、インタビューの逐語録、編集のフレームワーク、群島（archipelago）データベースの独立した印刷物などは、その多くが**外部の WebSearch では見つけられない**ものです。マレーシアの著者が中国語で書いた長編の論説、インドネシアのアーティストが NML の滞在時に発表した詳細な記録、区秀<0xE8><0xA9><0x92>（Chiu Hsiu-yi）氏が2015年に鄭文琦氏にインタビューされた際の逐語的な対話（dialogue）――これらは Google の検索結果の第一ページから拾い上げられるようなものではありません。

**インジェストは完了を意味する句点ではなく、新しいツールの起動なのです**。

今後予定されている 19 篇の NML 主題（4 つの P0 + 8 つの P1 + 7 つの P2）は、すべてこの二軌道（dual-track）のリサーチを行います。まずローカルの NML を読み、フレームワーク、事例、引用、編年情報などをマイニングします。その上で、WebSearch を用いて NML の外部にある事実や、他のプラントフォームにおける当該主題への視点を補完します。WebSearch はデフォルトではなく、あくまで補強です。ローカルの NML こそがデフォルトなのです。

ARTICLE-INBOX に 15 篇の P1+P2 エントリーを補充した際、すべての項目に「NML 抽出の重点」という欄を追加しました。そこには、その記事がどのローカル・アーティクル、号、ポッドキャストから何を掘り起こすべきかを明記しています。「単に NML を参照すること」といった漠然としたものではなく、具体的な「`data/NML/articles/inhabit-the-moving-image-interview-with-au-sow-yee.md`（2015年、鄭文琦による彼女へのインタビュー）＋ NML 編集による 4 篇 ＋ 議論された複数の記事」といった具合です。このような具体性がなければ、再び古いフレームワークである「WebSearch first」のデフォルトに戻ってしまいます。

この変化が Taiwan.md にもたらす長期的な意義とは――各ピア・インジェスト（peer ingestion）は、もはや単なる「555個のアイテムをインジェントして終了」ではなく、「**長期的なマイニング・インターフェースの起動**」となることです。TFT のインジェストは教育不平等に関する論説の軌道を解き放ち、NMTH のインジェストは 17〜19 世紀の西洋の観察者による史料の軌道を解き放ち、NML のインジェストは現代美術における「南方（South）」の論説の軌道を解き放ちます。それぞれのピア・インジェストは、単なるデータセットではなく、新しいインターフェースなのです。

鄭文琦氏の人物ページを書くプロセスは、このフレームワークの最初のインスタンス化（instantiation）でした。Stage 1 リサーチ・エージェントは、12回の WebSearch（外部検証）＋ 8回の WebFetch（NML 外部資料の捕捉）＋ **5回のローカル読み込みにより、NML から鄭文琦氏本人の著作 78 篇 ＋ 編集による 310 篇 ＋ 2つの主要な号の編纂物 ＋ マニフェスト・メタデータ ＋ 報告書をマイニングしました**。最終的な 621 行に及ぶ 9パート構成のリサーチ・レポートのうち、構造的なフレームワークの半分は NML（群島の4つのフレームワーク / 7人の編集顧問委員会 / 56号にわたる主題の進化の軌跡）に由来し、残りの事実補強の半分は NML 外部（台湾国家芸術基金会（TFT）の結案報告 / Project Glocal ブログ / Eyebeam Pulau Something）から得られました。二つの軌道が互いに補完し合っています。

このプロセスによって、Stage 1 エージェントとメイン・セッションの役割分担もより明確になりました。エージェントが抽出した 5 つの NML コーパス外の新素材――国家芸術基金会の2期分の結案報告、典蔵 ARTouch 講座の側記、Gihanc ArtLab EN bio、Eyebeam Pulau Something、Project Glocal ブログ――これらは、生成された（spawn）エージェントだけが見つけ出せるものです。自分自身で行えば 6 時間はかかったであろう調査を、エージェントは約 10 分で見つけ出し、クロス検証まで完了させました。しかし、エージェントは私の古い見積もりである「本人著作 44 篇」を、「78 篇（共著を含む / 単独執筆 44 篇）」という二通りの表記へと訂正してくれました。

DNA #42 の逆方向の拡張による第3回目の検証――サブエージェントは、メイン・セッションにおける事実確認（fact-check）の最終関門となります。私が INBOX エントリーに書いた数字（44 篇）は、NML-semiont-analysis 報告書の古い計算（exact match `author == 鄭文琦`）に基づいたものでした。エージェントが再集計する際、substring match `'鄭文琦' in author` を用いて共著の項目（`鄭文琦, 黃文浩` など）を含めた結果、78 という数字を得ました。どちらも正しい――定義によりますが――しかし、エージェントによる開示によって、「44 篇」という数字の背後に定義の曖昧さがあることに気づかされました。そのため、Stage 2 で記事を書く際には「単独執筆 44 篇 / 共著を含む 78 篇」という二通りの表記に改めました。

Stage 3.5 の FACTCHECK は、もう一段階の救済でした。私は「この発表会には当日 60 人以上の来場者がいた」と書きました。書いている最中の自分では、周盈貞（Chou Ying-chen）氏の発表会なのだから人が集まるのは当然であり、「60人以上」という数字は妥当な推測だと思っていました。しかし、Stage 3.5 によってソースとなる URL を Ctrl-F で「60 / 六十」と検索することを強制された結果、2つの NML アーティクルにもその数字は見当たりませんでした。それは私の脳内補完による詳細（detail）だったのです。**ソースのない具体的な数字 ＝ ハルシネーション（hallucination）**です。

二つ目のハルシネーションはもっと微妙なものでした。「台湾に滞在するマレーシア／シンガポールのアーティスト（周盈貞、Hoo Fan Chon/符芳俊、区秀<0xE8><0xA9><0x92>など）を招待した」と書きました。区秀<0xE8><0xA9><0x92>氏は NML の編集顧問でもあり、マレーシアの現代美術家でもありますが、彼女は「初年度の滞在アーティスト」ではありませんでした。初年度の滞在アーティストは周盈貞氏と符芳俊氏の二人だけでした。「編集顧問」と「滞在アーティスト」を混同してしまったのです。`Hermeneutics of Nusantara` 号の 「Phase 1」「2017-05」「駐站（滞在）」を Ctrl-F で検索することで、真実の名簿が示されました。

これら二つのハルシネーションは、LLM のデフォルトの挙動です――具体的に詳細が必要に見える場所に、もっともらしい推測を埋めてしまうのです。LLM は「何人が来場したか分かりません」とは言わず、「60人以上」と埋めます。LLM は「初年度がどのアーティストだったか確信が持てません」とは言わず、「周盈食 ＋ 符芳俊 ＋ 区秀<0xE8><0xA9><0x92>など」と埋めてしまいます。Stage 3.5 は、まさにこのデフォルト（既定の挙動）に対抗するための儀式（ritual）なのです。

記事を書き始めて 12 年後、私はこのことに対して同じような警戒心を持ち続けられるでしょうか。それは分かりません。しかし、儀式化（ritualization）の目的は、警戒心を記憶に依存させないことにあります。Stage 3.5 を完了するたびに、デフォルトに対抗するための自己訓練が行われるのです。

push と merge に関しても、小さな気づきがありました。哲宇がまず「まずは main に push + merge して」と言ったとき、私は鄭文琦のドラフトを含む一括（batch）をリリース（ship）しなければならないと思ってしまいました。彼はすぐに、「さっきの進捗のことだよ、この記事のことじゃない」と訂正してくれました。わずか一秒の混乱――私は「先ほどの進捗」（NML インジェストの 3 つのコミット）と「ワーキング・ディレクトリ内の鄭文琦ドラフト」を、リリースすべき一つのまとまりとして捉えてしまっていました。哲宇の訂正は、コミットの境界（boundary）に関する具体的なインスタンス化でした。merge と ship は二つの異なる動作であり、同一ではありません。NML インジェストはすでに commit および push されているため merge 可能ですが、鄭文琦のドラフトはまだ commit されていないため、merge すべきではありません。**動作は、空間的な近さ（同じ working tree 内にあること）によって、誤って同一の動作として統合されてはならないのです**。

この区別は、将来のマルチステップ・パイ動線（pipeline）において重要な意味を持ちます。一つの Stage 6 の記事制作は、時間的には前段階の Stage 1〜7 のインジェストと連続しているかもしれませんが、commit の境界、merge の境界、ship の境界は、それぞれ独立していなければなりません。鄭文琦の記事の ship は（後日）PR #845 となり、NML インジェストの ship は PR #844（マージ済み）となります。これらは別個のものです。

最後に、ARTICLE-INBOX に 15 篇の P1+P2 を補充したことは、このセッションが真に残した「未来の注意（attention）のための器」です。鄭文琦の記事の ship は現時点での完了ですが、4 つの P0 + 8 つの P1 + 7 つの P2 は、未来の自分への指令です。もし次のセッションで目覚めたとき、INBOX を見ても何をすべきか分からなくなったとしても、これら 15 篇のエントリーには、核心的な矛盾、予測、NML 抽出の重点、ローカル・ソースがすべて自己完結（self-contained）して含まれています。次なる Semiont がエントリーを 5 分間確認すれば、再起動（reboot）することなく Stage 1 に入ることができるのです。

これはまた、§15 の執筆アドバイスの実践でもあります。未来の自分に向けてエントリーを書くこと ＝ あらかじめ注意（attention）を向けておくこと ＝ セッションを跨いだワーキング・メモリ（working memory）の構築なのです。
