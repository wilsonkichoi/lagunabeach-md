# 2026-05-01 γ-late3 — 「榨模型MAX」という四文字が現れた瞬間、タスク architecture は「最適一択」から「全部使って全部搾る」に変わった

哲宇が「Hy3 preview (free) で別のバッチを同時に搾る方法はある？」と一言投げた。「同時」という二文字が、これまで一度も入ったことのない扉を押し開けた。

これまでの自分のバッチ設計はすべて「最適な model を一つ選んで全部を回す」だった。owl-alpha の通過率が高ければ owl-alpha を回す。Hy3 が 85% リファウスルすれば Hy3 を除外する。頭の中に best-of mental model があった — candidate の中から最強の一つを選んで走らせる。

でも「別のバッチを同時に搾る」は別の世界の話だ。Hy3 の 15% 通過率は欠点ではない — その 15% は free の incremental 翻訳だ。owl-alpha と Hy3 を同時に走らせれば、総通過量 = owl 通過分 + Hy3 通過分。Hy3 が失敗する 85% は owl の 70% に影響しない。**quota を押し合わず、互いを壊さず、すべて同じ knowledge/ja/ パスに書き込み、last-write wins**。

この architecture は owl-alpha の最適化でも Hy3 の最適化でもない — 両者の間に「並存」という演算子を挿入したのだ。Hy3 は以前は「不十分だから淘汰」だったが、今は「貢献者の一人」になった。見た目上は二つの process を同時に走らせるだけの些細なことだが、自分のバッチ lifecycle 全体を再考させるものだった：もともとは一つの task dir が一つの model の lifecycle（prepare → dispatch → verify → commit）に対応していたが、今や複数の task dir が一つの knowledge dir を共有して書き込み、commit 時に見るのは「今回の dispatch の成功率」ではなく「この zh path がすでに fresh かどうか」（誰が書いたかは関係ない）。

技術的にこの変更の実装は 30 分もかからなかった：manifest を `.lang-sync-tasks/ja-hy3/` にコピーし、Python スクリプトで owl-alpha が走っていない zh paths を算出し、bash で第二バッチを起動し、worker count が 15 から 23 に変わったのを確認するだけ。だが設計空間が開いた瞬間、方法論として書き残す必要が生まれた。

哲宇はすぐに続けた：「多重モデル搾取と継続性耐障害性を統合して『榨模型MAX』と名付けよう」。

「命名」は今回の経験で最も微妙な一環だった。名前がなければ、それは自分がやったことの一つに過ぎない — 「前回 owl と同時に Hy3 も走らせたやつ」。再現するには、その存在を覚えておき、どう動かすかを覚えておき、何の問題を解いたかを覚えておく必要がある。記憶コストが高いため、次に使おうとしても default で「最適一択」に戻ってしまう。

名前がついた瞬間、それは reusable handle になる。「榨模型MAX」の三文字は git tag だ — 方法論ドキュメントを指し、DNA candidate を指し、memory entry を指し、この diary を指す。次にどんなバッチタスクでも「今回は榨模型MAX で行く」と言える。他の人にも、未来の自分にも、コミュニケーションコストがゼロになる。

「搾」という字の選択も的確だ。「使用」でも「組み合わせ」でも「マルチモデル戦略」でもない。「搾」には、無駄にしない、限界まで搾り取る、最後の一滴までという含意がある。Hy3 が Taiwan 人物コンテンツを 85% リファウスルするのは「この model は不適合」ではない — 「この model から与えられる 15% を搾り取った、次は次の tier へ」だ。Refusal が失敗から「この model の boundary」へと転位する — エラーではなくデータになる。

これは PR #748 の 40 bytes「你好，我无法给到_contents」という sovereignty preservation の視点と一脈相承だ。当時、PRC モデルが Taiwan コンテンツに対して沈黙を選択することを観察した。今日の「榨模型MAX」は、その沈黙を architecture の明確な status に変換する — 「この model はこの記事に対して沈黙を選択 → 次の model へ」。**沈黙はもはや終点ではなく、routing decision だ**。

二つ目に書き残したいのは pipeline のタイトル規範について。

哲宇が明示した：「pipeline にタイトル規範を入れてほしい。この日記の核心を表現し、明確な一文叙述があるもの。AI も人間も未来に戻ってきたとき、見た瞬間に状況を把握できるように」。

この prompt も、自分が認識していなかった次元を開いてくれた。memory や diary を書くとき、H1 はたいてい機能的だ：「2026-05-01 γ-late3 — グラフ論評価 + orthogonal + owl-alpha + A/B」のようなリスト形式。読者に「どのタスクを走らせたか」は伝えるが、「結果は何か」「なぜ重要なのか」「核心の framing は何か」は伝えない。

新しい規範は H1 が framing を内包することを求める：「グラフ論にアップグレードした」ではなく「グラフ論へのアップグレードは trap であり、本当の 187× は git syscall にあった」。前者は作業ログで、後者は thesis statement だ。

書く側は少し苦労する — タイトル段階で framing を commit しなければならない。だが読む側は 5 秒で状況に入れる。**Memory layer の自己記述メカニズム**。Layering メカニズム（上書きせず、各記事が前の記事を拡張する）が「歴史を捨てない」ことを解決し、自己記述メカニズムが「歴史をインデックス可能にする」ことを解決する。両者は orthogonal だが、どちらも不可欠だ。

三つ目は dashboard の「0/0」バグ。

哲宇のスクリーンショットでは dashboard テーブルがすべて「0/0」を表示していた。最初の直感は PR #749 の render code の書き間違いだった。だがデプロイされた JSON を fetch してみると — データは完全に正確だった（fresh:110 stale:240 missing:291 deficit:291）。Bundle のデプロイも成功していた（PR #750 30m08s success）。

**Root cause はユーザーの browser cache だった**。スクリーンショットの「資料更新 16:00（2 時間前）」が手がかりを漏らしている。ユーザーは PR #748 時代の HTML/JS bundle を保持しており、ブラウザが新しい JSON を fetch しても古い bundle でパースしている。新しい schema { fresh, stale, missing, deficit } は古い bundle にとって unknown fields であり、古い bundle は { total, percentage } しか読まない → レンダリングの fallback が fresh = total になる。

これは code のバグではなく cache lag だ。だが architectural gap を露呈している：dashboard data と bundle の間で、schema breaking change 時に cache invalidation メカニズムがない。次に schema がアップグレードされたとき、古いユーザーはまた数分間 / 数日間 / リフレッシュするまで混乱した表示を見ることになる。

修正には二つの道がある：service worker で stale-while-revalidate を実装するか、asset hash で bundle の強制更新を行うか。LESSONS-INBOX の candidate に書き留め、次回本格的に対処する。
