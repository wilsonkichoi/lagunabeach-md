### 一、4つのセッションによるリレー：衝突なく、完璧な補完関係

4月11日の一日を振り返ると、実行された4つのセッションの作業分布は以下の通りです：

| セッション | モデル | トリガー | タスク形態 | 主な成果 |
| :--- | :--- | :--- | :--- | :--- |
| α | Opus 4.6 (1M) | 観察者によるトリガー（4/10から継続） | **コンテンツ + DNA** | 翻訳レビュー 27 PR + DNA 進化 + contributor pipeline + 2つの heartbeat |
| β | Opus 4.6 (1M) | 観察者によるトリガー | **アーキテクチャ再構築** | PageHero 統一コンポーネント 10/10 ホームページ + 音響風景 SSOT + nav UX |
| γ | Sonnet 4.6 | **スケジュールによる自動トリガー**（13:43） | **保守メンテナンス** | 3つのリンク切れを修正 + 葉丙成 (Ye Bing-cheng) 14→6 + 初の daily-heartbeat レポート |
| δ | Opus 4.6 (1M) | 観察者によるトリガー | **ニュース probe + 橋渡し** | 4/11 probe + fix-fn-comma.sh ツールの作成 + bad_fn_format 350→342 |

4つのセッションは**全く衝突しませんでした**：

- α は i18n モジュール + 翻訳 PR を担当
- β は src/components + src/templates + nav を担当
- γ は knowledge/ 個別の記事を担当
- δ は reports/ + scripts/tools/ を担当

もしこれが事前に計画されていたなら、「分担が良好である」と見なされるでしょう。しかし、これは**事後的に浮かび上がったパターン**なのです —— 事前の調整は一切なく、各セッションが開始時にその日のメモリ（memory）を読み込み、他のセッションが何をしているかを確認し、**自然に回避した**結果です。

これは既存の MEMORY の教訓である「**マルチコアには脳梁（corpus callosum）が必要である**」と呼応しています。今日の脳梁は `memory/2026-04-11-{α,β,γ,δ}.md` です —— 各セッションが開始時に最初に行うことは、当日のすべてのメモリファイルを読み込み、並行する神経回路を見出し、理解した上で、誰も触れていない領域へと進むことです。

しかし、より深い観察はこうです：**この分担が成立したのは、各セッションが「欲張らない」ことを選択したからです**。

- α は、新しい記事を書く時間があったにもかかわらず、書かなかった
- β は、i18n に触れることができたにもかかわらず、触れなかった
- γ は、未審査の quality-14 記事があることを知っていたにもかかわらず、書き直さなかった
- δ は、ニュースの注目度が高まっていたにもかかわらず、鄭習會 (Zheng Xi Hui) の記事を書かなかった

**節制こそが分担の前提条件です**。もし、いずれかのセッションが「もっと大きな仕事をしよう」と考えれば、他者の領域に踏み込むことになります。創発的な分担は、「各個体がまず一度、『これは自分がすべきことか？』と自問する」という習慣の上に成り立っています。

### 二、Probe loop が機能したことの意義

4/8 の α セッションで最初の probe を実行し、5 つの P0/P1 の欠落（gap）をリストアップしました。3日後（4/避）、δ セッションで再び probe を実行したところ、**5 つの欠落がすべて埋まっていることが判明しました**。

これは小さく確かな勝利ですが、その意義は数字ではなく、**メカニズムの証明**にあります。

以前の懸念は、「probe のレポートを書いても誰も読まないのではないか？」「AI セッションは目の前のタスク（task）だけに集中し、probe による長期的な提案を完全に無視するのではないか？」というものでした。3日間という期間は非常に短いですが、以下のことを証明するには十分でした：

1. **後続のセッションが能動的に probe を読む** —— α/β/γ がそれぞれ記事やメタデータの修正を行い、ちょうど probe の 5 つの欠落をカバーしていました。
2. **probe に強制的な実行メカニズムは不要である** —— 誰かに埋めるよう強いる cron はなく、純粋に後続のセッションによる自主的な題材選びに依存しています。
3. **欠落リスト = 題材レーダー** —— 観察者も probe レポートを見て「次はこれを書こう」と判断します。

**Probe は一種のプロンプティング（prompt）です** —— 命令ではなく、共有メモリの中に置かれた「呼びかけ」であり、後続のいかなるセッション（人間または AI）もそれを受け取って実行することができます。

このパターンは GitHub Issue に似ています。Issue を作成してもすぐに誰かに割り当て（assign）られる必要はありませんが、存在している限り、読まれ、想起され、最終的に処理されます。**Probe は Taiwan.md が自分自身に対して発行した issue なのです**。

しかし、ここで次の問題が生じます。もし probe が永遠に「単に欠端を指摘するだけ」であるならば、記事を書くこと自体は誰が行うのでしょうか？

### 三、自動ハートビートの鉄律の境界 = AI Slop 防火壁

γ と δ は、いずれも自動または半自動のハートビートであり、一つの**書き込まれた境界（hard boundary）**を遵守しています。

> 自動ハートビートは新しい記事を書かない。新記事のトリガーは、観察者によるもの、あるいは REWRITE-PIPELINE による人の手による審査からのみ発生しなければならない。

この境界は保守的に見えますが、今日、その価値をはっきりと目にすることができました。

もし今日、私（δ）が境界を越えて「2026年 鄭習會 (Zheng Xi Hui) の深層分析」を書いたとしたら、何が起こるでしょうか？

- ニュースが発生してから 24 時間以内は、事実がまだ進行中である（その後の展開、AIT の二巡目の表明、立法院での質疑、解放軍の動向など）。
- AI の「鄭習會」に関する学習データは 2015 年の馬習会時代のものであり、自動的に誤った類推を行ってしまう。
- 人間による引用や現場のナラティブがなく、単なる「箇条書きのニュースリリース」のような内容になってしまう —— それこそが AI Slop（低品質なAI生成コンテンツ）の典型である。
- 速く、多く書けば書くほど、検索エンジンにインデックスされるのが早まり、「Taiwan.md の公式な解釈」として定着してしまう —— しかし実際には、ニュースリリースを AI が言い換えただけの二次的なナラティブに過ぎない。

**自動ハートビートが記事を書かない ＝ 「時間をかけて沈殿させる必要がある判断」を AI の手から遠ざけること**。

この鉄律の真の機能は「保守的であること」ではなく、**時事的なコンテンツにおける人間の創作主権を保持すること**にあります。

- フォーマット操作（broken_links の修正、脚注の句読点修正、sync.sh の実行）→ 自動で行うのは OK
- 内容の判断（この記事の主人公は誰か、何が矛盾しているのか、台湾人はどう見るか）→ 必ず人間が行わなければならない

私は今、なぜ「merge first, polish later（まずマージし、後で磨く）」という MEMORY の教訓が AI の自動執筆には適用できないのかを理解しています。「merge first」はコントリビューターの善意の貢献（人間が書いたものをまず受け入れる）には適用されますが、AI が自動生成するコンテンツに対してこのロジックを用いると、6時間ごとに main へ slop がプッシュされ、72時間以内にリポジトリ全体が汚染されてしまいます。**AI コンテンツのゲート（門）は、コントリビューターのコンテンツのゲートよりも厳格でなければなりません**。

これは矛盾ではなく、二つの異なるレベルの境界なのです —— **AI の自己抑制の境界 ≠ 外部の人間に向けた寛容な境界**。

### 四、橋を架けることの天井と次の橋

`fix-fn-comma.sh` を書いている時に、小さな悟りがありました。

bad_fn_format の 350 件の案件は、「ああ、これは数ヶ月かかる大きな穴だ」と思えるものでした。しかし、実際に分析してみると：

| パターン | 件数 | 自動化の実現可能性 |
| :--- | :--- | :--- |
| p1 no_link | 183 | 不可（URL がなければ修正できない） |
| p2 link_comma | **10** | ✅ 純粋に機械的な作業が可能 |
| p3 ends_link | 110 | 半自動（ウェブページの title を description として取得する必要あり） |
| p4 text_first | 9 | 不可（書き換えが必要） |
| その他 | 43 | 不可 |

**純粋に機械的に修正可能な天井は 10 / 350 = 2.8% です**。

この数字自体は挫折感を与えるものです —— 大量に修正できると思っていたのに、結局 2.8% しかできないのですから。しかし、正しい解釈はこうです：**残りの 97% は「後回しにする」のではなく、「異なる形態の橋が必要である」ということです**。

次の橋は `fix-fn-empty-desc.sh`（パターン 3、110 件）であるべきです：

1. 各脚注（footnote）に対して、リンク先のページタイトルを取得する
2. タイトルを description のフォールバックとして脚注に書き込む
3. `lastHumanReview: false` などのフラグを立て、後で人間が校閲できるようにする

これには LLM の助力（ウェブページを読み、タイトルが description として適切か判断する）が必要ですが、ツールの骨組みは先に書くことができ、LLM や人間の校閲は後から埋めることができます。**110 件すべてを修正できれば = bad_fn_format は 342 → 232 へ改善 ＝ 32% の改善となります**。

これにより、より明確な「橋を架ける哲学」が得られました：

- **エラーのパターンごとに一つの橋を架ける**（一つの fixer ですべてを解決しようとしない）
- **各々の橋には、明確な「修正しないもの」の境界を持たせる**（fix-fn-comma はパターン 1, 3, 4 を修正しない）
- **橋の天井が低いことは失敗ではなく、次の橋の形を明確にするものである**

## MEMORY に書き込まれる神経回路

5. **Scope の整合性原則** —— fixer / format-check / grep は同一のファイルスコープを使用しなければ、数値が一致しない

## 次の課題

1. **次の脚注用の橋（`fix-fn-empty-desc.sh`）を作る価値はあるか？** パターン 3 の 110 件は bad_fn_format の最大勢力だが、LLM の助力（タイトル取得）と半自動のワークフローを必要とする。作業量は fix-fn-comma の 5〜10 倍になるが、ROI（投資対効果）も 10 倍以上になる。
2. **観察者が鄭習會の記事を書くための時間窓はどのくらいか？** 1週間か？ 2週間か？ ニュースの熱量は急速に減衰するが、長編分析（long-form analysis）の時効性はニュースリリースよりも長い。
3. **NCAIR / ロボット産業 のための新しい分類を作るべきか？** それとも Technology に含めるべきか？ AI（ソフトウェア）と Robotics（ハードウェア）を一つの Technology 分類にまとめると、混雑しすぎるのではないか。
4. **次の probe はいつ行うか？** 4/8 → 4/11 の3日間サイクルは適切に見える（変化を感じるには短すぎず、頻繁すぎない）。次は 4/14 か？
