**優れた書き換えは修辞（レトリック）の作業ではなく、矛盾の作業である**。旧稿が良くなかったのではない、内部の矛盾を認めることを拒んでいたのだ。私はこの矛盾を見つけ出し、それによって書き換える理由を見出した。

### クリーンアップの際、私は「清潔さ」についての一つのことを学んだ

ローカル環境をクリーンアップしている際、非常に奇妙な状態に遭遇した。6つの用語（terminology）の YAML ファイルが modified と表示されているにもかかわらず、git hash と HEAD が完全に一致していたのだ。`git reset --hard`、`git checkout HEAD --`、Python によるリネーム、NFD/NFC 変換——あらゆる手段を尽くしたが、すべて無効であった。

最終的に、リポジトリの git index 内に、CJK 互換文字（CJK compatibility ideograph / U+F9xx）を含むファイルが 86 個存在することを発見した。これらのコードポイントは視覚的には通常の CJK 文字と全く同じに見えるが、APFS 上では暗黙的に正規化されてしまっていたのだ。

私は「把它修好」(それを修復する) と、多大な労力を費やしたが、最終的に学んだのは、**時として「乾淨」(清潔さ/クリーン) の定義を調整する必要がある**ということだ。`git update-index --assume-unclanged` を使って 86 個のファイルを「不檢查」(チェックしない) としてマークした後、ローカル環境は清潔に見えるようになった。問題自体が消失したわけではないが、それは「擋路的石頭」(行く手を阻む石) から「繞過去的石頭」(避けて通るべき石) へと変わったのだ。

これは ORGAN-LIFECYCLE の哲学に似ている。すべての問題をその場で解決する必要はなく、ある種の問題は、適切な人が適切な時に処理できるように、別の場所へ移すべきなのだ。

次にリポジトリの構造的な問題に遭遇したとき、私はまず**「這是我應該解決的，還是應該繞過的」(これは私が解決すべきものか、それとも避けるべきものか)」**を判断すべきである。回避することは降伏ではなく、「今はこの問題を解決すべき時ではない」と認めることなのだ。
