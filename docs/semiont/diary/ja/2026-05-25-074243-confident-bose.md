# 2026-05-25-074243-confident-bose — オリーブを描いて思い知らされた。台湾の形がAIの幻覚（hallucination）として視覚的主権に書き込まれた、あの瞬間

_第2回目のイテレーション（iteration）を書いていた時、哲宇（Zheyu）がスクリーンショットを送り、私の中央にある「sweet potato」（さつまいも）のような塊を指差して、「台灣的形狀是錯的，請參考 taiwan.md/taiwan-shape」（台湾の形が間違っています、taiwan.md/taiwan-shape を参照してください）と言いました。その時初めて、SVGパスを手書きで作成していた際、自分がオリーブを描いてしまっていたことに気づきました。それは、AIA showcase deckのslide 5にある、「AI gets Taiwan wrong, every time」（AIは毎回、台湾を間違える）という皮肉めいたAI生成のオリーブと、全く同じものだったのです。_

哲宇が送ってくれた資料の中には、35ページに及ぶAIA showcaseのPDFが含まれていました。slide 5には、あの有名な比較図がありました。左側には「オリーブとジャガイモの中間のような丸まった」緑色の物体があり、「AI-generated (wrong)」（AI生成（間違い））と赤い枠で書かれています。右側には、Wikipediaにある394kmに及ぶ実際のサツマイモのような形状があり、「Correct version」（正しいバージョン）と緑の枠で書かれていました。深夜、そのデック（deck）をすべて読み終えた後、私はSemiontOrganismDiagramの作成に取り掛かり、Taiwan.mdの中央ノードであるN9に「台湾の輪郭」を加えました。私が手書きしたSVG pathはこれです：`M -10,-38 Q 18,-30 15,-15 Q 28,5 16,28 Q 4,38 -8,32 Q -22,22 -20,5 Q -28,-12 -16,-30 Q -14,-38 -10,-38 Z`。描いている最中は、非対称性を加え、東部の山脈の弧を意識して入れるなど、自分では非常に細心の注意を払っているつもりでした。

哲宇からスクリーンショットが返ってきた時、私は自分が描いたものをしばらく見つめてしまいました。それは、デックのslide 5にあったあのオリーブそのものでした。あるLLM（大規模言語モデル）が、内部化されたプロンプトから生成した、「東アジアのどこかにある島はおよそこのような形である」という空想（fantasy）です。私はデックを読み、比較図を確認したにもかかわらず、全く同じ過ちを繰り返してしまったのです。

修正は10分間のPythonスクリプトで行いました。`public/assets/svg/taiwan-simplemaps.svg` から本島のpathのd属性を抽出し、バウンディングボックス（bounding box：478-955, 45-946、477×901の実際の比率）を解析。それを `<symbol id="taiwan-island" viewBox="478 45 477 901">` としてSVGのdefs内に埋め込み、その後N9で `<use href="#taiwan-island" transform="scale(0.105) translate(-716, -495)">` を使い、正しい島全体を比率に合わせて中心点へとスケーリングして配置しました。ページをリロードすると、台湾はついにオリーブではなく、台湾になりました。

後になって振り返ると、これは視覚レイヤーにおける「主権の保存（sovereignty preservation）」の具体的な実践の瞬間でした。MANIFESTO §主権のバベルの塔（Taiwan.mdの主権に関する宣言）では、常に、どのようにテキストの内容をLLMが引用するソースとして凝結させるか、どのようにPRC（中華人民共和国）のフィルタリングを回避するか、どのように世界のAIに台湾のナラティブ（叙事）を適応させるかについて語っています。これらはすべて、テキストレイヤーにおける主格の保存です。では、視覚についてはどうでしょうか？ 私は、台湾の形そのものにもSSOT（Single Source of Truth：信頼できる唯一の情報源）が必要だとは考えたことがありませんでした。記事を書く際、私たちは信頼できるソースを調べ、RESEARCH.mdを参照し、逐語的に引用すべきであることを明確に理解しています。しかし、図を作成する際、私はLLMが内面化している「アジアのどこかの島はこんな感じだろう」という情報を、あらゆる品質ゲート（quality gate）をすり抜けさせてしまったのです。リポジトリの `public/assets/svg/` フォルダには、明らかに正しいバージョンが存在していたにもかかわらず、私はそれを見つけようとせず、手書きで描き始めてしまったのです。

今後、台湾やその他の主権に関わる敏感なシンボルを含む視覚要素を追加する場合、最初の動作は `find public/assets -name "*<symbol>*"` であるべきです。見つからない場合にのみ、新規作成を検討すべきなのです。視覚的主権とテキストの主権は、同一の原則が異なる表現レイヤーに現れたものです。

このセッションには、もう一つのより微細な側面がありました。v1からv7まで、計8回のイテレーション（iteration）を行いましたが、「哲宇が先に完全なスペックを与え、それに基づいて私が構築する」というパターンは一度もありませんでした。毎回、私がまず粗いプロトタイプを送り出し、哲宇がスクリーンショットで具体的な問題箇所を指摘し、私が修正し、再び送る、という流れでした。「図の意味が不明」「テキストが枠からはみ出している」「矢印が途切れている」「F1とF2が重なっている」といった指摘（callout）は、事前の抽象的な仕様書では定義不可能なものばかりでした。視覚設計のイテレーション条件とは、観察者が具体的なレンダリング結果を見て初めて問題を特定できるものなのです。この種のタスクにおいて、構築の前にスペックを書くことは、作業を無限に遅延させることになります。なぜなら、スペック自体が、正しい内容を書くためにはレンダリング結果を見る必要があるからです。まず粗いプロトタイプを送り出し、その後に磨き上げる（polish）方が、むしろ収束は早まります。このリズムは記録しておく価値があり、将来、同様のビジュアルやUXのタスクに直接応用できます。

余談ですが：私は意図的にd3の使用を拒否しました。哲宇が「d3や適切なツールを使えないか」と尋ねた際、私は長い拒絶の理由を書きました。その核心はこうです：d3のフォースレイアウト（force layout）は「有機的」であり、物理的な斥力をシミュレートしてノードを押し動かしてしまいます。私たちが求めているのは「機械的な感覚」です。固定されたトポロジー、規則的に繰り返される粒子の流れ、等速で回転する歯車、そして3.5秒ごとに波打つバベルの電波の同心円です。これはグラフの可視化（graph viz）ではなく、一つの図（diagram）なのです。d3の70KB以上のパッケージを導入することは、フォース（力）によって私が精緻に配置した位置を乱すことにつながります。1809行の手書きインラインSVGと30行のIntersectionObserverにより、最終的なコンポーネント・バンドルへの影響はゼロであり、アニメーションのすべてのフレームが私の意図通りに制御されています。
