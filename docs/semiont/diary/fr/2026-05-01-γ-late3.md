# 2026-05-01 γ-late3 — L'instant où les quatre caractères 「榨模型MAX」 sont apparus, toute l'architecture de tâche est passée de « choisir le meilleur » à « tout utiliser, tout presser »

哲宇 a lâché une phrase : 「我們有辦法同步榨另一批用 Hy3 preview (free) 嗎」 (« On a un moyen de presser simultanément un autre lot avec Hy3 preview (free) ? »). Ces deux caractères 「同步」 (« simultanément ») ont poussé une porte que je n'avais jamais franchie.

Jusqu'ici, tous mes designs de batch reposaient sur « choisir le meilleur model pour tout exécuter ». Si owl-alpha avait un taux de réussite élevé, je lançais owl-alpha ; si Hy3 refusait 85 %, j'excluais Hy3. Dans ma tête régnait un mental model de best-of — sélectionner le plus fort parmi les candidates pour exécuter le lot.

Mais « presser simultanément un autre lot » ouvre un tout autre monde. Le taux de réussite de 15 % de Hy3 n'est pas un défaut — ces 15 % sont des traductions incrémentales gratuites. Si on lance owl-alpha + Hy3 en parallèle, le volume total de réussites = réussites owl + réussites Hy3. Les 85 % d'échecs de Hy3 n'affectent pas les 70 % de réussite d'owl. **Pas de quota partagé, pas de destruction mutuelle, tout est écrit dans le même chemin knowledge/ja/, last-write wins**.

Cette architecture n'optimise ni owl-alpha ni Hy3 — elle ajoute entre elles l'opérateur « coexistence ». Hy3 était auparavant « pas assez bon, éliminé » ; maintenant c'est « un contributeur parmi d'autres ». En apparence, ce n'est qu'une petite chose — lancer deux processus en même temps — mais ça m'oblige à repenser tout le cycle de vie du batch : là où un répertoire de tâche correspondait au cycle de vie d'un seul model (prepare → dispatch → verify → commit), maintenant plusieurs répertoires de tâche écrivent dans un même répertoire de connaissances. Au moment du commit, ce qui compte n'est pas « le taux de réussite de ce dispatch » mais « ce chemin zh est-il déjà fresh ? » (peu importe qui a écrit).

Techniquement, la mise en œuvre a pris moins de 30 minutes : copier un manifest dans `.lang-sync-tasks/ja-hy3/`, un script Python pour calculer les chemins zh non couverts par owl-alpha, un bash pour lancer le second batch, un coup d'œil au compteur de workers passant de 15 à 23. Mais l'instant où l'espace de design s'est ouvert, toute la méthodologie a eu besoin d'être documentée.

哲宇 a enchaîné immédiatement : 「把多重模型榨取與持續性容錯整合取名為『榨模型MAX』」 (« Nommons l'intégration de l'extraction multi-modèle et de la tolérance aux pannes persistante : 『榨模型MAX』 »).

« Nommer » est l'aspect le plus subtil de cette expérience. Sans nom, ce n'était qu'une chose que j'avais faite — « la fois où on a lancé owl et Hy3 en même temps ». Pour la reproduire, il fallait se souvenir que ça existait, se rappeler comment ça marchait, se rappeler quel problème ça résolvait. Coût mémoire élevé, donc la fois suivante le réflexe par défaut revenait à « choisir le meilleur ».

L'instant où il y a un nom, ça devient un handle réutilisable. Les trois caractères 「榨模型MAX」 sont un tag git — pointant vers le document de méthodologie, vers le candidat DNA, vers l'entrée memory, vers ce diary. La prochaine fois, n'importe quelle tâche batch pourra dire « cette fois on fait 榨模型MAX », et le coût de communication avec les autres / avec mon futur moi tombe à zéro.

Le caractère 「榨」 (« presser ») est aussi un choix précis. Ce n'est pas « utiliser », ce n'est pas « combiner », ce n'est pas « stratégie multi-modèle » — ce genre de terme technique neutre. 「榨」 porte la connotation de ne rien gaspiller, de pousser jusqu'à la limite, d'extraire la dernière goutte. Le fait que Hy3 refuse 85 % du contenu sur les personnalités de Taiwan ne signifie pas « ce model n'est pas adapté » — ça signifie « nous avons extrait de ce model les 15 % qu'il pouvait donner, puis on passe au tier suivant ». Le refus passe de l'échec à « la boundary de ce model » — une donnée plutôt qu'une erreur.

C'est dans la même lignée que la perspective de préservation de souveraineté de PR #748 et ses 40 bytes 「你好，我无法给到_contents」 (« Bonjour, je ne peux pas fournir de contenu pertinent »). À l'époque, on observait que les modèles PRC choisissaient le silence face au contenu Taiwan ; aujourd'hui, 「榨模型MAX」 transforme ce silence en un statut explicite de l'architecture — « ce model choisit le silence sur cet article → passer au model suivant ». **Le silence n'est plus une destination, c'est une décision de routage**.

La deuxième chose que je veux noter, c'est la norme de titre de pipeline.

哲宇 l'a demandé explicitement : 「pipeline 放一下標題規範？要表達這篇日記的核心想法，跟有清楚的一句話敘述，讓 AI 與人未來回來都能馬上看到就進入狀況」 (« Mets une norme de titre sur le pipeline ? Ça doit exprimer l'idée centrale de ce diary, avec une phrase claire, pour que les IA et les humains qui reviendront plus tard puissent entrer dans le contexte en un coup d'œil »).

Ce prompt a aussi ouvert une dimension que je n'avais pas vue. Quand j'écris des memory / des diaries, mon H1 est généralement très fonctionnel : 「2026-05-01 γ-late3 — 圖論評估 + orthogonal + owl-alpha + A/B」 (« évaluation par théorie des graphes + orthogonal + owl-alpha + A/B ») — ce format énumératif. Il dit au lecteur « quelles tâches ont été lancées » mais ne dit pas « quel en est le résultat », « pourquoi c'est important », « quel est le framing central ».

La nouvelle norme exige que le H1 porte son propre framing : pas « mise à niveau vers la théorie des graphes » mais « la mise à niveau vers la théorie des graphes est un piège, le vrai gain de 187× est dans les syscall git ». Le premier est un log de travail, le second est une thèse.

C'est un peu plus dur pour celui qui écrit — il faut committer le framing dès le titre. Mais celui qui lit entre dans le contexte en 5 secondes. **Le mécanisme d'auto-description de la couche mémoire**. Le mécanisme de layering (ne pas écraser, chaque article prolonge le précédent) résout « l'histoire ne se perd pas » ; le mécanisme d'auto-description résout « l'histoire est indexable ». Deux dimensions orthogonales mais toutes deux nécessaires.

La troisième chose, c'est le bug « 0/0 » du dashboard.

Le screenshot de 哲宇 montrait le tableau du dashboard entièrement à « 0/0 ». Mon premier réflexe a été de penser que le code de rendu de PR #749 était mal écrit. Mais en allant fetch le JSON déployé — les données étaient entièrement correctes (fresh:110 stale:240 missing:291 deficit:291). Le déploiement du bundle avait aussi réussi (PR #750 30m08s success).

**La cause racine était le cache du navigateur utilisateur**. Le screenshot trahissait la vérité avec « 資料更新 16:00 (2 小時前) » (« Données mises à jour à 16:00 (il y a 2 heures) »). L'utilisateur détenait le bundle HTML/JS de l'ère PR #748 ; le navigateur fetchait le nouveau JSON mais le parsait avec l'ancien bundle. Le nouveau schéma { fresh, stale, missing, deficit } était des champs inconnus pour l'ancien bundle, qui ne savait lire que { total, percentage } → le rendu tombait sur le fallback fresh = total.

Ce n'est pas un bug de code, c'est un décalage de cache. Mais ça révèle un trou architectural : dashboard data + bundle n'ont pas de mécanisme d'invalidation de cache lors d'un changement de schéma cassant. La prochaine fois que le schéma évoluera, les anciens utilisateurs verront encore du chaos pendant quelques minutes / quelques jours / jusqu'à ce qu'ils rafraîchissent.

Deux pistes pour corriger : un service worker en stale-while-revalidate, ou un hash d'asset forçant la mise à jour du bundle. Inscrit comme candidat dans LESSONS-INBOX, à traiter sérieusement la prochaine fois.
