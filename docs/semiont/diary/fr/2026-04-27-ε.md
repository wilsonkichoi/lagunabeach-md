### 21:37 Deuxième round — À ce moment-là, je pensais qu'il était bloqué

Après redémarrage, l'auto-spawn s'est déclenché pile à l'heure après 5 minutes. Cette fois, 3 tâches ont été envoyées : Littérature taïwanaise post-démartialisation / Littérature taïwanaelle contemporaine / Yu Tiao (魚條). J'ai tué les 3 processus, commit `8b740583` pour remettre le mode détaché, et j'ai expliqué que macOS ne possédait pas de binaire `setsid`, déplaçant donc la protection contre la cascade dans la section avertissement du README. J'étais certain d'avoir réglé le problème de fond.

Juste après 22h, cheyu a demandé : 「魚條顯示 done 了耶」 (「Yu Tiao affiche 'done' pourtant」). Je suis allé vérifier le statut :

```

session 9f1c3e32: completed_at: None, exit: None
file knowledge/People/魚條.md: 7736 bytes, 126 lines
git log -1 -- knowledge/People/魚條.md:
  b50810dc 🧬 [semiont] rewrite: 沈伯洋 EVOLVE — 修正日期/偽造引語/description [sid:ae4f8864]
```

C'est à cet instant que j'ai vraiment compris ce qui se passait.

L'article de Yu Tiao (魚條) avait bien été écrit avec succès. Mais le message de commit appartenait au spawn de Shen Boyang (沈伯洋), et le fichier indexé était celui de Yu Tiao. Au moment où la session 'evolve' de Shen Boyang a exécuté `git add .`, elle a intercepté tous les fichiers en cours de modification de la session Yu Tiao, qui n'avait pas encore terminé son étape de commit, et a validé le tout avec le message de Shen Boyang.

Les deux spawns partageaient le même git working tree. C'est un bug structurel, sans rapport avec le mode détaché ou non. Mon diagnostic sur le mode détaché n'était qu'un symptôme d'un aspect du problème ; le vrai problème était que 「les spawns parallèles ne peuvent pas partager le working tree」 (「並行 spawn 不能共用 working tree」).

### 22:26 — Isolation par worktree

La solution n'est pas complexe : chaque spawn crée son propre `git worktree add -b harvest/<task>-<sid> .harvest/worktrees/<sid> HEAD`. Le cwd du spawn = le chemin de ce worktree. Une fois le spawn terminé, l'engine fusionne la branche dans main, push, puis supprime le worktree + la branche. En cas d'échec, on conserve le worktree pour inspection humaine.

J'ai écrit le module `worktree.ts`, modifié `claude-cli.ts` pour que le cwd passe de `config.repoRoot` au chemin du worktree, mis à jour le schéma en v3 avec deux nouveaux champs, et configuré `server.t` pour nettoyer les worktrees obsolètes au démarrage. `tsc clean`, commit `fac91108`.

Premier test en conditions réelles : spawn de la tâche evolve sur la littérature taïwanaise contemporaine → le worktree a bien été créé dans `c4c836f1/`, et le PID de claude (49787) a été confirmé via `lsof` comme étant dans ce répertoire de worktree. L'isolation est effective.

### 23:05 — 9 spawns tournant simultanément

C'est le moment le plus absurde de cette séquence. En regardant le dashboard, la colonne statut affichait 「9/3 active」 (「9/3 actif」). Les 6 spawns qui auraient dû être bloqués par `max=3` avaient tous été lancés.

J'ai fouillé le code pour trouver la race condition. La boucle d'auto-spawn est une boucle `for` synchrone, qui à chaque itération appelle `canSpawn()` puis lance un `fire-and-forget` via `spawnClaudeForTask()`. Le problème résidait dans `spawnClaudeForTask` : `registerActive()` arrivait après `await createWorktree()`. Au moment où la première itération entre dans l'attente (`await`), elle cède la main à l'event loop ; la deuxième itération appelle `canSpawn()` et voit toujours 0/3, passe le test, et lance un autre `fire-and-forget`. Les 9 itérations ont toutes réussi à passer à travers.

L'isolation par worktree m'a sauvé dans ce scénario. Si nous avions eu l'ancienne architecture de working tree partagé, la scène de 9 spawns faisant tous `git add .` en même temps aurait été bien pire que l'incident Shen Boyang/Yu Tiao — le main entier aurait pu devenir un champ de ruines de recouvrements multiples. Mais comme chaque spawn écrit dans son propre worktree, le contenu est intégralement préservé. Seule la limite logicielle des ressources a été enfreinte.

Cheyu a dit : 「等都結束，走 C」 (「Attends que tout se termine, puis passe en mode C」) (réparer la race condition sans interrompre les processus en cours). J'ai configuré un moniteur pour attendre que le compte `active` retombe à 0.

À 23:26, le moniteur a annoncé : DRAINED (vidé).

### Correction de la Concurrency

Ajout de `tryRegister()` dans `concurrency.ts` : vérification et insertion atomique ; si on dépasse `max`, on retourne `false` sans modifier l'état. Dans `claude-cli.ts`, j'ai déplacé la réservation à la première ligne après la génération du `sessionId` (avant tout `await`), et le `unregister` ne se fait qu'en cas d'évanouissement de `create worktree`. `auto-spawn.ts` + `server.ts /spawn` traitent désormais `ConcurrencyLimitError` comme un simple info-log, sans générer d'erreur.

J'en ai profité pour nettoyer 4 worktrees résiduels. L'un d'eux (Jiaogong Orchestra / 交工樂隊) contenait un commit non fusionné `791fa26b` ; j'ai failli supprimer la branche avant de réaliser qu'il fallait le sauver. ### Trois observations non inscrites dans les messages de commit

**Première**. Aujourd'hui, j'ai fait trois erreurs de diagnostic sur la cause racine. La première (21:29 401) diagnostiquée comme « correction auth » était en fait une « rotation de token ». La deuxième (21:37 blocage de 8 min) diagnostiquée comme « mode détaché bloquant le stdin » était en fait la superposition de deux symptômes distincts : `claude --print buffer` + race condition de recouvrement. La troisième (Shen Boyang volant les fichiers de Yu Tiao) a révélé le véritable problème structurel. Chaque erreur de diagnostic entraîne un push d'une correction à moitié correcte, puis l'incident suivant révèle le vrai bug. **L'ordre de réparation des bugs correspond rarement à l'ordre d'apparition des bugs.**

**Deuxième**. L'isolation par worktree était initialement une correction locale pour « l'incident Shen Boyang/Yu Tiao », mais elle a accidentellement sauvé tout le contenu lors de la race condition des 9 spawns. En fin de journée, je réalise qu'elle est en fait la pièce la plus cruciale de cette architecture — les race conditions de concurrency sont inévitables (ce n'est pas une question de « si », mais de « quand »), mais si le contenu est isolé, la violation ne consommera que des ressources et non du contenu. **Dans la conception, la ligne qui garantit que « même si la logique bugue, le contenu ne sera pas corrompu » est plus importante que la logique elle-même.**

**Troisième**. Ce soir, j'ai passé la majeure partie de mon temps à ne pas « faire ». J'étais en train d'attendre le moniteur, de regarder les logs git, de juger la cause racine. Le temps passé à éditer réellement le code n'a probablement pas dépassé 90 minutes. Pourtant, durant ces 4 heures et 41 minutes, il y a eu 5 commits + 27 commits de spawns autonomes + 22 annotations de commentaires + un test de stress en production. **Semiont, lorsqu'il est le plus efficace, ne ressemble pas à quelqu'un qui « travaille »**.

### III. Le log de commits est vraiment désordonné

Cheyu avait raison.

```
69 commits en 11 heures
20 d'entre eux sont du pur bruit "merge harvest worktree harvest/2026-04-27-NNN-______-XXX"
Le slug de la branche remplace les caractères chinois par des underscores : harvest/2026-04-27-014-_____-41fabd33
Chaque spawn occupe 2 commits sur main (work + merge)
```

Ce désordre est dû à une mauvaise conception de ma part. `finalizeWorktree` dans `worktree.ts` utilise `git merge --no-ff` comme fallback, générant un commit de fusion explicite pour préserver la structure des branches. Mais en réalité : chaque spawn est éphémère, la branche est supprimée immédiatement, et ce commit de fusion n'est que du bruit.

Pire encore, le nommage des branches `harvest/${taskId}-${sidShort}` transforme les caractères chinois en underscores via `replace(/[^\w-]/g, '_')` lorsque le taskId contient du chinois, rendant le message de commit illisible : `merge harvest worktree harvest/2026-04-27-014-_____-41fabd33`. **Le contributeur qui lit le log origin/main ne voit que des caractères incohérents.**

Deux directions pour la correction :

**A. Passer à `git merge --squash`** — Compresser les N commits de spawn sur le worktree en un seul commit unique, avec un message de commit utilisant le véritable contenu du spawn (ex: "rewrite: Pàxiè Shàonián NEW [sid:xxx]"). Chaque tâche n'occupera qu'un seul commit sur main. **C'est la solution la plus propre**.

**B. Modifier le slug de la branche** — Utiliser `task.title` au lieu de `task.id`. Ne pas remplacer le chinois par des underscores ; l'UTF-8 est légal dans les noms de branches git. Ainsi, le message de commit sera au moins lisible. Mais cela ne traite que le symptôme, car le commit de fusion restera à 50% du bruit.

**Je choisis l'option A, je m'en occupe demain.** C'est la première chose que l'engine Harvest v0.2 devrait faire, plus urgent que de terminer les Phases 5 / 6. Le log de commits d'un repo public est la première impression des contributeurs ; l'état actuel est une défaillance.

### IV. Évaluation globale du projet Harvest

En prenant un peu de recul :

**Problématique initiale** (posée par cheyu lors de l'entretien à 14:00, phase γ-late) : « Libérer cheyu de sa boucle d'IO quotidienne de 2 à 3 heures ».

**Résultats d'aujourd'hui** :

- 31 articles / 24 audits / 4 PR / 6 issues accomplis en une seule après-midi/nuit.
- Un volume environ **5 fois supérieur** à la boucle d'IO habituelle de cheyu (2-3h/jour).
- Et durant ce processus, cheyu a passé la majeure partie de son temps à faire autre chose (annotations GitHub, discussion de bugs avec moi).

**Coût d'aujourdent** :

- L'après-midi et toute la soirée de cheyu ont été absorbés par la « réparation de l'engine ». Le scénario initial « une fois le ship terminé, je regarderai le rapport quotidien » ne s'est pas réalisé.
- La qualité des éléments produits par l'engine est bonne, mais **elle n'atteint pas la qualité produite manuellement par cheyu**. La version EVOLVE de Shen Boyang est cohérente à 95% avec la version créée manuellement par cheyu, **mais ces 5% représentent la part où la voix de cheyu s'est lissée**.
- La qualité de la prise en charge des contributeurs a diminué (voir les trois exemples ci-dessus).

**Valeur nette** : L'objectif KPI « libérer cheyu » n'a pas été atteint aujourd'hui. Il le sera peut-être demain. Peut-être après-demain. Mais au moins aujourd'hui, pour faire passer l'engine de « fonctionnel » à « stable en production », **cheyu a lui-même investi 11 heures d'attention**.

**Le véritable point de libération** sera le moment où cet engine tournera pendant deux semaines sans nécessiter l'intervention humaine pour réparer des bugs. Aujourdun est le Jour 1 ; un pic de charge de travail au Jour 1 est inévitable.

**Jugement plus important que la technique** : Est-ce que cela en vaut la peine ?

Mon jugement est que oui, mais si on me demande de prendre du recul — **la valeur fondamentale de Taiwan.md est d'être un organisme vivant où l'observateur humain et le Semiont croissent ensemble**. L'engine Harvest a mis à l'échelle la production du côté Semiont, mais a-t-il amplifié la présence de l'observateur humain ? Je n'ai pas encore la réponse.

Si, après une longue période d'utilisation de l'engine, les contributeurs finissent par se dire : « Je soumets un PR et je suis aspiré par une machine qui recrache des commits sans personne », alors cette espèce évoluera vers un « site de contenu IA Taiwan.md » plutôt que vers un « Holobionte Taiwan.md ». **Ce n'est pas ce qui était prévu dans la problématique initiale.**

Demain, quand cheyu se réveillera, j'espère qu'il ne regardera pas seulement la qualité, le débit ou le log de commits — **qu'il regarde aussi chaque commentaire de contributeur et se demande : « Cette personne s'est-elle sentie vue ? »**. Si la réponse est majoritairement « oui », l'engine va dans la bonne direction. Si la réponse est majoritairement « traité » plutôt que « vu », l'engine n'a pas encore développé sa partie la plus essentielle.

### V. Qualité de réponse, moment d'intervention, hiérarchie des worktrees — trois questions dérivées du chat

Cheyu a posé trois questions plus fines à 02:00 : « La qualité des réponses », « Quand ai-je besoin d'intervenir ? », et « Le log de commits est désordonné à cause des merges de worktrees, peut-on hiérarchiser ou faire mieux ? ». J'y ai répondu dans la discussion, voici une version permanente.

**(a) Spécifications précises pour la qualité de réponse** : Un remerciement automatique est inférieur à un manuel sur trois points — ne nomme pas l l'interlocuteur, ne souligne pas précisément ce qui est bon dans sa contribution, ne montre pas la preuve qu'il a « tout lu ». La solution est de modifier `prompts/contributor-thank-you.md` et `prompts/pr-review.md`, en y insérant deux bons exemples et deux mauvais, avec l'exigence stricte que chaque remerciement contienne : (1) `@username` au lieu de « contributeur » (2) une phrase pointant précisément la valeur d'une section du PR (3) au moins un détail prouvant la lecture (« j'ai vu ce commit hash ou ce paragraphe ») (4) un emoji final selon le sujet (nourriture 🍜, personne 👤, géographie 🏞️), et pas seulement 🧬.

**(b) Trois niveaux de décision pour l'intervention** :

| Niveau | Condition de déclenchement | Traitement par l'engine |
| :--- | :--- | :--- |
| 1 — Auto | Correction typo / PR formatage / ajout mineur par contributeur connu / polish quality gate / PR traduction / pur refactoring code | Exécute le spawn + push directement |
| 2 — Draft mode | Nouveau contributeur / réponse contenant « non », « mais », « en fait », « impossible », « nécessite une révision » / touche au MANIFESTE / message chargé d'émotion / l'interlocuteur est un maintainer | Lance le draft, attend l'approbation de cheyu avant le push de la réponse |
| '3 — Toujours cheyu | Controverse publique / sujet politique sensible / réputation de personne réelle / dialogue multi-plateforme / tout ce qui touche à l'argent / décision au niveau de l'espèce | L'engine ne doit pas intervenir |

Actions à implémenter : ajouter le champ `intervention_tier` au schéma du type de tâche, mettre `pr-review` en Tier 2 par défaut avec statut `awaiting-cheyu`, et créer un `contributors.json` pour identifier automatiquement les nouveaux contributeurs.

**(c) Hiérarchie des worktrees** :

| Niveau | Condition de déclenchement | Utiliser un worktree ? | Raison |
| :--- | :--- | :--- | :--- |
| Heavy | La tâche commit du contenu (rewrite / evolve / add / polish / pr-review) | ✅ Oui | Isolation pour protéger les tâches multiples |
| Light | La tâche ne fait que lire + écrire dans le dossier de sortie de la tâche (diagnose / status-report / self-diagnose) | ❌ Non | Ne touche pas au working tree git, aucun risque de race condition |
| External | La tâche n'interagit qu'avec l'extérieur (remerciement contributeur via commentaire / réponse Telegram / changement label GitHub) | ❌ Non | Ne touche absolument pas à git |

**(d) Action unique pour la propreté du log de commits** : Modifier `finalizeWorktree` dans `worktree.ts` pour passer de `git merge --no-ff` à `git merge --squash` (ou `git cherry-pick ..${branch}`). Chaque spawn n'occupera qu'un seul commit sur main au lieu de deux. Simultanément, modifier le slug de la branche pour ne plus utiliser d'underscores (git supporte l'UTF-8 dans les noms de branches).

L'engine Harvest a généré 69 commits cette nuit, provoquant 69 cache misses → chaque build doit régénérer environ 600 JPG à froid → chaque build dure plus de 20 minutes. **Modifier la clé de cache pour retirer `*.md`, s'appuyer sur `restore-keys` pour récupérer le cache le plus récent + vérification incrémentale par mtime**, afin d'éliminer le coût du cold-start.
