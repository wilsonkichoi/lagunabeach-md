image-alt 抓三類問題：empty alt (`![](src)`)、generic alt（圖片 / image / photo 等空洞詞）、filename alt（alt 直接抄 filename）。SEVERITY=WARN soft-launch。全站 698 篇掃過 0 violation — Taiwan.md alt 文化已經很成熟，黑冠麻鷺 alt「一隻黑冠麻鷺在高雄市區綠地低頭專注覓食的側面景象，喙朝下、頸部前傾，身旁是修剪過的草坪」是 baseline 水準。Plugin 仍然 ship 當 regression guard，未來新 contributor 加圖片時擋住空 alt。

seo-meta 抓 title/description 長度 + brand-prefix description + title-desc 重複。初版用 `_cjk_aware_len` 估算（全形 ×2 + 半形 ×1），跑全站撞出 615 violations 太刺眼。回頭讀 [EDITORIAL §Description 四原則](../../docs/editorial/EDITORIAL.md) 看到 canonical 是「**120-160 字**」**字數**（CJK chars）不是混排 chars。改成 `_cjk_count`（只算 CJK 字符）並 align canonical 範圍 100-180 字（容忍上限至 180）。重跑：579 files 615 warns，主要是 description 太短 571 篇 — Taiwan.md 真的有「description 沒長到 120 字」的 legacy heal backlog。對應 [LESSONS-INBOX 2026-04-18 ε](../../docs/semiont/LESSONS-INBOX.md)「Description ≠ 30 秒概覽複寫」。

Plugin 設計細節學到一條：寫新 plugin 前先 grep canonical（EDITORIAL）的精確閾值跟單位，不憑記憶估算。第一版的混排 chars 估算偏離 canonical 的 CJK chars 計算方法，跑出來的 615 violations 大多沒實質意義。Commit `ffa5c0b5e`。

## 免疫分數 V2（A 線 — Phase 6）

核心。`scripts/core/generate-dashboard-immune.py` 330 行新檔，6 dimension weighted formula：

| Dim | 公式 | 值 | weight | 貢獻 |
|---|---|---|---|---|
| review_coverage | T1×3 + T2×1.5 + T3×0.5 tier-weighted human review % | 21.9 | 0.30 | 6.6 |
| plugin_pass_rate | HARD×0.7 + WARN×0.3 全站 pass rate | 70 | 0.25 | 17.5 |
| plugin_health | meta-health（per-plugin drift） | 100 | 0.15 | 15.0 |
| citation_density | A=100/B=80/C=60/D=40/F=0 全站 mean | 88.4 | 0.15 | 13.3 |
| tool_freshness | EDITORIAL + plugin git log %ai vs now | 100 | 0.10 | 10.0 |
| drift_velocity | 7d 新文 violation rate heuristic | 90 | 0.05 | 4.5 |

**V1 score 22 → V2 score 67**。V1 過度悲觀只看 review %，忽略 plugin HARD 100% 跟 citation A-grade 73% 等強項。

Tier breakdown 揭露真正 leverage point：T1 329 篇（人物 / 社會 / 歷史）才 20.1% reviewed，是 review bottleneck。T1 提升到 80% review 比例 → weighted +180 points，T3 同改變只 +30。tier-weighted 設計讓「該 review 的篇拉上去」比「全站 review % 上升」更高 leverage。

`scripts/core/generate-dashboard-data.js` 加 feature flag `IMMUNE_V2` env：default V1 不動，opt-in 才用 V2。安全 fallback warns if JSON missing。

寫的過程踩兩個坑值得記：

1. **citation_density 公式從 plugin violation 拿不到 grade A/B** — footnote-density 只 emit C/D/F violations，A/B 是 silent pass。改成 compute_citation_density 獨立 replicate 公式（同 plugin `_grade()` 邏輯），別 grep violation message。
2. **tool_freshness 用 file mtime 在 worktree 會全 reset** — 開 worktree 時 fs mtime 全變成 worktree 創建時間，所有 plugin 看起來都「fresh」但 EDITORIAL.md 也是 worktree 創建時的 mtime。改用 `git log -1 --format=%ai` 拿 commit time，REFLEXES #18 「時間是結構」第 N 次驗證。

## Beat 5 — 反芻

六個 phase 順利 ship 之外，更大的收穫在中間連續踩到三個一樣形狀的坑：寫 plugin 用混排 chars 估算偏離 canonical 字數定義、citation_density 從 violation message 拿不到 grade A/B、tool_freshness 用 fs mtime 在 worktree 全 reset。三個問題共通結構：**「直覺估算 / 流程偷吃步」vs「回去看 canonical / 用 git log」**。每個踩進去都要回頭重做。

REFLEXES #18「時間是結構」第 N 次驗證 — 範圍應該擴大到任何「時間相關 metric」（tool freshness / plugin age / EDITORIAL drift），都不能用 fs mtime，必須走 git log %ai。worktree 的 fs mtime 是 worktree 創建時間，整個跟現實脫節。

另一個收穫是「砍冗餘工具」是錯誤直覺。盤點 18 standalone 後發現 SSOT migration 後留下來的都是有意保留的 complementary tool（facade / advisory / batch-heal / build-time mode）。Phase C 從「砍」改「補文檔」是更高 leverage 的 reframe。哲宇 directive 用「盤點 + 進化」而不是「砍 + 整理」也許就已經暗示了這個 reframe。
