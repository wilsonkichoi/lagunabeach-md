兩個 task 表面是「執行 directive」，但骨子是同一個架構動作：補 intake layer。歷史街區補的是 article 開發層 + 老街 catalog 之間的 sub-unit deep dive 中間層、SPORE-INBOX 補的是繁殖系統 process 跟 identity 之間的 intake 緩衝層。兩個都在做「兩層之間的 missing layer」這個 pattern。

回頭看心臟系統三層 (ARTICLE-INBOX intake / knowledge/ SSOT / ARTICLE-DONE-LOG log) 跟繁殖系統三層 (SPORE-INBOX / SPORE-LOG / SPORE-HARVESTS) — 完成 SPORE-INBOX 之後兩個系統的 anatomy 才對稱。對稱本身不是目的，但兩個 work 在系統內走同一個 metabolic 路徑（intake 收 idea → process 加工 → log 沉澱）讓未來 navigation 跟 reasoning 都更便宜。

Bias 1 過濾過了兩次。第一次是收到歷史街區 directive — 規模 80-120 篇潛在很大但不在 §自主權邊界 list 上，先用 AskUserQuestion 4 題框 scope 才動筆。第二次是 SPORE-INBOX 228 entry — 哲宇 prompt 直接寫「中國正在散播一些關於 228 的假歷史」，但 Bias 1 過濾後 entry 內部 framing 必須抽出「對抗 frame」改為「first-person 紀實 frame」(per MANIFESTO §策展非百科 + §怎麼說話 + REFLEXES #28 紀實非煽情)。這條轉譯不在哲宇明示 directive 內，是 default-filter 補位的工作。

EVERGREEN-TOPIC 鐵律是這次 session 留下最有 reuse 潛力的 pattern：「沒 article 先發孢子等於空轉」這個 framing 強制兩個 system 互鎖（SPORE-INBOX 有 EVERGREEN entry 就一定要對應 ARTICLE-INBOX entry）。未來如果有「想法 buffer」類器官（如 EXHIBITION-INBOX 策展點子、PARTNERSHIP-INBOX 合作點子），可以 reuse 同一個 spawn 鐵律 pattern — 兩個 system 間的 dependency 用 schema field 強制顯化，而非靠人記得。
