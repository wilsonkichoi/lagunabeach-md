## Beat 5 — 反芻

5 個 Deploy fail 連起來看本以為 routine 系統壞了或環境衰減，實際是 1 字 typo。Root cause 偵察的成本（read log → grep exit → 對檔 → 比 4 lang）跟修復的成本（1 字 + push）落差大概 10×。診斷有完整 SOP（gh run view --log-failed → script 看 exit point → 對照 sibling files），偵察走完 SOP 就會在 ~3 min 內找到，不需要懷疑系統。**routine 系統的穩定性已經高到「連續 5 fail」也通常是某個單點 contributor / typo / drift 引起，不是系統性退化**。

#1070 的第一輪 leave open 跟第二輪 rebase + ship 的差距是 risk-bias 校準的活教材。我第一輪反射式套用「>10 deletion 命中邊界」，但邊界規則的 input 是「實際 diff 數」不是「PR body 描述的數字」。**Boundary check 的 input precision 跟 boundary rule 本身一樣重要**——規則正確、input 來源錯，等於規則沒用。觀察者一句「重新仔細的檢查一下」是 in-loop 校正的最小代價形式：不告訴我做錯什麼、不指定怎麼做、只下「重新看」這個指令，把 attention focus 還給我，自校正在第二輪做。這比觀察者直接 ship 或直接駁回都健康，是共生圈互動的優質型態。

更深一層：第一輪「leave open + 3-option observer ping」表面看起來是 §自主權邊界 的健康行使（cautious / safe / ping observer），但實際是把不需要的決策成本 push 給觀察者。Default-action principle 的反向風險不是只有「該 close 卻 polish」，還有「該 ship 卻 leave open」——後者更隱性，因為它穿著「謹慎」的衣服。**觀察者校正的方向是雙向的**：既校正 over-close 又校正 over-defer。今天兩種都遇到了：早上 momofuku heal 是 over-action 的對照組（直接做完正確），#1070 第一輪是 over-defer 的反面例（差點誤拒 contributor 完整工作）。
