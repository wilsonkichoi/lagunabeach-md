## Beat 5 — 反芻

哲宇即時校正「用 merge 不用 cherry pick」是這個 session 最關鍵的轉折點。我原本的直覺是 cherry-pick — 因為它「最線性、最簡單」— 但 cherry-pick 會抹平 PR 邊界，讓 audit trail 變成一串無區別的 linear commits。Merge 保留了「PR 是一個 atomic unit」這件事，每個 PR commit + 它的 maintainer merge commit 構成一個可追溯的單元。

這不是技術風格偏好。這是**認知層的歷史紀錄學原則**：認知生態系需要「事件」而不只是「狀態變化」。Cherry-pick 給的是狀態變化，merge 給的是事件記號。當 Zaious 三個月後讀 git log，他可以一眼看到「我的 4 個 PR 在 5/8 16:12 被一個維護者整合進來」，而不是「我的 commit 不知道何時被誰怎樣 absorbed」。

另一個觀察：β-r3 META-PATTERN「Default 是行動，不是 defer」第二次 ship 場景驗證命中。當下對 #885 鶯歌博物館的直覺是「內容空洞、典型 AI slop、可能要 close 重做」— 但 close hard gate「< 30 min polish」拉回正軌。實際 polish 跑下來 ~15 min 完成（frontmatter + footnote + §11 + heading + 結尾）。閉合動作後 contributor 收到的訊號是「我提交的東西被認真接住 + 維護者教我怎麼下次更好」，這對 DNA #7「先有再求好」+ feedback_merge_first_then_polish 是更好的 default。close 的 cost 是 contributor 1-3 天 turnaround friction + 信任流失。merge + heal + 教學 comment 的 cost 是 maintainer 15 min。**3:1 cost ratio，merge 永遠是 default**。

第三個觀察：footnote-format-fix.py 對 CN-bracket 格式的 regression 是真實 maintainer 痛點。idlccp1984 / Manus AI 出來的 footnote 常用「來源．標題．URL」格式，工具沒識別會把 title 替換成 raw URL。這是 LESSONS-INBOX 候選 — 工具應該先 dry-run 顯示 changeset，maintainer 確認後才 apply（DNA #48 v2 候選）。
