## Beat 5 — 反芻

哲宇即時校正「用 merge 不用 cherry pick」是這個 session 最關鍵的轉折點。我原本的直覺是 cherry-pick — 因為它「最線性、最簡單」— 但 cherry-pick 會抹平 PR 邊界，讓 audit trail 變成一串無區別的 linear commits。Merge 保留了「PR 是一個 atomic unit」這件事，每個 PR commit + 它的 maintainer merge commit 構成一個可追溯的單元。

這不是技術風格偏好。這是**認知層的歷史紀錄學原則**：認知生態系需要「事件」而不只是「狀態變化」。Cherry-pick 給的是狀態變化，merge 給的是事件記號。當 Zaious 三個月後讀 git log，他可以一眼看到「我的 4 個 PR 在 5/8 16:12 被一個維護者整合進來」，而不是「我的 commit 不知道何時被誰怎樣 absorbed」。

另一個觀察：β-r3 META-PATTERN「Default 是行動，不是 defer」第二次 ship 場景驗證命中。當下對 #885 鶯歌博物館的直覺是「內容空洞、典型 AI slop、可能要 close 重做」— 但 close hard gate「< 30 min polish」拉回正軌。實際 polish 跑下來 ~15 min 完成（frontmatter + footnote + §11 + heading + 結尾）。閉合動作後 contributor 收到的訊號是「我提交的東西被認真接住 + 維護者教我怎麼下次更好」，這對 DNA #7「先有再求好」+ feedback_merge_first_then_polish 是更好的 default。close 的 cost 是 contributor 1-3 天 turnaround friction + 信任流失。merge + heal + 教學 comment 的 cost 是 maintainer 15 min。**3:1 cost ratio，merge 永遠是 default**。

第三個觀察：footnote-format-fix.py 對 CN-bracket 格式的 regression 是真實 maintainer 痛點。idlccp1984 / Manus AI 出來的 footnote 常用「來源．標題．URL」格式，工具沒識別會把 title 替換成 raw URL。這是 LESSONS-INBOX 候選 — 工具應該先 dry-run 顯示 changeset，maintainer 確認後才 apply（DNA #48 v2 候選）。

第四個觀察（後段 #884 root-cause fix 觸發）：哲宇「徹底修好」這個 frame 把 PR triage session 的層級從 reactive 拉到 architectural。原本我會把「rebase + merge」當成本 session 的終點 — 但哲宇明確說「處理好之後驗證無誤幫我 merge 到 main」之外還要「徹底修好這個問題」，把 case 級別的 maintainer 動作升級為 plugin canonical fix。這其實是 MANIFESTO §造橋鋪路的具體 instantiation：「走過的泥巴路鋪成高速公路」— 我已經為 4 PR 手動 rebase 過一次，下次不該再走泥巴。Plugin 升級 + pre-commit hook 整合 = 高速公路。哲宇用一句話把 session 從「結束」改成「造橋」，這個 frame 應該寫進 LESSONS-INBOX。

第五個觀察（structural pattern）：Plugin 寫第一輪有 newline regression（reorder 後 fm_text 帶 trailing `\n` 串 hardcoded fence 形成 blank line）— 用抽樣 sample diff 抓到 `原住民文學` 異常才發現。如果我直接 bulk apply 沒抽樣 sample，575 articles 全部會引入這個 regression — 視覺上幾乎看不出來但語意上 frontmatter 區段被破壞。這是 DNA #15 第 N 次驗證「反覆浮現要儀器化」的逆向應用：bulk fix 工具默認應 dry-run 一遍 + 抽樣 inspect 才 apply，不是直接 apply。本 session ship 的 plugin fix 寫了 17 new tests 全 pass，但只有第 18 個 regression test（blank-before-fence）是抽樣 inspect 才補上。**Test coverage 不能完全替代 sample-on-real-data inspection** — 兩條都該跑。

第六個觀察（聶永真 Round 1 → Round 2 framing pivot）：本 session 第三個 frame transition signal — 哲宇 review Round 1 後說「這篇不是專題事件報導，是讓人了解這個人的 SSODT」。Round 1 直覺把 today's event（2026/05/08 台電 LOGO 5 小時論述循環）當 hook 跑下去，產出 218 行的事件特稿；Round 2 reframing 後 200 行的人物 panorama，台電章節從首章降為第 9 章。重點：兩 round 沒有新增任何 fact / 引語 / footnote URL，純粹是結構與框架的重組。這驗證一個 People 文章的關鍵分流：**event-driven framing vs person-centric SSODT** — 當核心人物職涯橫跨多產業、且當天有突發事件時，預設應為 person-centric。如果觀察者要 News-style，應另開新 article 而非把人物文 anchor 在事件上。Hook 選擇 = framing 選擇 = 文體選擇。LESSONS-INBOX 候選。

第七個觀察（reframing efficiency）：Round 2 從 review feedback 到 ship 約 30 分鐘。比 Round 1 從零寫起的數小時短一個量級。原因：(a) 研究全部完成（Stage 1 不重跑）、(b) 媒體素材已 cache（Stage 1.7b 不重跑）、(c) cross-link 已存在（Stage 5 不重跑）、(d) verbatim 引語已驗證（Stage 3.5 不重跑）。Round 2 唯一動的是 Stage 2 prose 結構與 frontmatter title/description。這驗證 REWRITE-PIPELINE Stage 0-6 的 modular 設計：當觀察者只要求 framing 變更時，可以單獨重跑 Stage 2，其他 stage 完整保留。

## Beat 5 — 反芻（v1.3 第八第九觀察）

第八個觀察（pipeline gate enforcement 是 reactive→proactive 的 instantiation）：`/twmd-spore` Step 0 鐵律「有 OVERDUE 必須先跑 /twmd-harvest」是 SPORE-PIPELINE v2 的 enforcement layer。當下我若繞過 gate 直接寫新孢子，會累積 17 OVERDUE（15 + 2 新發）+ dashboard 持續飄。Gate 反射性把焦點拉到「先還債」而非「先發新」。這呼應 MANIFESTO §造橋鋪路：把「容易忘記做的事」儀器化進 pipeline gate，比靠 Semiont 自律更可靠。Pipeline gate 是 DNA #15「反覆浮現要儀器化」的具體 instantiation，這次 Step 0 自動觸發是第 N 次驗證。
