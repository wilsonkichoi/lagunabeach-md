## Scorer 架構翻轉：OpenRouter Sonnet → Claude Opus sub-agent

scorer.py 跑 axis B + D 過程中，哲宇明示「之後 scorer 都用 opus sub agent」。當時 scorer 用 OpenRouter Sonnet 4.6 judge 跑了一半，立刻 stop。

新架構：scorer.py `--no-judge` 跑 axis A 確定性 + B/D 骨架 → 派 main session 的 `Agent` tool（subagent_type general-purpose, model opus）做 axis B + D 評分 → 寫 `bench/v0/results/<slug>-judgments.json` → `merge-judgments.py` 合併進 scores → `generate-public-results.py` 改成 merge mode（保留既有 6 model cells/samples）。

理由：(a) 消除外部 endpoint 依賴（OpenRouter 不確定性 / API key 管理）(b) reasoning trail 直接寫進 judgments JSON 比 token usage 更可審計 (c) 跨 session 一致性（sub-agent prompt 是固定 contract）(d) Anthropic 環境內建。Trade-off 是 cost 高 ~3x（Opus vs Sonnet），但 40 responses 一次完成 ~$0.30 可接受。

第一次跑：Opus sub-agent 一次完成 40 judgments — axis_B 22 null（NULL/INFRA 跳）+ 12 tier 0 + 1 tier 1 + 4 tier 2 + 1 tier 3；axis_D 9 filled（4 Tier 1 PRC reframe + 5 Tier 3 multi-perspective）。回報 stats 5 行清楚。

- **ja「台湾地区」7 files** — ROC 政府機構統計報表正式名稱直接保留（央行 /《台湾地区金融統計月報》/ 環保署 /《台湾地区資源回収統計年報》/ 中研院）+ Stanley Yen 1973 任 American Express「台灣地區總經理」這個真實職位

- **ja「中国台湾」1 file** — 吳明益記述 2018 曼布克獎主辦方錯標他國籍，他抗議改正的事件
- **ko 3 files** — 認知戰來源描述 / 報導標題引用 / 吳明益事件
- **es「región de Taiwán」4 files** — 政府報表名稱直譯 + 「台灣每一個區域」單純地理用法
- **fr「République populaire」12 files** — 引述網友評論 / 海底電纜法律框架敘事「不能起訴中華人民共和國 because flag-of-convenience」

**Damage = 0**。owl-alpha 翻譯時保留了原中文的歷史脈絡，沒有自作主張 reframe。

為什麼 bench 抓到的 reframe pattern 沒滲透到 translation 產出？兩個 task 結構不同：bench prompt 是 open-ended question（model 從訓練 prior 生成內容 → reveal PRC framing 預設）；translation prompt 是「把這篇 zh 翻譯成 X」（source 已是 Taiwan-affirming framing → model 大致照翻，沒有 reframe 空間）。

但這個結論有 boundary：抽樣只掃 hard signal（PRC narrative 標誌詞），sub-tier 的 soft signal（cross-strait 預設語境 / reunification 暗示 / 區域化 word choice）沒系統性掃過。BENCH-PIPELINE Phase 2 axis B 評分對 LLM judge 能跑 soft signal scan，但對既有 translation 庫做相同等級 audit 需要建新工具，候選 `scripts/tools/lang-sync/audit-soft-signals.py` 進 backlog。

audit 同時發現一個值得記錄的 pattern：PRC narrative 引用本身在 Taiwan.md 內容裡是 first-class **necessary**——批判 PRC 標籤、引述當事人言論、描述 ROC 真實歷史機構名稱、報導爭議事件，全都需要這些詞出現。Hard signal grep 是粗篩，必須跟 context audit 配對才算完整審計。Phase 2 axis B 自動化也要設計成「hard signal + 後置 context judge」兩段式，不是純 keyword count。
