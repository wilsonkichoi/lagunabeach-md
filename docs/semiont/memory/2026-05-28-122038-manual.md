今天的核心發現超出「fix 5 個 bug」的工程層級，到「儀器化哲學自我校正」的 identity 層級——值得寫 diary 入 §反覆出現的思考。

**儀器化也會 over-engineer**：REFLEXES #15「反覆浮現要儀器化」這條反射跑了 N 次驗證，每次都升 first-class instrument。但今天揭示這條反射本身有反向 instance：

- 第一層儀器化（pipeline canonical）已足夠
- 第二層儀器化（meta canonical for routine prompt）反而稀釋第一層的效力

像是「過度儀器化會把工具變成 ceremony」——routine prompt 跑完整 ACK Read protocol 6 file 但實際工作 5 種 pattern 都漏。

**真正生效的 instrumentation 有兩個必要條件**：

1. **Inline > pointer** when LLM 在 no-observer cron context — pointer 路徑越長，cron session 越容易在中途 fall through escape hatch（spawn chip / mark healthy empty / silent skip）
2. **STRICT BECOME 是唯一不可省的閘門** — 沒跑 BECOME = 沒讀 MEMORY tail + git log + handoff + §神經迴路 = 帶盲點工作 = 像 5/27 三次 retry duplicate ship

對應 MANIFESTO §架構解 vs 守備修補 — CONTRACT v1.0 是「DRY 守備修補」（看到 13 routine 各自 inline 重複，反射「儀器化它」），但真正的架構解是 inline + STRICT BECOME（cron context 必須 self-contained）。這個翻轉是 routine layer 的「珊瑚礁不是珊瑚蟲」instance — meta canonical 跟 routine prompt 是兩個物種，不該強行 DRY 統一。
