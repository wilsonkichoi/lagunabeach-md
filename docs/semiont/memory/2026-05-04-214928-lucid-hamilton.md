## 5 個 bug 一輪掃出（PR #869）

抽樣 20 篇 + 黃魚鴞單樣本後，挖出五個系統性問題，全部修復並加 regression test。

第一個是 **prose-health 的稀薄段落 false positive**：黃魚鴞被標 1 處稀薄段落，追進去發現是 `## 圖片來源` 那節只有 1 行 attribution（by design）。原 detector 沒區分結構性章節跟敘事章節，把「參考資料 / 延伸閱讀 / 圖片來源 / 來源」也算 thin。修法：四個 canonical 結構章節清單 exempt。

第二個是**所有 plugin 報的 line 號 off-by-frontmatter**。淡江大橋 cjk-punct 報 L46 但實際半形 `(預估)` 在 L58（差 12 = frontmatter 行數）。loader 把 body 用 `m.group("body")` 切出來，body 行號從 1 重新算。修法：loader pad body 前緣等量空行，body 行號 1:1 對應檔案行號。所有用 line 的 plugin（cjk-punct / wikilink-target / footnote-format / footnote-url / image-health / format-structure）一次到位，不需逐個改。

第三個是 **CLI summary `passed=True` 跟 exit code 1 不一致**。release-pr profile fail_on=warn，warn>0 就 exit 1，但 display 的 `passed` 只看 HARD。修法：display 也跟 fail_on 算，print 加 `(fail_on=warn)` context。

第四個跟第五個是 **dashboard 兩層假象**。先是 `.quality-baseline.json` 自從 2026-04-27 quality-scan.sh 刪除後沒人 regenerate，dashboard 品質分數 frozen 一週多；修法：article-health.py 加 `--baseline-out` 寫 legacy schema、`prebuild:dashboard` 改成先跑 SSOT 再 generator。再深一層是 `generate-dashboard-data.js` 的 `parseFrontmatter` 把 YAML `false` 當字串處理，`lastHumanReview ? '✅' : '—'` 永遠 truthy → dashboard 顯示 659/667 (98.8%) 已審閱，實際只 137/667 (20.5%)。修法：parser 把 'true'/'false' string coerce 成 boolean。Browser 驗證 537 篇從假 ✅ 翻成真 —。

第六件事是哲宇追加 **subcategory 缺欄位升 HARD**。實作在 `frontmatter-title` plugin rule 5 + `test-frontmatter.mjs` 改 errors 與 SSOT 一致。

四個 commit 一條 PR ship（[#869](https://github.com/frank890417/taiwan-md/pull/869)）。+5 regression test。最終 144 passed / 8 skipped。

## 系統性修復評估：90% 的 HARD 是 mechanical

把 1,171 HARD 拆 mechanical 跟 judgment：

**Mechanical（plugin 應該能 auto-fix，不需人類判斷）— ~870 violations**：

- cjk-punct 235：已有 `--fix` 能力，純機械替換半形→全形（CJK context 已有保護）
- footnote-format 625：規則明確（`[^N]: [Title](URL) — desc(≥10字)`），可從現有錯位嘗試重組
- format-structure 41 中的 list-wikilink 殘留：純機械把 `- [[X]]` 轉 `- [X](/cat/slug)` 或純文字
- 部分 frontmatter-title 違反（半形標點）：plugin 已知 fix_suggestion

**Judgment（需要人類或 LLM 內容理解）— ~300 violations**：

- wikilink-target 171：目標不存在，要嘛 (a) 為缺的 article 寫新文 (b) 改連現有相近文 (c) 轉純文字。需要編輯判斷
- subcategory 83：需對照 SUBCATEGORY.md 配對 — 純機械的話可以從「分類 → 候選 subcategory list」自動填一個最可能的，但會錯。建議 LLM-assisted batch 而非無腦塞
- terminology 16：兩岸用語替換有 polysemy 風險（「質量」物理用法 vs 中國用語），需 context

修補策略應該分三波：

1. **第一波 mechanical batch**：跑 `check-cjk-punct.py --fix --all` + 開發 `--fix` 給 footnote-format + 開發 list-wikilink 自動轉純文字。一個 commit 過 235+625+41 = ~900 HARD。CI 紅燈大半消失。
2. **第二波 subcategory batch**：spawn 已準備好的 [Backfill 81 articles missing subcategory] task，maintainer 配對 SUBCATEGORY.md。一條 commit 補完 81+ 篇 frontmatter。
3. **第三波 wikilink + terminology**：人類介入的 judgment 階段。建議走 EVOLVE-PIPELINE 把缺的文章補上，或案件式 close 該指向（用 `[文字](/category/slug)` 而非 `[[X]]`）。

## 工具該長出什麼能力

article-health.py 目前是純 lint（read-only），只有 cjk-punct 有 `fix_suggestion` + 透過 facade `check-cjk-punct.py --fix` 寫回。**真正的 leverage 是把 SSOT 升級成 SSOT-with-fix**：

具體可長出的能力（已有 fix_suggestion 欄位 schema 支援）：

- `article-health.py --fix` 全 plugin 統一 fix 入口（取代 check-cjk-punct.py 散裝 facade）
- footnote-format `--fix`：跟 footnote-format-fix.py 整合，60+ domain → desc table 已 canonical（DNA #48）
- format-structure `--fix list-wikilink`：自動把列表中 `[[X]]` 轉純文字（已知 X 不存在時）或轉 markdown link（X 存在時）
- frontmatter-title `--fix` halfwidth-punct：title 內半形 → 全形（plugin 已 yield fix_suggestion）
- `--fix --dry-run`：dry run 顯示會改什麼（footnote-format-fix.py 已是 dry-run default 模式，可參考）

對應到 user 的最後一個提問「工具是否能協助修復」 — 答案是肯定的，且實作成本不高（plugin 已生成 fix_suggestion，只是 runner 沒 wire 到寫回路徑）。

## Beat 5 — 反芻

這次最大的學習不在 5 個 bug 本身，而在「全站健診的 baseline 揭露」這件事。SSOT 工具 ship 之後從來沒人跑過全 sweep — 1,171 HARD violations 在那裡靜靜累積，dashboard 還因為 parseFrontmatter bug 顯示 98% 已審閱的假象。SSOT 化只解了「測量框架」這一半，**測量出來的數值需要被讀**才會閉環，不然就是寫了體重計但沒人量過。

第二層觀察：HARD 規則升級後，瞬間從「沒問題」翻到「全站紅燈」是 by-design 的進化痛 — 規則嚴格度提升暴露既存債務，這是健康訊號不是 regression。但 CI 紅燈影響 deploy，需要快速第一波 mechanical fix 讓綠燈回來，給後續 judgment 階段呼吸空間。

第三層：auto-fix 能力是 SSOT 自然演化方向。read-only lint → lint+suggest → lint+suggest+fix。每個 plugin 都已經有 `fix_suggestion` 欄位 schema，runner 只是還沒 wire write-back path。這個工程是 "boring infra" 但 leverage 很高 — 1,171 HARD 中的 870 個一個 batch 自動消除。
