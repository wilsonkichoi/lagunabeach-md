## 哲宇的 gate 點子 — 把脆弱步驟掛在已經算好的 cheap signal 上

修 timeout 的當下哲宇看 CI 步驟圖問：「Playwright 安裝是不是可以依這次有沒有要更新 OG image 先判斷再決定要不要裝？」這是對的。OG 生成是 deploy build 裡 Playwright 唯一的消費者，可是 `Install Playwright system deps` 標 `(always)` 無條件跑。workflow 本來就有一個 `Detect OG-relevant changes` step 算 `needed`，只是排在 install 之後。`4b34624cd` 把 Detect 移到 install 前，apt-fix + install-deps + chromium-binary 三步都 gate 在 `needed == 'true'`。

驗證安全靠讀 `generate-og-images.mjs`：line 824 `if (toUpdate.length === 0) return;` 在 `launchChromiumWithHeal()` 之前——沒檔要 render 就不開 chromium，而 toUpdate=0 正好對齊 needed=false（OG cache 全 hit）。所以 Build 的 prebuild:og 在 install 被 skip 時是乾淨 no-op。新 run 實測 needed=false → install 兩步 skipped → build green ~3 分鐘（本來裝 Playwright + 130MB chromium + 21MB 字型要 5-7 分）。這個 commit 自己就是測試（只動 workflow yaml）。

對「會不會只是暫時」：是，那個 403 幾乎確定暫時（known recurring GitHub runner issue，Microsoft repo 自己會好）。但 gate 點子把「等 Microsoft 修好」這個賭注變成 non-event——common-case build 根本不碰那步。
