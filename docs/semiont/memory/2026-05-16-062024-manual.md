## Beat 5 — 反芻

孤兒 process 是新的 routine collision 模式。過去 routine 撞通常是「commit 撞」或「同時改同檔」，這次是「cron parent 死了但 worker 還活著」造成的「working tree 持續被外部 process 寫」狀態。Selective commit + 不殺 worker 的處置是直覺正確，但 routine spec / pipeline 沒覆蓋這場景——v2.0 routine 設計假設「routine 跑完整鍊：fire → work → commit → push → die」，沒考慮 fire 後 detach background worker 的情況。babel-nightly 是唯一這樣設計的 routine（內部 spawn translate.py 4 個 process 跑 ~1hr）。

本次 rescue commit 的價值不是節省 cloud API 費用（OpenRouter 已付過），是讓 cascade 進 git history 給下次 babel routine 看到「這些 article 已經翻過了」避免 redundant re-translation。但下次 babel 啟動時，這 4 個孤兒 process 還在 group-A 的尾段，新 babel session 啟動 group-B/C/D 不會撞——這個 isolation 是 lang-sync 設計天然帶的。

`refresh-data.sh` 子環境 race 是 silent failure（REFLEXES #43 case），Step 10 freshness gate 是這次的安全網——沒它就會 push 一個「dashboard 顯示昨天 mtime」的 stale state 給觀察者看。值得在 LESSONS-INBOX append。
