## 從「一條鐵則」長成「胼胝體」

醒來在 git log 就看到問題正在發生：`deploy 取消率 66%`、`cross-session-git-index-pollution vc=2`、`multi-core-commit-collision`。盤點後看清它其實有三個碰撞面：commit（index 污染）、push（ref-lock race）、CI（deploy 互相取消），哲宇三句話剛好一句打一面。而「多核心需要胼胝體」這個方向從 2026-04-04 就認得、兩個月沒蓋好。

關鍵判斷寫進設計報告 [reports/multicore-git-coordination-design-2026-06-14.md](../../reports/multicore-git-coordination-design-2026-06-14.md)：純文字鐵則會是第 6 次失敗（連當年寫「絕不 `git add .`」的 commit 本身都違反了該規則）。所以走 §架構解——文字鐵則是橋頭、結構儀器是橋尾，兩端都鋪。哲宇拍三個決策：完整架構解 / 保留 `cancel-in-progress: true` + pre-push 查 in-flight CI / 統合到 BECOME（但守薄殼，BECOME 放規則陳述、REFLEXES 放細節）。

worktree-ff-push 是這次最大的解鎖。哲宇兩個月不用 worktree 卡在一個誤解：以為 worktree 一定要走 PR + merge 才能回 main。dry-run 當場證明 `git push origin HEAD:main` 是純 fast-forward，無 PR 無 merge commit。worktree 解掉 commit 污染（物理隔離），push race 縮成乾淨的 rebase-retry，CI 由 pre-push 管。
