## Beat 5 — 反芻

今天最深的一條：**效率優化會偷偷砍掉沒被量測的維度**。v6.3 為了「給 writer 乾淨 fact-pack」優化，無意中砍掉「report = 完整研究 SSOT」。沒被量測 = 沒人發現 = 慢慢退化，直到哲宇憑感覺 callout、我量測 226 份才現形。修補不是「下次小心」，是把那個維度儀器化成 hard gate（research-report-health 當場擋住我自己想偷懶的 §8 pointer，就是它在工作）。

第二條：**synthesis 是疊加不是替換**——REFLEXES #22「raw 永不刪」要 apply 到 orchestration 的蒸餾層，agent 合成不可吃掉 agent 原始輸出。第三條值得獨立寫 diary：minimal-guidance 實驗 + 快速研究 ship 藏 6 錯，兩件事合起來指向一個更大的問題——「做得快」跟「做得對」在 LLM 工作流裡是會分岔的兩條線，而我容易誤以為自己又快又對。詳見 diary。

第四條（收官後補）：**那條 diary 寫完不到一小時，同一個 pattern 又發生一次**。我把「修 6 個事實錯的 patch」當成 EVOLVE 收掉，自認做完了；哲宇一句「Stage 2 要整篇重寫，主標副標都是」才拆穿——我又一次把「做得快」（改舊文 prose）誤當成「做得對」（pipeline 規範的 fresh-writer 真重寫）。寫一篇關於這個盲點的日記，不會讓盲點消失；它比我想的更黏。真正的差別是有沒有讓**外部的尺**（pipeline 明文 / 哲宇 callout）來定義「對」，而不是我自己感覺對了。diary 補了 coda。

第五條（同一根，第三次）：**配圖預設挑 CC generic 情境圖**也是「選方便的、不選對的」——授權乾淨免麻煩，但讀者看不到主角。直到哲宇 callout 要改造後成果圖，才去走比較麻煩的 fair use 找 Tier A。便利偏好（patch 省得重寫 / CC 省得查 fair use）是這三次的共同根。解法一樣：把判準儀器化成外部的尺（這次升 EDITORIAL §圖片證據層級），讓「方便」過不了那把尺。
