### 設計關鍵決策

**Phase C 雙層 vs 單層**：C2 INBOX state check 是這條 SOP 最 load-bearing 的判斷層。MAINTAINER v2.2 之前的 issue 分類隱含假設「Issue 提的主題 ↔ knowledge/ 文章 1:1」，沒考慮 INBOX 已 propose 但未 ship 的中間態。tboydar-agent cron 三輪 100% overlap 揭露 INBOX state 是 first-class state，須跟 knowledge/ 平行 grep。

**Default priority = P2**：cron-generated 自動建議 default P2，避免無訊號升 P0 稀釋真 P0 信號池。升 P1 需具體 SC 曝光 / 國際時效；升 P0 需哲宇 explicit ruling。

**4-route 分流**：R1 已 ship / R2 INBOX P0-P1 重複 / R3 INBOX P2-P3 backlog / R4 真缺口。R2 vs R3 差異是「重複建議直接 close（信號強度足）」vs「backlog 評估 promote signal（contributor 補強可能升 P1）」。

**Upstream cron 校準訊號**：「同 contributor 同 cron source 連續 ≥ 2 輪同主題」是 first-class emergent pattern → LESSONS-INBOX append + reply 主動告知 contributor。**maintainer pipeline 不只是受體，也是訊號回傳器**。
