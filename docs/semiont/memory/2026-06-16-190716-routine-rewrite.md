## Beat 5 反芻

**最大洞察**：fresh Opus writer 在乾淨 context 寫 5500 字長文，所有結構性 hard gate 都 PASS——但 4 條紅線（中譯回譯 / venue extrapolation / over-claim / verbatim drift）+ 8 個 footnote URL placeholder 還是漏網。這 4 條全部是「writer 自己以為 OK 但 SSOT 對不上」級的。Stage 2.5 主 session 比對是必須的 hard gate，不是 nice-to-have——把 writer 的 self-report 當 oracle 就會 ship 出帶語意瑕疵的成品。REFLEXES #31「sub-agent claim 是線索不是 oracle」第 N 次驗證；6/16 報導者 4-agent 全綠後 manual 又抓 7 真錯也是同 pattern。

**過程洞察**：staging 檔分離 + 主 session 比對覆蓋（v7.5 / 6/15 哲宇 callout 引發）的價值不在「writer 不污染舊文」這層而已，更在「主 session 強制有一個 review 視窗」。如果 writer 直接寫 canonical，主 session 容易因「commit 就好」跳過 review；staging 檔位於不同 path，逼著我做 cp 動作前先 Read + diff，4 紅線就在這層被抓到。架構閘比意志力閘可靠。

**工具學到**：footnote URL placeholder 是 writer agent 隱蔽 hallucination 的高發區。research §7 verification table 雖有 source 標記，但若沒有完整 URL（bare domain / slug 未填），writer 會以 placeholder pattern「domain/article/...」吐回。Stage 2.5 必跑 `grep -nE "https?://[^)]*\.\.\.|https?://[^/)]+/$"` 抓 placeholder + bare-domain。下次寫 prompt 要直接要 writer 用 `[Title 不明]` 而非 `domain/...` placeholder（更明顯）。

**Spore defer 學到**：routine wall-clock budget + Chrome MCP token cost 之間的 trade-off。6/16 報導者 manual session 把全弧（rewrite + spore + 串文 + heal）一次跑完是哲宇 in-loop 才能撐的；routine context 無觀察者，硬跑會 burn cache 但無 review gate，反而是 silent failure 風險。Defer 不是失敗，是「把 social broadcast 放回有 review 的 session」。
